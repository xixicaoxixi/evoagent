## 二、八维度双项目深度对比

### 维度 1：Agent 编排

#### 1.1 核心设计对比

| 子维度 | claude-code-sourcemap | openclaw | 推荐方案 |
|--------|----------------------|----------|---------|
| **编排模式** | Coordinator-Worker（LLM 自主编排） | Harness + Runtime（代码编排） | **混合模式**：代码编排保证可控性，LLM 辅助任务分解 |
| **任务分解** | Prompt 驱动（四阶段工作流） | 配置驱动（Agent 定义 + 技能系统） | 配置驱动为主，Prompt 辅助动态调整 |
| **结果聚合** | LLM 自主综合 | 事件流回调 | 事件流回调 + LLM 综合摘要 |
| **并发控制** | LLM 自主判断 + Git Worktree 隔离 | Actor Queue 强制串行化 | Actor Queue + 可选 Worktree 隔离 |
| **通信机制** | 邮箱文件 + XML 通知 | ACP 协议 + WebSocket | 消息队列（可替换底层传输） |
| **Agent 定义** | Markdown frontmatter + 五源合并 | 配置文件 + 插件注册 | Markdown 声明式 + TypeScript 编程式双轨 |
| **故障转移** | 无（单模型） | 模型级故障转移链 + Harness 回退 | 模型级故障转移链（借鉴 openclaw） |
| **生命周期** | 同步/异步 + 前后台切换 | 运行时缓存 + 空闲驱逐 | 异步为主 + 空闲驱逐 + 可选前台 |

#### 1.2 通用编排模式

**推荐架构：分层编排**

```
┌─────────────────────────────────────────────────────────────────┐
│                    编排决策层（可选 LLM 辅助）                     │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  任务分解 → 依赖分析 → 并行规划 → 结果综合               │  │
│  │  （简单任务跳过此层，直接进入执行层）                     │  │
│  └──────────────────────────┬───────────────────────────────┘  │
├─────────────────────────────┴─────────────────────────────────┤
│                    执行调度层（代码编排）                        │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  AgentHarness 注册表 + 优先级选择 + 兜底策略              │  │
│  │  Actor Queue 串行化 + 可选 Worktree 隔离                  │  │
│  └──────────────────────────┬───────────────────────────────┘  │
├─────────────────────────────┴─────────────────────────────────┤
│                    运行时层（Agent 执行引擎）                    │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  模型调用 + 流式传输 + 工具执行 + 上下文管理              │  │
│  └──────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

**关键设计决策**：
1. 简单任务（单步工具调用）直接进入运行时层，避免编排开销
2. 复杂任务（多步、多Agent）经编排决策层分解后进入执行调度层
3. 编排决策层可选用 LLM 辅助（借鉴 claude-code 的 Coordinator 模式），但最终执行由代码控制
4. Actor Queue 保证同一会话的操作串行化（借鉴 openclaw），Worktree 提供可选的文件级隔离（借鉴 claude-code）

### 维度 2：工具系统

#### 2.1 核心设计对比

| 子维度 | claude-code-sourcemap | openclaw | 推荐方案 |
|--------|----------------------|----------|---------|
| **工具抽象** | `Tool<I,O,P>` 泛型接口（~40 方法） | `AgentToolWithMeta` + 配置驱动 | 精简泛型接口（~15 核心方法） |
| **工厂函数** | `buildTool()` + 7 个安全默认值 | 配置声明 | 工厂函数 + fail-closed 默认值 |
| **权限检查** | AST 级命令解析 + AI 分类器 + 10 层管线 | 工具级 allow/deny + 审计扫描 | 分级权限（只读/写入/执行/危险） |
| **输入验证** | 每工具 `validateInput()` + Zod | 配置级 Zod | 每工具 Zod Schema + 全局约束 |
| **结果控制** | `maxResultSizeChars`（每工具独立） | 无显式限制 | 每工具独立上限 + 持久化溢出 |
| **沙箱** | macOS Sandbox（Seatbelt） | Docker 多维纵深防御 | 插件式沙箱后端（借鉴 openclaw） |
| **LSP 集成** | 完整（9 种操作 + 被动诊断） | 无 | 可选 LSP 模块（借鉴 claude-code） |
| **Prompt Cache** | 工具排序优化（内置前缀） | 无显式优化 | 工具按名称排序 + 内置前缀 |

#### 2.2 推荐工具接口设计

```typescript
interface Tool<Input, Output> {
  // === 身份与发现 ===
  name: string;
  description(input: Input): Promise<string>;
  inputSchema: ZodType<Input>;
  
  // === 核心执行 ===
  call(input: Input, context: ToolContext): Promise<ToolResult<Output>>;
  validateInput?(input: unknown): ValidationResult;
  checkPermissions?(input: Input, context: ToolContext): PermissionResult;
  
  // === 安全分类 ===
  isReadOnly?(input: Input): boolean;       // 默认 false
  isConcurrencySafe?(input: Input): boolean; // 默认 false
  isDestructive?(input: Input): boolean;     // 默认 false
  
  // === 结果控制 ===
  maxResultSizeChars?: number;               // 默认 30000
  
  // === 生命周期 ===
  isEnabled?(): boolean;                     // 默认 true
  dispose?(): void;
}
```

**设计原则**：
1. **精简核心**：从 claude-code 的 ~40 方法精简到 ~15 个，移除 UI 渲染相关方法
2. **安全默认**：`buildTool()` 工厂提供 fail-closed 默认值（借鉴 claude-code）
3. **Zod 统一**：输入验证统一使用 Zod Schema（两个项目共识）
4. **结果溢出**：超过 `maxResultSizeChars` 的结果持久化到磁盘（借鉴 claude-code）

### 维度 3：上下文管理

#### 3.1 核心设计对比

| 子维度 | claude-code-sourcemap | openclaw | 推荐方案 |
|--------|----------------------|----------|---------|
| **架构模式** | 固定实现（内置） | 可插拔引擎（ContextEngine 接口） | **可插拔引擎**（借鉴 openclaw） |
| **Prompt 分层** | 6 层拼接（核心→记忆→追加→系统→用户→初始化） | 引擎驱动（assemble 返回 systemPromptAddition） | 6 层拼接 + 可插拔引擎双轨 |
| **压缩触发** | 四级递进（API原生→Micro→Auto→Reactive） | 两级修剪 + 多阶段摘要 + 质量守卫 | 多级递进 + 质量守卫 |
| **压缩模型** | 使用同一模型 | 支持独立配置 | 支持独立压缩模型 |
| **摘要策略** | 9 段式结构化摘要 | 多阶段（分块→部分→合并）+ 渐进降级 | 多阶段 + 结构化 + 渐进降级 |
| **质量保证** | 无显式检查 | 质量守卫（5 段必需 + 标识符保留） | **质量守卫**（借鉴 openclaw） |
| **Tool Result** | maxResultSizeChars | 两级截断（单条 + 总量）+ 智能尾部 | 两级截断 + 智能尾部保留 |
| **Session 文件** | 持续增长 | 压缩后自动截断 | 压缩后自动截断 |
| **Token 预算** | 90% 阈值 + 边际收益检测 | 上下文窗口守卫（16K/32K） | 双重停止（百分比 + 边际收益） |
| **缓存策略** | lodash memoize（会话级） | 引擎自行决定 | 引擎缓存 + 可配置 TTL |

#### 3.2 推荐上下文引擎接口

```typescript
interface ContextEngine {
  readonly info: ContextEngineInfo;
  
  // 必需方法
  ingest(params: IngestParams): Promise<IngestResult>;
  assemble(params: AssembleParams): Promise<AssembleResult>;
  compact(params: CompactParams): Promise<CompactResult>;
  
  // 可选生命周期
  bootstrap?(params: BootstrapParams): Promise<void>;
  maintain?(params: MaintainParams): Promise<void>;
  afterTurn?(params: AfterTurnParams): Promise<void>;
  dispose?(): Promise<void>;
  
  // 子代理支持
  prepareSubagentSpawn?(params: SubagentParams): Promise<SubagentPreparation>;
  onSubagentEnded?(params: SubagentParams): Promise<void>;
}
```

**关键设计决策**：
1. 采用 openclaw 的可插拔引擎架构，支持运行时切换引擎实现
2. 借鉴 claude-code 的 6 层 Prompt 拼接作为默认引擎实现
3. 压缩系统融合两者优势：claude-code 的多级递进触发 + openclaw 的质量守卫
4. Token 预算采用 claude-code 的双重停止条件（90% + 边际收益）

## 七、关键洞察与设计哲学

### 7.1 两个项目的核心设计哲学对比

| 维度 | claude-code-sourcemap | openclaw |
|------|----------------------|----------|
| **核心理念** | 深度 > 广度 | 广度 > 深度 |
| **编排哲学** | 信任 LLM 的判断力 | 信任代码的可控性 |
| **安全哲学** | 运行时逐次检查（精细） | 启动时全量扫描（全面） |
| **扩展哲学** | 编译时固定（稳定） | 运行时动态（灵活） |
| **状态哲学** | 内存为主（快速） | 持久化为主（可靠） |
| **优化哲学** | 极致性能（Prompt Cache） | 极致灵活（可插拔一切） |

### 7.2 融合哲学：深度可控的广度

> **推荐的设计哲学**：在 openclaw 的可插拔广度基础上，融入 claude-code 的深度优化。
>
> - **可插拔为默认**：所有核心组件通过接口抽象，支持运行时替换（借鉴 openclaw）
> - **深度优化为增强**：对关键路径（Agentic Loop、工具执行、Prompt 组装）进行极致优化（借鉴 claude-code）
> - **安全为底线**：运行时逐次检查 + 启动时全量扫描双模式（融合两者）
> - **渐进式复杂度**：简单场景零配置可用，复杂场景逐步启用高级功能

### 7.3 三个最重要的设计决策

1. **Agentic Loop 是心脏**：无论架构如何变化，"模型推理 → 工具执行 → 结果反馈 → 继续推理"的循环是 Agent 系统的核心。claude-code 的 AsyncGenerator 实现是最佳参考。

2. **可插拔是生命力**：openclaw 证明了"通过接口抽象 + 注册表 + 工厂模式"可以让系统在不修改核心代码的情况下演进。ContextEngine、AgentHarness、SandboxBackend 都是这一理念的体现。

3. **安全是不可妥协的底线**：两个项目在安全上的投入都极大（claude-code 的 AST 级命令分析、openclaw 的 70+ 文件审计系统）。安全不是一个模块，而是贯穿所有层级的横切关注点。

## 九、总结

经过对 claude-code-sourcemap（v2.1.88 sourcemap 还原源码）和 openclaw（开源多频道 AI 助手平台）两个项目共 ~2,990 KB 源码的系统性分析，我们提炼出了 **68 个设计模式**、**12 条设计原则** 和 **20 个高价值通用模式**。

**最核心的三个发现**：

1. **Agentic Loop 是 Agent 系统的心脏**——claude-code 的 AsyncGenerator + while(true) + Continue/Terminal 类型体系是教科书级的实现，应作为任何 Agent 系统的基础。

2. **可插拔架构是系统的生命力**——openclaw 证明了通过 ContextEngine、AgentHarness、SandboxBackend 等接口抽象，可以让系统在不修改核心代码的情况下持续演进。

3. **安全是贯穿所有层级的横切关注点**——两个项目在安全上的投入（AST 级命令分析、70+ 文件审计、多维纵深防御）表明，安全不是一个模块，而是一种设计文化。

**推荐的技术路线**：以 openclaw 的可插拔广度为骨架，融入 claude-code 的深度优化，构建一个"深度可控的广度"Agent 系统。分三阶段落地：核心引擎（2-3 周）→ 编排与扩展（3-4 周）→ 生产加固（4-6 周）。
