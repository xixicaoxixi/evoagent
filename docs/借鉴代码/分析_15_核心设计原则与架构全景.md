## 一、端到端请求旅程

Claude Code 的每一次用户交互，从键盘输入到最终输出，经历一条精密编排的多阶段管线。这条管线不是简单的线性调用链，而是一个由 `AsyncGenerator` 驱动的、可中断、可恢复的 agentic 循环。

### 1.1 query() — AsyncGenerator 驱动的 Agentic 循环

整个 Agent 的核心入口是 `query()` 函数，它被设计为一个 `async function*`，即异步生成器。这一选择不是偶然的——它从根本上决定了整个系统的执行模型。

```
用户输入
  │
  ▼
┌─────────────────────────────────────────────────────────┐
│  query() : AsyncGenerator<AgentEvent, void, undefined>  │
│                                                         │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐          │
│  │ 构建     │───▶│ 调用 LLM │───▶│ 解析     │          │
│  │ Context  │    │ API      │    │ Response │          │
│  └──────────┘    └──────────┘    └──────────┘          │
│       │               │               │                 │
│       │               ▼               │                 │
│       │       ┌──────────────┐        │                 │
│       │       │ yield 事件   │◀───────┘                 │
│       │       │ (流式输出)   │                          │
│       │       └──────┬───────┘                          │
│       │              │                                  │
│       │              ▼                                  │
│       │       ┌──────────────┐                          │
│       │       │ 工具调用?    │─── 否 ──▶ yield 最终结果 │
│       │       └──────┬───────┘                          │
│       │              │ 是                               │
│       │              ▼                                  │
│       │       ┌──────────────┐                          │
│       │       │ 执行工具     │                          │
│       │       │ (分区并行)   │                          │
│       │       └──────┬───────┘                          │
│       │              │                                  │
│       └──────────────┘                                  │
│              (循环继续)                                  │
└─────────────────────────────────────────────────────────┘
```

**关键洞察**：`query()` 不是一次性返回结果的函数，而是一个持续产出事件的流。每一次 `yield` 都是系统与外部世界交互的边界——UI 可以在此处渲染中间结果，用户可以在此处中断执行，错误可以在此处被捕获和处理。

### 1.2 yield 作为流式输出与中断检查点

`yield` 在这个架构中承担双重职责：

1. **流式输出**：将 LLM 的 token 流、工具执行进度、状态变更等事件推送给消费者（UI 层），实现实时反馈。
2. **中断检查点**：每次 `yield` 返回控制权给调用者，调用者可以选择不再调用 `next()`，从而优雅地终止整个 Agent 循环。

```
时间线 ─────────────────────────────────────────────────────▶

query() 内部:    [构建ctx]──[LLM调用]──[yield token流]──[解析工具]──[执行工具]──[yield结果]──[循环]
                                      ▲                                    ▲
                                      │                                    │
UI 消费者:       ─────────────────────┼────────────────────────────────────┼──
                                      │ 渲染 token                         │ 渲染工具结果
                                      │ 检查用户中断                        │ 检查用户中断
                                      │                                    │
中断点:                              ◉                                    ◉
                  (每次 yield 都是一个潜在的中断/取消点)
```

**设计模式**：协程模式（Coroutine Pattern）——通过 `yield` 暂停执行，将控制权交还给调度器（消费者），实现协作式多任务和优雅取消。

### 1.3 五层上下文管理管线

每次 LLM API 调用之前，系统需要构建一个完整的上下文窗口。这不是简单的字符串拼接，而是一个五层管线，每层负责不同维度的信息注入。

```
┌─────────────────────────────────────────────────────────────┐
│                   上下文管理管线 (5 层)                       │
│                                                             │
│  Layer 5: System Prompt                                     │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 角色定义 + 行为约束 + 工具使用指南 + 输出格式要求     │    │
│  └─────────────────────────────────────────────────────┘    │
│                         ▼                                   │
│  Layer 4: 工具定义 (Tool Definitions)                        │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 可用工具列表 + 参数 Schema + 权限状态 + 使用示例     │    │
│  └─────────────────────────────────────────────────────┘    │
│                         ▼                                   │
│  Layer 3: 对话历史 (Conversation History)                    │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 历史消息 + 工具调用记录 + 工具结果 + 截断策略         │    │
│  └─────────────────────────────────────────────────────┘    │
│                         ▼                                   │
│  Layer 2: 会话上下文 (Session Context)                       │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 当前工作目录 + 环境变量 + 文件权限 + 项目配置         │    │
│  └─────────────────────────────────────────────────────┘    │
│                         ▼                                   │
│  Layer 1: 用户输入 (User Input)                             │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ 原始用户消息 + 隐式上下文 (选中文件、当前行等)         │    │
│  └─────────────────────────────────────────────────────┘    │
│                         ▼                                   │
│              ┌──────────────────┐                           │
│              │  组装为 API 请求  │                           │
│              │  发送给 LLM      │                           │
│              └──────────────────┘                           │
└─────────────────────────────────────────────────────────────┘
```

**关键洞察**：这五层管线的每一层都有独立的截断策略和 token 预算。系统不是简单地"塞满上下文窗口"，而是在有限的空间内做最优信息密度分配。System Prompt 拥有最高优先级（不可截断），用户输入次之，历史消息按时间衰减优先淘汰。

### 1.4 工具分区并行执行

当 LLM 返回多个工具调用时，Claude Code 不会串行执行它们，而是将工具按依赖关系分组为"分区"（partition），同一分区内的工具并行执行。

```
LLM 返回工具调用: [Read(A), Read(B), Grep(C), Write(D)]

依赖分析:
  Read(A) ── 无依赖
  Read(B) ── 无依赖
  Grep(C) ── 无依赖
  Write(D) ── 依赖 Read(A) 的结果

分区结果:
  ┌─────────────────────────────────────────┐
  │  Partition 1 (并行)                     │
  │  ┌─────────┐ ┌─────────┐ ┌─────────┐   │
  │  │ Read(A) │ │ Read(B) │ │ Grep(C) │   │
  │  └────┬────┘ └─────────┘ └─────────┘   │
  │       │                                  │
  └───────┼──────────────────────────────────┘
          ▼
  ┌─────────────────────────────────────────┐
  │  Partition 2 (依赖 Partition 1)          │
  │  ┌─────────┐                            │
  │  │ Write(D)│                            │
  │  └─────────┘                            │
  └─────────────────────────────────────────┘
```

**设计模式**：任务分区模式（Task Partition Pattern）——将无依赖的任务分组并行执行，有依赖的任务按拓扑排序串行执行，在保证正确性的同时最大化吞吐量。

## 二、七项核心设计原则

通过对 ch01、ch02、ch03 的深度阅读，可以提炼出 Claude Code 架构背后隐含的七项核心设计原则。这些原则并非显式声明为"设计原则文档"，而是从代码结构、错误处理策略、权限模型、沙箱机制等具体实现中系统性涌现的。

### 2.1 Streaming-First（流式优先）

**一切皆为流。** AsyncGenerator 是系统最基本的执行原语，而非 Promise 或回调。

```
传统请求-响应模型:
  请求 ──────────────────────────────────────▶ 响应
  (用户等待... 等待... 等待... 得到完整结果)

流式优先模型:
  请求 ──▶ [token] [token] [token] ──▶ [工具调用] ──▶ [结果] [token] ──▶ ...
           ▲       ▲       ▲              ▲              ▲       ▲
           │       │       │              │              │       │
         实时     实时    实时          实时          实时    实时
         渲染     渲染    渲染          渲染          渲染    渲染
```

这一原则体现在三个层面：

- **执行层面**：`query()` 作为 `AsyncGenerator`，每个 `yield` 点都是流式输出的边界。
- **API 层面**：LLM API 调用使用 streaming 模式，token 逐个产出。
- **UI 层面**：React+Ink 终端 UI 实时消费流式事件，逐字符渲染输出。

**关键洞察**：流式优先不仅仅是"用户体验优化"——它是系统架构的基础约束。因为一切皆为流，所以取消、中断、错误注入都变得自然——只需停止消费流即可。这比在 Promise 链中实现取消要优雅得多。

### 2.2 Default-Isolated（默认隔离）

**工具默认在沙箱中运行。** 系统不信任任何工具调用，包括系统自带的工具。

```
┌─────────────────────────────────────────────────────────────┐
│                    隔离边界                                   │
│                                                             │
│  ┌───────────────────────────────────────────────────────┐  │
│  │  Claude Code 进程                                     │  │
│  │                                                       │  │
│  │  ┌─────────────┐     ┌─────────────────────────────┐  │  │
│  │  │ Agent 核心   │     │ 工具执行沙箱                │  │  │
│  │  │ (编排逻辑)   │────▶│                             │  │  │
│  │  │             │     │  ┌─────────┐ ┌───────────┐  │  │  │
│  │  │             │     │  │ 文件操作 │ │ Bash 命令 │  │  │  │
│  │  │             │     │  │ 目录白名单│ │ firejail  │  │  │  │
│  │  │             │     │  └─────────┘ └───────────┘  │  │  │
│  │  │             │     │  ┌─────────┐ ┌───────────┐  │  │  │
│  │  │             │     │  │ 网络请求 │ │ 搜索操作  │  │  │  │
│  │  │             │     │  │ URL 白名单│ │ 路径限制  │  │  │  │
│  │  │             │     │  └─────────┘ └───────────┘  │  │  │
│  │  └─────────────┘     └─────────────────────────────┘  │  │
│  └───────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

隔离的具体实现：

- **文件操作**：通过 `allowedDirectories` 白名单限制文件访问范围，工具只能读写被授权的目录。
- **Bash 命令**：通过 `firejail` 沙箱执行，限制网络访问、文件系统访问和系统调用。
- **网络请求**：URL 白名单机制，防止 SSRF 攻击。
- **搜索操作**：路径限制，防止搜索超出项目范围。

**设计模式**：沙箱模式（Sandbox Pattern）——将不可信代码的执行环境与宿主系统隔离，通过白名单机制控制资源访问。

### 2.3 Least-Privilege（最小权限）

**权限系统强制执行最小必要访问。** 工具声明自己需要什么，用户授予具体权限。

```
权限请求流程:

  工具声明需求          权限运行时检查           用户决策
  ┌──────────┐       ┌──────────────┐       ┌──────────┐
  │ "我需要   │──────▶│ 检查已有授权 │──────▶│ 允许/拒绝 │
  │ 写入文件X"│       │ 匹配规则     │       │ 记住选择 │
  └──────────┘       └──────────────┘       └──────────┘
                            │
                     ┌──────┴──────┐
                     ▼             ▼
               已有授权         需要新授权
               直接执行         提示用户
```

权限的三个维度：

1. **工具级权限**：用户可以全局禁用某个工具（如禁止 Bash 执行）。
2. **操作级权限**：同一工具的不同操作可以有不同的权限（如允许读文件但禁止写文件）。
3. **资源级权限**：针对特定资源的细粒度控制（如只允许写入 `/tmp` 目录）。

**关键洞察**：最小权限原则不是一次性配置，而是持续演进的。系统从最严格的权限开始，随着用户交互逐步放宽。每一次权限放宽都是显式的、有记录的、可撤销的。

### 2.4 Progressive-Enhancement（渐进增强）

**从安全默认值开始，按需添加能力。** 系统不是"全有或全无"，而是一个渐进的能力光谱。

```
能力渐进光谱:

  低能力 ◀────────────────────────────────────────────────▶ 高能力
  安全   ◀────────────────────────────────────────────────▶ 强大

  ┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐
  │ Plan   │─▶│ Read   │─▶│ Edit   │─▶│ Execute│─▶│ Full   │
  │ Only   │  │ Only   │  │ Files  │  │ Bash   │  │ Access │
  └────────┘  └────────┘  └────────┘  └────────┘  └────────┘
    只读分析    文件读取    文件编辑    命令执行    完全访问
    无副作用    无副作用    需确认      需确认      需确认
```

渐进增强的具体体现：

- **Plan 模式 → Execute 模式**：先让 Agent 分析问题、制定计划（无副作用），用户确认后再执行。
- **Read-only → Read-write**：先以只读模式探索代码库，理解上下文后再请求写权限。
- **单文件 → 多文件 → 全项目**：文件操作权限从单个文件扩展到目录再到整个项目。
- **安全工具 → 危险工具**：先使用 Read、Grep、Glob 等安全工具，再按需启用 Bash、Write 等危险工具。

**设计模式**：渐进披露模式（Progressive Disclosure Pattern）——初始状态下只暴露最基本、最安全的功能，随着用户需求的明确和信任的建立，逐步解锁更强大的能力。

### 2.5 User-Controllable（用户可控）

**每一个自动化决策都可以被用户覆盖。** 系统从不假设"用户一定想要这个"。

```
自动化决策点与用户覆盖:

  ┌─────────────────────────────────────────────────────────┐
  │  Agent 决策                    用户覆盖方式              │
  ├─────────────────────────────────────────────────────────┤
  │  选择使用哪个工具               工具审批提示              │
  │  决定操作哪些文件               权限确认对话框            │
  │  判断是否需要执行 Bash          命令预览 + 确认          │
  │  决定上下文截断策略             用户可手动添加上下文      │
  │  选择模型 (Opus/Sonnet)        用户可显式切换            │
  │  决定重试策略                   用户可中断并修改          │
  │  自动继续 (auto-continue)       用户可关闭此功能         │
  └─────────────────────────────────────────────────────────┘
```

**关键洞察**：用户可控原则与最小权限原则形成互补。最小权限限制了 Agent "能做什么"，用户可控则确保 Agent "做什么"都经过用户知晓和同意。两者共同构建了一个"人在回路"（Human-in-the-Loop）的安全框架。

### 2.6 Graceful-Degradation（优雅降级）

**当某件事失败时，降级而不是崩溃。** 系统的默认失败模式是"提供有限但可用的服务"，而非"完全不可用"。

```
降级策略层次:

  ┌─────────────────────────────────────────────────────────┐
  │  Level 0: 完全可用                                       │
  │  所有功能正常运行                                        │
  └───────────────────────┬─────────────────────────────────┘
                          │ 某个组件失败
                          ▼
  ┌─────────────────────────────────────────────────────────┐
  │  Level 1: 功能降级                                       │
  │  模型降级: Opus → Sonnet (成本/可用性)                   │
  │  工具降级: 禁用失败工具，用替代方案                       │
  │  上下文降级: 激进截断，保留最关键信息                     │
  └───────────────────────┬─────────────────────────────────┘
                          │ 更多组件失败
                          ▼
  ┌─────────────────────────────────────────────────────────┐
  │  Level 2: 核心功能保留                                   │
  │  只保留最基本的对话能力                                   │
  │  禁用所有工具，仅作为聊天助手                             │
  │  提示用户当前限制                                        │
  └─────────────────────────────────────────────────────────┘
```

优雅降级的具体实现：

- **模型回退**：当 Opus 不可用或超时时，自动降级到 Sonnet。用户也可以手动选择模型。
- **特性标志（Feature Flags）**：新功能通过特性标志控制，可以按用户群体、按比例灰度发布，出现问题可以快速关闭。
- **熔断器（Circuit Breaker）**：对 API 调用实现熔断机制，连续失败达到阈值后暂停调用，避免雪崩效应。
- **工具分区错误隔离**：并行执行的工具分区中，一个分区的失败不会影响其他分区的执行。

**设计模式**：熔断器模式（Circuit Breaker Pattern）+ 装饰器模式（Decorator Pattern）——通过在组件外部包装降级逻辑，在不修改核心逻辑的前提下添加容错能力。

### 2.7 Observability（可观测性）

**每一个决策都被记录。** 权限决策、工具执行、错误处理全部产生遥测数据。

```
可观测性数据流:

  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ 权限决策  │  │ 工具执行  │  │ LLM 调用 │  │ 错误处理 │
  └────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬─────┘
       │              │              │              │
       ▼              ▼              ▼              ▼
  ┌─────────────────────────────────────────────────────┐
  │              OTel Span 链路追踪                      │
  │                                                     │
  │  [query:start]──[llm:call]──[tool:exec]──[yield]   │
  │       │             │            │          │        │
  │       ▼             ▼            ▼          ▼        │
  │  span_id:abc   span_id:def  span_id:ghi  span:jkl  │
  │  parent:none   parent:abc   parent:def   parent:ghi│
  └─────────────────────────────────────────────────────┘
       │              │              │              │
       ▼              ▼              ▼              ▼
  ┌─────────────────────────────────────────────────────┐
  │              结构化日志 + 指标                        │
  │                                                     │
  │  logger.info("permission.granted", { tool, path })  │
  │  metrics.increment("tool.execution.count", { name }) │
  │  metrics.histogram("llm.latency.ms", { model })     │
  └─────────────────────────────────────────────────────┘
```

可观测性的三个支柱：

1. **链路追踪（Tracing）**：通过 OTel spans 追踪请求在系统中的完整路径，每个 span 包含开始时间、结束时间、属性和事件。
2. **结构化日志（Structured Logging）**：所有日志以 JSON 格式输出，包含上下文信息，便于机器解析和聚合分析。
3. **指标（Metrics）**：关键操作的计数、延迟分布、错误率等指标，用于监控和告警。

**关键洞察**：可观测性不是为了"事后调试"，而是系统正常运行的必要条件。没有可观测性，就无法实现优雅降级（不知道什么失败了）、最小权限（不知道什么被访问了）、用户可控（无法向用户展示 Agent 做了什么）。

Claude Code 的代码组织遵循严格的四层架构，每一层有明确的职责边界和依赖方向。

### 3.2 Layer 1 — UI Layer（表现层）

UI 层的核心原则是**零业务逻辑**。

```
┌─────────────────────────────────────────────────────────┐
│  UI Layer 职责边界                                       │
│                                                         │
│  ✅ 应该做的:                                            │
│     · 消费 AgentEvent 流并渲染                            │
│     · 捕获用户输入并转发给编排层                           │
│     · 管理本地 UI 状态（滚动位置、焦点、主题）              │
│     · 显示权限确认对话框                                   │
│     · 渲染进度条和状态指示器                               │
│                                                         │
│  ❌ 不应该做的:                                           │
│     · 决定使用哪个工具                                     │
│     · 判断是否需要权限                                     │
│     · 构建上下文或截断历史                                 │
│     · 选择模型或处理 API 错误                              │
│     · 任何业务规则或策略逻辑                               │
└─────────────────────────────────────────────────────────┘
```

技术实现上，UI 层使用 React + Ink（React 的终端渲染引擎），将 Web 开发的组件化范式带入终端应用。这使得 UI 代码可以享受 React 的声明式渲染、Hooks 状态管理和组件复用等优势。

**关键洞察**：UI 层零业务逻辑原则是整个四层架构能够保持清晰的关键。如果 UI 层开始包含业务逻辑，就会导致逻辑分散、难以测试、难以替换。将所有决策集中在编排层和服务层，使得 UI 成为纯粹的"视图"，可以独立演进甚至完全替换（比如从终端 UI 切换到 Web UI）。

### 3.3 Layer 2 — Agent Orchestration（编排层）

编排层是 Claude Code 的"大脑"，负责协调所有组件完成用户的请求。

```
┌─────────────────────────────────────────────────────────┐
│  Agent Orchestration 核心组件                            │
│                                                         │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │ query()      │  │ Tool         │  │ Context      │  │
│  │ 主循环       │  │ Dispatcher   │  │ Manager      │  │
│  │              │  │ 工具调度器    │  │ 上下文管理器  │  │
│  └──────────────┘  └──────────────┘  └──────────────┘  │
│                                                         │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │ Permission   │  │ Error        │  │ Conversation │  │
│  │ Coordinator  │  │ Handler      │  │ Manager      │  │
│  │ 权限协调器    │  │ 错误处理器    │  │ 对话管理器    │  │
│  └──────────────┘  └──────────────┘  └──────────────┘  │
└─────────────────────────────────────────────────────────┘
```

编排层的核心职责：

- **query() 主循环**：驱动整个 agentic 循环，管理 LLM 调用和工具执行的交替。
- **工具调度器**：解析 LLM 返回的工具调用，进行依赖分析，构建执行分区，并行调度执行。
- **上下文管理器**：维护对话历史，执行截断策略，构建每次 API 调用的完整上下文。
- **权限协调器**：在工具执行前检查权限，必要时向用户请求授权。
- **错误处理器**：捕获和处理各种错误，决定重试、降级或终止。
- **对话管理器**：管理对话的创建、恢复、分支和持久化。

### 3.4 Layer 3 — Service Layer（服务层）

服务层封装了所有与外部世界交互的具体实现。每个服务都是独立的、可替换的。

### 3.6 层间依赖关系

```
依赖方向 (严格单向):

  Layer 1 (UI)
      │
      │ 消费事件流，不调用下层 API
      ▼
  Layer 2 (Orchestration)
      │
      │ 调用服务接口，不直接操作基础设施
      ▼
  Layer 3 (Service)
      │
      │ 使用基础设施，不感知编排逻辑
      ▼
  Layer 4 (Infrastructure)

  禁止:
    Layer 1 → Layer 3 (UI 不能直接调用服务)
    Layer 3 → Layer 2 (服务不能依赖编排层)
    Layer 2 → Layer 1 (编排层不能依赖 UI)
```

**关键洞察**：严格的单向依赖是保持架构清晰的关键。每一层只依赖其下一层，从不反向依赖。这使得每一层都可以独立测试、独立替换、独立演进。特别是 UI 层对编排层的单向依赖，使得整个系统可以在没有 UI 的情况下运行（如作为库被其他程序调用），也可以在不修改业务逻辑的情况下替换 UI。

## 四、AsyncGenerator 驱动的 Agent 循环

`query()` 函数作为 AsyncGenerator，是整个 Claude Code 架构中最精妙的设计。它不仅仅是一个"异步函数"，而是一个完整的 agentic 执行框架。

### 4.1 事件流模型

`query()` 产出的不是一个单一结果，而是一个类型丰富的事件流：

```
┌─────────────────────────────────────────────────────────────┐
│  AgentEvent 类型体系                                         │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐    │
│  │  AgentEvent (联合类型 / Discriminated Union)         │    │
│  │                                                     │    │
│  │  ├─ assistant_message    // LLM 文本输出 (流式)      │    │
│  │  ├─ tool_use            // 工具调用请求              │    │
│  │  ├─ tool_result         // 工具执行结果              │    │
│  │  ├─ permission_request  // 权限请求                  │    │
│  │  ├─ permission_response // 权限响应                  │    │
│  │  ├─ error               // 错误事件                  │    │
│  │  ├─ status              // 状态变更                  │    │
│  │  └─ complete            // 循环结束                  │    │
│  └─────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────┘
```

**设计模式**：判别联合类型（Discriminated Union）——通过 `type` 字段区分不同的事件类型，TypeScript 的穷举检查确保所有事件类型都被处理。

### 4.2 yield 作为检查点的完整语义

每次 `yield` 不仅是输出数据，更是系统的一个完整检查点：

```
yield 检查点的完整语义:

  ┌─────────────────────────────────────────────────────────┐
  │  yield(event) 执行时:                                    │
  │                                                         │
  │  1. 暂停 query() 执行                                    │
  │  2. 将 event 推送给消费者 (UI 层)                        │
  │  3. 消费者处理 event (渲染、记录、分析)                   │
  │  4. 消费者决定是否继续 (调用 next())                     │
  │     ├─ 调用 next() → query() 从 yield 处恢复执行        │
  │     └─ 不调用 next() → query() 被优雅取消               │
  │  5. 如果消费者传入参数 (如权限响应)                      │
  │     → 参数作为 next(value) 的返回值传递给 query()        │
  └─────────────────────────────────────────────────────────┘
```

这意味着 `yield` 不仅是输出点，还是输入点。UI 层可以通过 `next(value)` 向 `query()` 传回数据（如用户对权限请求的响应），实现了双向通信。

### 4.3 工具分区并行执行的详细机制

```
工具分区并行执行流程:

  LLM 返回: [toolA, toolB, toolC, toolD, toolE]
                │       │       │       │       │
                ▼       ▼       ▼       ▼       ▼
  ┌─────────────────────────────────────────────────┐
  │  Step 1: 依赖分析                               │
  │  · toolA: 无依赖                                │
  │  · toolB: 无依赖                                │
  │  · toolC: 依赖 toolA 的输出                     │
  │  · toolD: 无依赖                                │
  │  · toolE: 依赖 toolB 和 toolD 的输出            │
  └─────────────────────────────────────────────────┘
                │
                ▼
  ┌─────────────────────────────────────────────────┐
  │  Step 2: 分区构建                               │
  │                                                 │
  │  Partition 1: [toolA, toolB, toolD] (并行)      │
  │  Partition 2: [toolC]          (依赖 P1)        │
  │  Partition 3: [toolE]          (依赖 P1)        │
  └─────────────────────────────────────────────────┘
                │
                ▼
  ┌─────────────────────────────────────────────────┐
  │  Step 3: 分区执行                               │
  │                                                 │
  │  P1: ┌──────┐ ┌──────┐ ┌──────┐               │
  │      │toolA │ │toolB │ │toolD │  ← Promise.all │
  │      └──┬───┘ └──┬───┘ └──┬───┘               │
  │         │        │        │                     │
  │         ▼        ▼        ▼                     │
  │  P2: ┌──────┐           ┌──────┐               │
  │      │toolC │           │toolE │  ← 可并行      │
  │      └──────┘           └──────┘               │
  └─────────────────────────────────────────────────┘
```

### 4.4 错误缓冲与延迟报告

并行执行中，错误不会立即中断整个流程，而是被缓冲直到所有分区完成：

```
错误缓冲机制:

  Partition 1 执行:
    toolA: ✅ 成功 → 结果缓存
    toolB: ❌ 失败 → 错误缓冲 (不立即抛出)
    toolD: ✅ 成功 → 结果缓存

  Partition 2 执行:
    toolC: ✅ 成功 (toolA 结果可用)
    toolE: ❌ 失败 (toolB 结果不可用) → 错误缓冲

  所有分区完成后:
    ┌─────────────────────────────────────────┐
    │  汇总报告:                               │
    │  · toolA: 成功                           │
    │  · toolB: 失败 — 原因: ...               │
    │  · toolC: 成功                           │
    │  · toolD: 成功                           │
    │  · toolE: 失败 — 原因: 依赖 toolB 失败    │
    │                                         │
    │  → 将完整错误信息发送给 LLM               │
    │  → LLM 可以决定重试、使用替代方案或报告    │
    └─────────────────────────────────────────┘
```

**关键洞察**：错误缓冲策略是优雅降级原则在工具执行层面的具体体现。与其在第一个错误发生时就中断整个流程（可能浪费已经完成的工具执行），不如让所有独立的工具都完成，然后汇总报告。这最大化了每次 LLM 调用的信息获取量，减少了不必要的往返。

**设计模式**：延迟错误模式（Deferred Error Pattern）——不在错误发生时立即抛出，而是将错误收集起来，在合适的时机统一处理。这与"快速失败"（Fail-Fast）原则看似矛盾，但在并行执行场景下，延迟错误能带来更好的整体效率。

## 五、对 EvoAgent 的参考价值

### 5.1 AsyncGenerator 作为 Agentic 循环原语

Claude Code 的 `query()` 设计证明，`AsyncGenerator` 是构建 agentic 循环的理想原语。相比传统的 `async/await` 模型，它天然支持：

- **流式输出**：逐步产出中间结果，而非等待全部完成。
- **协作式取消**：通过停止消费流来优雅终止，无需复杂的取消令牌传播。
- **双向通信**：通过 `next(value)` 从消费者向生成器传回数据（如用户决策）。
- **自然暂停/恢复**：生成器函数的执行状态自动保存在每次 `yield` 之间。

对 EvoAgent 的建议：将核心 Agent 循环设计为 `AsyncGenerator`，而非返回 `Promise` 的普通异步函数。这将为流式输出、中断处理和用户交互提供统一的基础设施。

### 5.2 七项设计原则作为架构北极星

Claude Code 的七项核心设计原则不是孤立的"最佳实践"，而是一个相互支撑的原则体系：

```
原则间的支撑关系:

  Streaming-First ─────────────────────┐
  (一切皆为流)                          │
                                       ▼
  User-Controllable ◄──────── Graceful-Degradation
  (用户可控)          相互支撑        (优雅降级)
       ▲                               ▲
       │                               │
  Least-Privilege ◄─── Default-Isolated
  (最小权限)            (默认隔离)
       ▲
       │
  Progressive-Enhancement
  (渐进增强)
       ▲
       │
  Observability
  (可观测性)
```

对 EvoAgent 的建议：在架构设计初期就明确这七项原则（或根据自身需求调整），并将它们作为所有技术决策的评判标准。当面临"是否应该添加这个功能"的决策时，用这些原则来检验。

### 5.3 UI 零业务逻辑原则

Claude Code 严格遵循"UI 层零业务逻辑"原则，将所有决策集中在编排层和服务层。这一原则的价值在于：

- **可测试性**：业务逻辑可以脱离 UI 进行单元测试。
- **可替换性**：UI 可以从终端切换到 Web、桌面或移动端，而不影响核心逻辑。
- **可维护性**：修改业务逻辑不需要理解 UI 代码，反之亦然。
- **可复用性**：核心逻辑可以作为库被其他应用调用。

对 EvoAgent 的建议：在架构设计中明确划分 UI 层和业务层的边界。UI 层只负责渲染和事件转发，所有决策逻辑（包括权限检查、工具选择、错误处理）都放在编排层或服务层。

### 5.4 渐进增强用于安全的 Agent 能力扩展

Claude Code 的渐进增强策略为 Agent 系统的安全能力扩展提供了范本：

```
渐进增强在 EvoAgent 中的应用场景:

  初始阶段 (最安全):
  ┌──────────────────────────────────────┐
  │  · 只读操作 (代码分析、文档生成)      │
  │  · 无文件写入                        │
  │  · 无命令执行                        │
  │  · 无网络访问                        │
  └──────────────────────────────────────┘
       │ 用户信任建立后
       ▼
  中级阶段:
  ┌──────────────────────────────────────┐
  │  · 文件写入 (需确认)                 │
  │  · 安全命令执行 (白名单)             │
  │  · 受限网络访问                      │
  └──────────────────────────────────────┘
       │ 用户显式授权后
       ▼
  高级阶段:
  ┌──────────────────────────────────────┐
  │  · 完整文件操作                      │
  │  · 任意命令执行 (需确认)             │
  │  · 完整网络访问                      │
  └──────────────────────────────────────┘
```

对 EvoAgent 的建议：设计一个清晰的能力等级模型，从最安全的默认值开始，让用户逐步解锁更强大的能力。每一级能力的解锁都应该是显式的、有记录的、可撤销的。

### 5.5 优雅降级作为默认失败模式

Claude Code 将优雅降级作为系统的默认失败模式，而非事后添加的"容错补丁"。这意味着：

- **设计时考虑失败**：每个组件在设计时就定义了"失败时怎么办"，而非假设"永远不会失败"。
- **多层降级策略**：从功能降级到核心功能保留，确保系统在任何情况下都能提供至少基本的服务。
- **自动恢复**：熔断器在冷却期后自动尝试恢复，无需人工干预。

对 EvoAgent 的建议：在系统设计阶段就为每个关键组件定义降级策略。特别是 LLM API 调用（可能超时、限流、不可用）和工具执行（可能失败、超时、产生意外结果），都应该有明确的降级路径。将优雅降级作为架构的默认假设，而非例外处理。

---

> **总结**：Claude Code 的架构并非由某个宏大的"架构设计文档"驱动，而是从具体的工程问题中逐步涌现的。七项核心设计原则——流式优先、默认隔离、最小权限、渐进增强、用户可控、优雅降级、可观测性——相互支撑，共同构成了一个"安全优先、渐进开放、永不崩溃"的 Agent 系统架构范式。对 EvoAgent 而言，最有价值的参考不是某个具体的技术实现，而是这些原则所代表的架构哲学：**在 AI Agent 的能力与安全之间，永远选择安全作为默认值，然后通过渐进增强让用户自主选择开放多少能力。**
