## 二、概述：通信架构的分层哲学

通信协议是 AI Agent 系统的"神经系统"——它决定了 Agent 如何与用户、工具、外部服务以及自身的子进程进行交互。前五轮分析分别从宏观架构、Agent 编排、工具系统、查询引擎和进化引擎五个维度解剖了两个项目，但有一个横切关注点始终贯穿其中却未被独立审视：**信息如何在系统的各个边界之间流动**。

本轮分析聚焦于这一横切维度，覆盖以下核心问题：

| 问题域 | Claude Code | OpenClaw |
|--------|-------------|----------|
| **远程控制** | Bridge (v1/v2) + Remote (CCR) + Direct Connect | Gateway Protocol v3 + ACP |
| **进程间通信** | SDK 控制协议 (stdin/stdout JSON) | Gateway WebSocket + ACP NDJSON |
| **外部工具集成** | MCP Client/Server 双向实现 | ACP Runtime 抽象 + 插件系统 |
| **会话管理** | Crash Recovery Pointer + Session Ingress | Actor Queue + 身份协调 + 持久化绑定 |
| **权限审批** | control_request/control_response 控制面 | 8 级分类器 + 两阶段 RPC 审批 |

两个项目在通信架构上呈现出截然不同的设计哲学：

- **Claude Code** 采用"**协议联邦**"策略——多种专用协议（Bridge、Remote、Direct Connect、SDK、MCP）各自独立演进，通过消息适配器互联。这种设计允许每个协议针对特定场景深度优化，但增加了系统复杂度。
- **OpenClaw** 采用"**统一网关 + 协议翻译**"策略——所有通信汇聚到 Gateway Protocol v3 这一套核心协议，ACP 等外部协议通过翻译层桥接。这种设计简化了核心路径，但翻译层成为性能和功能瓶颈。

## 三、Claude Code 通信体系

### 2.1 四通道通信矩阵

Claude Code 存在四种独立的远程/SDK 通信通道，它们共享同一套 SDK Message 类型体系，但在连接方式、认证机制和适用场景上各有不同：

```
┌─────────────────────────────────────────────────────────────────┐
│                     Claude Code 通信矩阵                        │
├──────────┬──────────────┬──────────────┬───────────┬────────────┤
│   通道   │   连接目标    │   传输方式    │  认证方式  │  适用场景  │
├──────────┼──────────────┼──────────────┼───────────┼────────────┤
│  Bridge  │ claude.ai    │ SSE+HTTP/WS  │ OAuth+JWT │ Web远程控制│
│  Remote  │ api.         │ WS订阅+HTTP  │ OAuth     │ CCR云端会话│
│          │ anthropic.   │ POST发送     │ Bearer    │            │
│          │ com          │              │           │            │
├──────────┼──────────────┼──────────────┼───────────┼────────────┤
│  Direct  │ 自托管WS服务 │ WS双向NDJSON │ 可选Token │ 本地/团队  │
│  Connect │ 器           │              │           │ 自建服务   │
├──────────┼──────────────┼──────────────┼───────────┼────────────┤
│  SDK     │ 本地子进程   │ stdin/stdout │ 无需认证  │ 编程集成   │
│          │              │ JSON流       │           │            │
└──────────┴──────────────┴──────────────┴───────────┴────────────┘
```

**关键设计决策**：四个通道共享 `SDKMessage` 联合类型作为消息格式的基础，但各自的消息收发机制完全不同。这种"共享类型、独立传输"的设计使得消息适配器（如 `sdkMessageAdapter.ts`）成为系统中的关键粘合层。

### 2.2 Bridge 远程控制子系统

#### 2.2.1 双版本架构

Bridge 系统采用双版本并行架构，通过 GrowthBook 功能开关控制流量切换：

```
                    ┌─────────────────────┐
                    │   GrowthGate 决策    │
                    │ tengu_ccr_bridge    │
                    │ tengu_bridge_repl_v2│
                    └─────────┬───────────┘
                              │
              ┌───────────────┼───────────────┐
              ▼                               ▼
     ┌────────────────┐              ┌────────────────┐
     │  v1 (Env-based)│              │ v2 (Env-less)  │
     │  replBridge.ts │              │ remoteBridge   │
     │                │              │ Core.ts        │
     │ Environments   │              │                │
     │ API 调度       │              │ POST /bridge   │
     │                │              │ 获取 worker_jwt│
     │ HybridTransport│              │                │
     │ (WS+POST)      │              │ SSETransport   │
     │                │              │ + CCRClient    │
     └────────────────┘              └────────────────┘
```

**v1 路径**通过 Environments API 实现任务调度：注册环境 → 轮询工作 → 确认 → 心跳。这种设计类似传统的任务队列，但增加了 WebSocket 实时通道。

**v2 路径**简化了调度模型：直接 POST `/bridge` 获取 `worker_jwt`，使用 SSE 接收消息、HTTP POST 发送消息。消除了环境注册和轮询的开销。

#### 2.2.2 消息路由与去重

Bridge 系统的核心消息路由器位于 `bridgeMessaging.ts`，实现了精细的消息过滤和去重机制：

```typescript
// 消息资格过滤 — 仅转发特定类型到服务器
function isEligibleBridgeMessage(m: Message): boolean {
  if ((m.type === 'user' || m.type === 'assistant') && m.isVirtual) return false
  return m.type === 'user' || m.type === 'assistant' ||
    (m.type === 'system' && m.subtype === 'local_command')
}

class BoundedUUIDSet {
  private readonly ring: (string | undefined)[]  // FIFO 环形缓冲区
  private readonly set = new Set<string>()        // O(1) 查找

}
```

**双缓冲区去重设计**：
- `recentPostedUUIDs`：过滤自己发出消息的回声（echo suppression）
- `recentInboundUUIDs`：过滤服务器的消息重放（replay protection）

两个缓冲区均使用 FIFO 环形缓冲区，O(1) 的添加和查找复杂度，内存占用恒定（2000 × UUID 长度）。

#### 2.2.3 控制请求协议

服务器可通过 Bridge 向本地 CLI 发送控制请求，实现远程操控：

| 控制类型 | 功能 | 安全约束 |
|---------|------|---------|
| `initialize` | 会话握手，交换能力声明 | 无 |
| `set_model` | 切换 AI 模型 | Outbound-only 模式下拒绝 |
| `set_max_thinking_tokens` | 调整思考预算 | Outbound-only 模式下拒绝 |
| `set_permission_mode` | 切换权限模式 | 需策略验证 |
| `interrupt` | 中断当前执行 | Outbound-only 模式下拒绝 |

**Outbound-only 模式**是一种安全降级——当连接仅用于向服务器推送消息时，所有可变控制请求均被拒绝，仅 `initialize` 正常响应。这防止了恶意服务器通过 Bridge 操控本地 CLI。

#### 2.2.4 四层认证体系

Bridge 系统实现了四层递进的认证机制：

```
第1层: OAuth Bearer Token — 主认证，所有 API 调用携带
    │
第2层: Environment Secret — v1 轮询专用，环境注册后返回
    │
第3层: Session Ingress Token / Worker JWT — 会话级，v2 通过 POST /bridge 获取
    │
第4层: Trusted Device Token — 可信设备认证，90天滚动过期，keychain 持久化
```

### 2.3 Remote (CCR) 远程会话管理

#### 2.3.1 WebSocket + HTTP 混合传输

Remote 模式采用"WebSocket 订阅 + HTTP 发送"的非对称传输架构：

```
                    Claude Code CLI
                    ┌──────────────┐
                    │  Remote      │
                    │  Session     │
                    │  Manager     │
                    └──┬────────┬──┘
                       │        │
          WS 订阅(接收) │        │ HTTP POST (发送)
                       ▼        ▼
              ┌─────────────────────────┐
              │   api.anthropic.com     │
              │   /v1/sessions/ws/{id}/ │
              │   subscribe             │
              │   /v1/sessions/{id}/    │
              │   events                │
              └─────────────────────────┘
```

**设计理由**：WebSocket 长连接适合服务器向客户端推送消息（LLM 流式输出、权限请求），HTTP POST 适合客户端向服务器发送消息（用户输入、权限响应），避免了 WebSocket 双向通信中的消息排队问题。

#### 2.3.2 WebSocket 生命周期管理

```typescript
// 连接参数
URL: wss://api.anthropic.com/v1/sessions/ws/{sessionId}/subscribe
     ?organization_uuid={orgUuid}
认证: Authorization: Bearer {token}
心跳: PING_INTERVAL_MS = 30000 (30秒)

// 重连策略
普通断线: 指数退避 2s × attempt, 最多 5 次
4001 (session not found): 线性退避 2s × retries, 最多 3 次
  — compaction 期间瞬态，会话短暂不可见
4003 (unauthorized): 立即终止，不重连

// 消息验证
宽松验证: 仅检查 type 字段为 string
  — 设计理由: 后端可能先于客户端更新发送新消息类型
  — 硬编码白名单会静默丢弃新消息
```

#### 2.3.3 远程权限桥接

Remote 模式下的权限审批流程涉及一个精巧的"合成消息"机制：

```
CCR 容器中 Agent 要使用工具
    │
    ▼
CCR 后端通过 WS 发送 control_request { subtype: 'can_use_tool', ... }
    │
    ▼
SessionsWebSocket 接收 → RemoteSessionManager.handleControlRequest()
    │
    ▼
存入 pendingPermissionRequests Map → 触发 onPermissionRequest 回调
    │
    ▼
createSyntheticAssistantMessage() — 从 SDKControlPermissionRequest 构造假的 AssistantMessage
    │
    ▼
createToolStub() — 为远程 MCP 工具创建本地 Tool 存根（CCR 有但本地不知道的工具）
    │
    ▼
本地权限 UI (ToolUseConfirm) 展示给用户
    │
    ▼
用户决定 → respondToPermissionRequest() → control_response → WS 发回 CCR
```

**合成消息的设计必要性**：在 CCR 模式下，Agent 运行在 Anthropic 的云端容器中，本地 CLI 没有 `AssistantMessage` 和 `Tool` 对象。但本地的权限 UI 组件需要这些对象才能渲染。因此 `remotePermissionBridge.ts` 创建了"合成"版本——包含足够的信息驱动 UI，但不关联真实的 Agent 状态。

### 2.4 Direct Connect 直连服务器

#### 2.4.1 与 Remote 模式的关键差异

Direct Connect 是面向自托管场景的轻量级方案：

| 特性 | Direct Connect | Remote (CCR) |
|------|---------------|-------------|
| **消息收发** | 同一 WebSocket 双向 | WS 接收 + HTTP POST 发送 |
| **消息格式** | NDJSON（换行分隔） | 单条 JSON |
| **认证** | 可选 Bearer Token | 必须 OAuth |
| **重连** | 无自动重连 | 指数退避重连 |
| **心跳** | 无 | 30s ping |
| **会话持久化** | `~/.claude/server-sessions.json` | 服务端管理 |
| **闲置超时** | 可配置 `idleTimeoutMs` | 无 |

#### 2.4.2 会话状态机

```typescript
type SessionState = 'starting' | 'running' | 'detached' | 'stopping' | 'stopped'

// 状态转换
starting → running    (Agent 就绪)
running  → detached   (客户端断开，Agent 继续运行)
detached → running    (客户端重连)
running  → stopping   (收到停止请求)
stopping → stopped    (Agent 退出)
```

**Detached 状态**是 Direct Connect 的独特设计——允许客户端断开后 Agent 继续在后台运行，客户端重连后可以恢复交互。这为"长时间运行任务 + 按需查看"的场景提供了支持。

#### 2.4.3 消息过滤

Direct Connect 对消息进行了更积极的过滤，丢弃以下类型：
- `streamlined_text` — 精简文本（仅用于 SDK 消费）
- `streamlined_tool_use_summary` — 精简工具摘要
- `post_turn_summary` — 轮次后摘要

这些消息类型是 SDK 优化输出格式，对终端用户无意义，过滤后减少了 WebSocket 流量。

### 2.5 SDK 控制协议

#### 2.5.1 协议概述

SDK 控制协议是 Claude Code 与外部 SDK 消费者（如 Python SDK）之间的通信接口，基于 stdin/stdout JSON 流：

```
┌──────────────┐     stdin (SDKMessage)     ┌──────────────┐
│   SDK 消费者  │ ─────────────────────────► │  Claude Code │
│  (Python等)  │ ◄───────────────────────── │   CLI 进程   │
└──────────────┘    stdout (StreamEvent)     └──────────────┘
```

#### 2.5.3 消息类型联合体

SDK 定义了丰富的消息类型联合体 `SDKMessageSchema`，用于 CLI 向 SDK 消费者推送状态：

```typescript
SDKMessage =
  | { type: 'assistant' }         // 助手消息（含 API 响应）
  | { type: 'user' }              // 用户消息
  | { type: 'user_replay' }       // 用户消息回放
  | { type: 'result' }            // 会话结果
  | { type: 'system', subtype: 'init' }           // 初始化
  | { type: 'system', subtype: 'status' }         // 状态变更
  | { type: 'system', subtype: 'compact_boundary' } // 压缩边界
  | { type: 'system', subtype: 'api_retry' }      // API 重试
  | { type: 'system', subtype: 'hook_started' }   // Hook 生命周期
  | { type: 'system', subtype: 'task_started' }   // 任务事件
  | { type: 'system', subtype: 'session_state_changed' } // 会话状态
  | { type: 'stream_event' }      // 流式事件
  | { type: 'tool_progress' }     // 工具执行进度
  | { type: 'auth_status' }       // 认证状态
  | { type: 'rate_limit_event' }  // 速率限制
  | { type: 'tool_use_summary' }  // 工具使用摘要
  | { type: 'elicitation_complete' } // MCP 输入完成
  | { type: 'prompt_suggestion' } // 提示建议
```

**设计特点**：所有消息类型通过 `type` + `subtype` 的 discriminated union 组织，Zod Schema 在运行时验证。这种设计使得新增消息类型不需要修改解析逻辑——只需扩展联合体。

#### 2.5.4 Hook 系统（28 种事件）

SDK 协议定义了 28 种 Hook 事件，每种有独立的 Input/Output Schema：

**生命周期 Hooks**：`SessionStart`, `SessionEnd`, `Stop`, `StopFailure`

**工具 Hooks**：`PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `PermissionRequest`, `PermissionDenied`

**子代理 Hooks**：`SubagentStart`, `SubagentStop`

#### 2.6.5 OAuth 2.0 认证流程

MCP Client 实现了完整的 OAuth 2.0 + PKCE 流程：

```
1. 检查配置的 authServerMetadataUrl
2. RFC 9728: 探测 /.well-known/oauth-protected-resource → authorization_servers
3. RFC 8414: 对 AS URL 探测 /.well-known/oauth-authorization-server
4. 回退: 路径感知的 RFC 8414 探测
5. PKCE: 生成 code_verifier + code_challenge
6. 授权: 打开浏览器，用户授权
7. 回调: 接收 authorization_code
8. 换取: code + code_verifier → access_token + refresh_token
```

**Token 安全存储**：使用 `SecureStorage`（macOS Keychain / Linux secret-service），存储键为 `sha256(name+type+url+headers)[:16]`。

**Step-Up 认证**（CIMD/SEP-991）：检测 403 `insufficient_scope` 响应，通过 `markStepUpPending(scope)` 标记，`tokens()` 返回时省略 `refresh_token`，迫使 SDK 走重新授权流程。

#### 2.6.6 Channel 系统（双向通知）

Channel 允许 MCP Server 向对话中推送用户消息（如 Slack、Telegram）：

```
MCP Server ──notifications/claude/channel──► Claude Code ──<channel> tag──► LLM
                                                    │
                                              enqueue() ──► SleepTool polls
```

**七层门控检查**：

```
1. capability: 服务器必须声明 experimental['claude/channel']
2. disabled: 运行时开关 tengu_harbor
3. auth: 必须有 claude.ai OAuth token
4. policy: Teams/Enterprise 需要 channelsEnabled: true
5. session: 必须在 --channels 列表中
6. marketplace: 插件来源必须匹配用户指定的 marketplace
7. allowlist: 必须在 GrowthBook ledger 或 org allowedChannelPlugins 中
```

**权限中继**（远程设备审批工具权限）：

```
1. Claude Code 发送 permission_request 到 MCP Server
2. Server 格式化消息发送给用户（如手机 Slack）
3. 用户回复 "yes tbxkq"（5 字母确认码）
4. Server 解析并发送 permission {request_id, behavior}
5. Claude Code 匹配 pending map，完成权限审批
```

确认码设计：25 字母表（去掉 'l'），5 字符 = 25⁵ ≈ 980 万空间，FNV-1a 哈希 + base-25 编码，大小写不敏感（手机自动纠错友好）。

#### 2.6.7 连接生命周期

```
pending → connected
pending → failed
pending → needs-auth
connected → failed (onclose)
connected → pending (reconnection)
connected → disabled (user toggle)
disabled → pending (user toggle)
needs-auth → connected (user auth)
```

**自动重连**：指数退避 1s → 2s → 4s → 8s → 16s，最多 5 次。仅对远程传输（SSE/HTTP/WS）自动重连，stdio 和 sdk 类型不重连。

**进程清理**（stdio 传输）：信号升级序列 `SIGINT (100ms) → SIGTERM (400ms) → SIGKILL`，总超时 600ms。

### 3.1 Gateway Protocol v3

#### 3.1.1 协议概述

Gateway Protocol 是 OpenClaw 的核心通信协议，当前版本为 v3。它定义了客户端与网关之间的完整交互规范：

```
┌──────────────┐                    ┌──────────────────────┐
│  CLI Client  │                    │  Gateway Server      │
│  iOS Client  │◄─── WebSocket ────►│  WebChat Client      │
│  Node Client │    (JSON frames)   │  ACP Translator     │
└──────────────┘                    └──────────────────────┘
```

#### 3.1.2 帧格式（三帧联合类型）

```typescript
// 请求帧 — 客户端调用服务端方法
RequestFrame = {
  type: "req",
  id: string,          // 请求 ID，用于匹配响应
  method: string,      // 方法名，如 "chat.send"
  params?: unknown     // 方法参数
}

// 响应帧 — 服务端返回方法结果
ResponseFrame = {
  type: "res",
  id: string,          // 匹配请求 ID
  ok: boolean,         // 是否成功
  payload?: unknown,   // 成功时的数据
  error?: {            // 失败时的错误信息
    code: string,
    message: string,
    details?: unknown,
    retryable?: boolean,
    retryAfterMs?: number
  }
}

// 事件帧 — 服务端主动推送
EventFrame = {
  type: "event",
  event: string,       // 事件名，如 "session.message"
  payload?: unknown,
  seq?: number,        // 序列号（间隙检测）
  stateVersion?: number // 状态版本号
}
```

**设计特点**：
- 使用 `type` 字段作为判别联合（discriminated union），解析时一步分发
- 请求/响应通过 `id` 匹配，支持异步并发调用
- 事件帧携带 `seq` 序列号，客户端可检测消息间隙并请求状态同步
- 错误形状包含 `retryable` 和 `retryAfterMs`，为客户端重连提供指导

#### 3.1.3 连接握手协议

采用**挑战-响应**模式，在 WebSocket 连接建立后进行认证和能力协商：

```

  │                                               │
  │──── connect.challenge (nonce + timestamp) ───►│
  │                                               │
  │◄─── connect (minProtocol, maxProtocol,        │
  │      client, auth, device) ──────────────────│
```

基于 `KeyedAsyncQueue` 的串行化队列，确保同一 sessionKey 的所有操作顺序执行：

```typescript
  private readonly pendingBySession = new Map<string, number>()

  async run<T>(actorKey: string, op: () => Promise<T>): Promise<T> {
    return this.queue.enqueue(actorKey, op, {

      onSettle: () => { /* 计数递减 */ },

  }
```

### 3.4 工具权限与审批体系

#### 3.4.1 八级审批分类器

OpenClaw 实现了精细的工具审批分类系统：

```typescript
type AcpApprovalClass =
  | "readonly_scoped"    // 只读且限定在 cwd 内 → 自动批准
  | "readonly_search"    // 只读搜索 → 自动批准
  | "mutating"           // 变更操作 → 需审批
  | "exec_capable"       // 可执行命令 → 需审批
  | "control_plane"      // 控制平面操作 → 需审批
  | "interactive"        // 交互式操作 → 需审批
  | "other"              // 其他 → 需审批
  | "unknown"            // 未知 → 需审批
```

**分类算法**：

```typescript
function classifyAcpToolApproval(params): AcpApprovalClassification {
  // 1. 工具名交叉验证（meta / rawInput / title 三来源）
  //    不一致时降级为 unknown
  const toolName = resolveToolNameForPermission(params)

  // 2. 只读文件读取 + cwd 范围检查
  if (toolName === "read" && isTrustedToolId) {
    const autoApprove = isReadToolCallScopedToCwd(params, cwd)
    return { approvalClass: autoApprove ? "readonly_scoped" : "other", autoApprove }
  }

  // 3. 安全搜索工具
  if (SAFE_SEARCH_TOOL_IDS.has(toolName)) return { approvalClass: "readonly_search", autoApprove: true }

  // 4. Owner-only 工具
  if (ownerOnlyApprovalClass) return { approvalClass: ownerOnlyApprovalClass, autoApprove: false }

  // 5. 可执行工具
  if (EXEC_CAPABLE_TOOL_IDS.has(toolName)) return { approvalClass: "exec_capable", autoApprove: false }

  // 6. 控制平面工具
  if (CONTROL_PLANE_TOOL_IDS.has(toolName)) return { approvalClass: "control_plane", autoApprove: false }

  // 7. 变更检测
  if (isMutatingToolCall(toolName, rawInput)) return { approvalClass: "mutating", autoApprove: false }

  return { approvalClass: "other", autoApprove: false }
}
```

#### 3.4.2 两阶段 RPC 审批协议

Exec 工具采用两阶段审批协议，通过 Gateway RPC 实现：

```
阶段1: 注册审批请求
  Agent → Gateway: exec.approval.request { toolCallId, toolName, input, ... }
  Gateway → 用户: 推送 exec.approval.requested 事件

阶段2: 等待审批决策
  Agent → Gateway: exec.approval.waitDecision { requestId }
  用户 → Gateway: exec.approval.respond { requestId, decision }
  Gateway → Agent: 返回审批结果
```

**审批结果类型**：

```typescript
type ExecApprovalResult =
  | { kind: "denied" }       // 用户拒绝
  | { kind: "finished" }     // 命令完成（审批通过并执行完毕）
  | { kind: "completed" }    // 命令成功
  | { kind: "other" }        // 其他
```

#### 3.4.3 多层策略叠加模型

OpenClaw 采用 9 层策略叠加模型，策略按顺序依次过滤工具列表：

```
1. Profile 策略 (tools.profile) — 预设工具集 (minimal/coding/messaging/full)
2. Provider Profile (tools.byProvider.*.profile) — 按 AI Provider 覆盖
3. 全局 Allow/Deny (tools.allow / tools.deny) — 全局白名单/黑名单
4. 全局 Provider (tools.byProvider.*.allow) — 按 Provider 白名单
5. Agent 策略 (agents.<id>.tools.allow) — 按 Agent 白名单
6. Agent Provider (agents.<id>.tools.byProvider.*.allow) — 按 Agent+Provider
7. Group 策略 — 基于 groupId/channel 的群组策略
8. Sandbox 策略 — 沙箱环境工具限制
9. Subagent 策略 — 子代理深度限制
```

**匹配算法**：deny 优先于 allow，支持 glob 模式和工具组展开（`group:fs` → `["read", "write", "edit", "apply_patch"]`）。

## 五、深度对比：通信架构范式差异

### 4.1 协议设计哲学

| 维度 | Claude Code | OpenClaw |
|------|-------------|----------|
| **协议数量** | 5+ 种专用协议（Bridge/Remote/DirectConnect/SDK/MCP） | 1 种核心协议（Gateway v3）+ 1 种翻译协议（ACP） |
| **协议风格** | 各协议独立演进，通过适配器互联 | 统一核心协议，外部协议通过翻译层桥接 |
| **帧格式** | JSON（各协议格式不同） | 统一三帧联合类型（req/res/event） |
| **Schema 定义** | Zod（TypeScript 原生） | @sinclair/typebox + Ajv（JSON Schema 标准） |
| **版本协商** | GrowthBook 功能开关（运行时） | minProtocol/maxProtocol 范围协商（握手时） |
| **能力声明** | 隐式（通过功能开关推断） | 显式（HelloOk 中的 features 列表） |
| **方法数量** | ~21 种控制请求 + MCP 工具 | 131 个 RPC 方法 |
| **事件系统** | SDK 消息类型联合体 | 带序列号的事件帧 + 作用域守卫 |

**设计取舍分析**：

Claude Code 的"协议联邦"策略允许每个协议针对特定场景深度优化（如 Bridge 的 v1/v2 演进、Remote 的 WS+HTTP 混合传输），但代价是系统复杂度高——需要维护多个消息适配器（`sdkMessageAdapter.ts`、`bridgeMessaging.ts` 等），且不同协议之间的功能对齐困难。

OpenClaw 的"统一网关"策略通过单一协议简化了核心路径，131 个 RPC 方法提供了丰富的功能覆盖。但翻译层（ACP Translator）成为瓶颈——每条 ACP 消息需要经过 ACP → Gateway → 后端 的多层转换，增加了延迟和出错概率。

### 4.2 会话生命周期模型

| 维度 | Claude Code | OpenClaw |
|------|-------------|----------|
| **会话 ID 格式** | `session_*` (v1) / `cse_*` (v2) | UUID + sessionKey 分层 |
| **会话存储** | 服务端管理（CCR） / 文件系统（Direct Connect） | 内存（ACP Store） + JSON 文件（Session Store） |
| **崩溃恢复** | Bridge Pointer 文件（4h TTL） | 代际断连 + 5s 宽限期调和 |
| **会话恢复** | 重放历史消息 | 转录回放（最多 100 万条） |
| **并发控制** | 无显式机制 | Actor Queue（按 sessionKey 串行化） |
| **闲置管理** | 无（由服务端控制） | 24h TTL + LRU 驱逐 |
| **容量限制** | 无显式限制 | 最大 5000 会话 |
| **身份协调** | 无（单一来源） | 三来源合并（meta/handle/status） |
| **持久化绑定** | 无 | SHA-256 哈希绑定外部对话到会话 |

**关键差异**：

Claude Code 的会话管理相对简单——会话状态主要由服务端（CCR）维护，客户端通过 Pointer 文件实现崩溃恢复。这种设计适合"一次连接、一个会话"的模型。

OpenClaw 的会话管理则复杂得多——Actor Queue 保证了同一会话的操作串行化，身份协调解决了多个来源的 ID 冲突，持久化绑定实现了外部对话到 ACP 会话的自动映射。这种设计适合"多通道、多会话、长时间运行"的模型。

### 4.3 权限与安全模型

| 维度 | Claude Code | OpenClaw |
|------|-------------|----------|
| **权限分类** | 无显式分类 | 8 级审批分类器 |
| **审批流程** | 单次交互式确认 | 两阶段 RPC 审批 |
| **自动审批** | 基于规则（allowlist） | 基于分类（readonly_scoped/search） |
| **远程审批** | Channel 确认码（5 字母） | Gateway RPC 事件推送 |
| **策略层级** | 2 层（全局 + 项目） | 9 层叠加 |
| **工具循环检测** | 无 | 3 种检测器 |
| **认证模式** | OAuth + JWT + Trusted Device | 6 种（Token/Password/Device/Bootstrap/Tailscale/TrustedProxy） |
| **认证限流** | 无显式限流 | 滑动窗口限流（可配置） |
| **连接预算** | 无 | Preauth 连接预算（防资源耗尽） |
| **TLS 安全** | 标准 HTTPS/WSS | 阻止非环回明文连接 + TLS 指纹验证 |
| **设备认证** | Trusted Device（keychain） | 公钥签名验证（2 分钟时间偏移容忍） |

**关键差异**：

Claude Code 的权限模型相对简单——基于 allowlist 的规则匹配，用户逐个批准工具使用。这适合"开发者桌面"场景，用户可以直接看到并控制每个操作。

OpenClaw 的权限模型则面向"多用户、多通道、长时间运行"场景——8 级分类器实现了细粒度的自动审批，两阶段 RPC 审批支持异步远程审批，9 层策略叠加实现了组织级的精细管控。

### 4.4 MCP 集成策略

| 维度 | Claude Code | OpenClaw |
|------|-------------|----------|
| **MCP 角色** | Server + Client 双角色 | 不支持 MCP（ACP 替代） |
| **配置层级** | 7 层优先级 | N/A |
| **企业策略** | Denylist + Allowlist + Plugin-Only | N/A |
| **传输类型** | 8 种（stdio/sse/http/ws/sdk 等） | N/A |
| **OAuth 认证** | 完整 PKCE 流程 + Step-Up | N/A |
| **工具发现** | LRU 缓存（20 server） | N/A |
| **Channel 系统** | 7 层门控 + 确认码权限中继 | N/A |
| **进程内通信** | InProcessTransport（对等链接） | N/A |

**设计取舍**：

Claude Code 深度集成了 MCP 协议，将其作为工具生态的核心扩展点。MCP 的标准化使得第三方工具可以无缝接入，OAuth 2.0 认证确保了安全性，Channel 系统实现了跨设备权限中继。

OpenClaw 选择不集成 MCP，而是通过 ACP 协议和插件系统实现类似功能。这种选择避免了 MCP 的复杂性，但也失去了与 MCP 生态的互操作性。

### 4.5 错误处理与韧性

| 维度 | Claude Code | OpenClaw |
|------|-------------|----------|
| **重连策略** | 指数退避（1s→16s，5次） | 指数退避（1s→60s，无上限） |
| **特殊错误码** | 4001 (compaction 瞬态), 4003 (永久) | 1000/1006/1008/1012 (语义化) |
| **消息验证** | 宽松（仅检查 type 字段） | 严格（Ajv 全量验证） |
| **错误形状** | 简单 status + message | 结构化 code + message + retryable + retryAfterMs |
| **事件间隙检测** | 无 | seq 序列号 + onGap 回调 |
| **状态同步** | 无显式机制 | HelloOk 初始快照 + stateVersion |
| **进程清理** | SIGINT→SIGTERM→SIGKILL (600ms) | AbortController + Actor Queue |
| **死 Token 退避** | 跨进程持久化（3 次失败冷却） | 设备令牌自动清除 |
| **故障注入** | Ant-only bridgeDebug | 无 |

**关键差异**：

Claude Code 采用"宽容验证"策略——消息仅检查 `type` 字段，允许后端先于客户端更新。这提高了前向兼容性，但牺牲了类型安全。

OpenClaw 采用"严格验证"策略——所有帧和参数通过 Ajv 全量验证。这提供了更强的类型安全，但可能导致新版本客户端与旧版本服务端不兼容。

OpenClaw 的事件间隙检测（`seq` 序列号）和状态同步机制（HelloOk 快照 + stateVersion）是 Claude Code 所缺乏的——这些机制使得客户端可以在断线重连后精确同步到最新状态，而不是依赖全量重传。

## 六、架构洞察与设计模式提炼

### 5.1 通信架构的核心设计模式

#### 模式1：协议翻译器 (Protocol Translator)

**应用位置**：OpenClaw `AcpGatewayAgent`、Claude Code `sdkMessageAdapter`

两个项目都实现了协议翻译层，但方式不同：
- Claude Code：多个点对点适配器（SDK↔Internal、Bridge↔Internal、Remote↔Internal）
- OpenClaw：一个统一翻译器（ACP↔Gateway）

```
Claude Code:                          OpenClaw:
  SDK ──► Adapter ──► Core           ACP ──► Translator ──► Gateway ──► Core
  Bridge ──► Adapter ──► Core
  Remote ──► Adapter ──► Core
```

**启示**：当系统需要支持多种外部协议时，"统一翻译层"比"点对点适配器"更易维护，但需要在翻译层中处理所有协议的语义差异。

#### 模式2：代际计数器 (Generation Counter)

**应用位置**：Claude Code JWT 刷新、OpenClaw 断连恢复、认证会话管理

```typescript
// 用递增的 generation 标记每次状态变更
// 异步回调中检查 generation 是否匹配当前值
// 不匹配则丢弃过期回调
```

**启示**：在异步系统中处理"取消"和"竞态"问题时，代际计数器比 `AbortController` 更轻量，且不需要传递 signal 引用。

#### 模式3：环形缓冲区去重 (Ring Buffer Dedup)

**应用位置**：Claude Code `BoundedUUIDSet`

```typescript
// FIFO 环形缓冲区 + HashSet
// O(1) 添加和查找
// 恒定内存占用
// 自动驱逐最旧条目
```

**启示**：当需要在大流量消息流中去重时，环形缓冲区比 LRU Cache 更适合——它不需要为每个条目维护独立的过期时间，且驱逐策略更可预测。

#### 模式4：签名缓存 (Signature Cache)

**应用位置**：OpenClaw `appliedControlSignature`、Claude Code MCP 去重签名

```typescript
// 将配置/选项序列化为签名
// 仅当签名变化时才重新应用
// 避免重复推送相同的控制指令
```

**启示**：当需要避免重复应用相同的配置时，签名缓存比深度比较更高效——它将 O(n) 的比较降为 O(1) 的字符串比较。

#### 模式5：作用域守卫扇出 (Scoped Fan-out)

**应用位置**：OpenClaw 事件广播系统

```typescript
// 每个事件类型关联所需的权限 scope
// 广播时检查每个客户端的 scopes
// 只有具备对应权限的客户端才能收到事件
```

**启示**：在多租户系统中，事件广播需要按权限过滤。作用域守卫比"每个客户端独立订阅"更简单，且不容易出现权限遗漏。

#### 模式6：挑战-响应握手 (Challenge-Response Handshake)

**应用位置**：OpenClaw Gateway 连接、Claude Code Trusted Device

```typescript
// 服务端发送 nonce + timestamp
// 客户端用私钥签名后返回
// 服务端验证签名和时间偏移
```

**启示**：在 WebSocket 连接上实现认证时，挑战-响应比简单的 Token 认证更安全——它防止了重放攻击，且可以通过时间偏移检测时钟偏移。

### 5.2 安全设计的层次化思路

两个项目在安全设计上都采用了**纵深防御**策略，但侧重点不同：

**Claude Code 的安全层次**：
```
L1: OAuth 认证（身份验证）
L2: JWT + Worker Token（会话隔离）
L3: Trusted Device（设备信任）
L4: Organization Policy（组织策略）
L5: Bridge Pointer TTL（会话过期）
L6: Debug 日志脱敏（信息泄露防护）
```

**OpenClaw 的安全层次**：
```
L1: 多模式认证（6 种认证方式）
L2: 角色与 Scope（最小权限）
L3: 设备公钥签名（设备信任）
L4: 认证限流（暴力破解防护）
L5: Preauth 连接预算（资源耗尽防护）
L6: TLS 明文阻止（传输安全）
L7: 9 层工具策略（操作权限）
L8: 输入大小限制（DoS 防护，2MB）
```

OpenClaw 的安全层次更全面，特别是在抗攻击方面（限流、连接预算、输入大小限制）。Claude Code 的安全层次更侧重于身份认证和会话隔离。

### 5.3 可靠性设计的对比

| 可靠性机制 | Claude Code | OpenClaw |
|-----------|-------------|----------|
| **消息去重** | 环形缓冲区（UUID 级） | 序列号（事件级） |
| **状态同步** | 无显式机制 | HelloOk 快照 + stateVersion |
| **断线恢复** | Crash Recovery Pointer | 代际断连 + 宽限期调和 |

| **背压保护** | 无 | dropIfSlow 事件广播 |

## 七、总结与核心发现

### 7.1 核心发现

1. **协议联邦 vs 统一网关**：Claude Code 的多协议策略提供了更好的场景适配性，但增加了系统复杂度；OpenClaw 的统一网关策略简化了核心路径，但翻译层成为瓶颈。

2. **会话模型的复杂度梯度**：Claude Code 的"单会话、服务端管理"模型适合桌面开发场景；OpenClaw 的"多会话、客户端管理"模型适合多通道长时间运行场景。

3. **权限精细度的取舍**：Claude Code 的简单 allowlist 适合信任环境；OpenClaw 的 8 级分类器 + 9 层策略适合多租户环境。

4. **MCP 集成的战略价值**：Claude Code 深度集成 MCP 获得了生态优势；OpenClaw 的 ACP 协议提供了更大的自主控制权。

5. **可靠性机制的互补**：两个项目在消息去重、状态同步、断线恢复等方面各有优势，可以互相借鉴。

### 7.2 通信架构的共性设计原则

通过对比两个项目的通信架构，可以提炼出以下共性设计原则：

1. **传输与协议分离**：无论采用哪种传输方式（WebSocket/HTTP/stdio），消息格式和语义应该独立于传输层。
2. **显式能力协商**：连接建立时应该交换能力声明，避免假设对方支持的功能。
3. **消息去重是必须的**：在任何双向通信系统中，回声抑制和重放防护都是基本需求。
4. **错误信息应该指导客户端行为**：`retryable` + `retryAfterMs` 比简单的错误码更有价值。
5. **安全需要纵深防御**：单一的安全机制不够，需要多层叠加。
6. **状态同步应该增量式**：全量重传在大型系统中代价过高，序列号 + 增量同步更高效。

## 八、为后续轮次准备的关键发现

### 8.1 第 7 轮（多模态能力与扩展架构）应关注的问题

1. **Claude Code 的 InProcessTransport（进程内 MCP 通信）是否可以扩展为通用的多模态内容传输通道？** 它的对等链接模式是否适合传输图片、音频等二进制内容。
2. **OpenClaw 的 ACP 协议声明 `image: true, audio: false`，其图片输入是如何在 ACP → Gateway → 后端的翻译链中传递的？** 是否存在编解码损失。
3. **Claude Code 的 Channel 系统（7 层门控 + 确认码权限中继）是否可以作为一种通用的跨设备内容推送机制？** 其背压保护和作用域守卫设计是否适用于多模态内容。
4. **OpenClaw 的两阶段 RPC 审批协议是否支持审批多模态工具调用（如浏览器截图、图片生成）？** 审批分类器如何处理这类工具。

### 8.2 第 8 轮（安全体系纵深防御）应关注的问题

1. **Claude Code 的 Bridge 四层认证体系与 OpenClaw 的六种认证模式，哪种更适合多租户 SaaS 部署？**
2. **OpenClaw 的 Preauth 连接预算和认证限流机制的具体实现细节是否可以独立提取为通用中间件？**
3. **Claude Code 的 MCP OAuth 2.0 + PKCE 流程中的 Step-Up 认证（CIMD/SEP-991）是否可以应用于其他需要渐进式授权的场景？**
