## 一、三层错误隔离架构

### 1.1 架构全景

claude-code的错误处理不是单一的try-catch，而是一个**三层递进的隔离体系**。每一层关注不同的故障域，使用不同的隔离策略，确保单点故障不会向上传播导致系统级崩溃。

```
┌──────────────────────────────────────────────────────────────────┐
│                    三层错误隔离架构                                 │
│                                                                  │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │  Layer 3: System-Level Isolation (系统级隔离)               │  │
│  │  ┌──────────────────────────────────────────────────────┐  │  │
│  │  │  Circuit Breaker + Cause Chain Tracking              │  │  │
│  │  │  - 连续失败触发熔断                                   │  │  │
│  │  │  - 因果链追踪（cause chain）                          │  │  │
│  │  │  - 优雅降级而非级联崩溃                               │  │  │
│  │  └──────────────────────────────────────────────────────┘  │  │
│  │                            ▲                               │  │
│  │  ┌─────────────────────────┼────────────────────────────┐  │  │
│  │  │  Layer 2: Query-Level Isolation (查询级隔离)          │  │  │
│  │  │  ┌────────────────────────────────────────────────┐  │  │  │
│  │  │  │  classifyAxiosError() → 5类错误分类            │  │  │  │
│  │  │  │  - auth → 认证失败策略                          │  │  │  │
│  │  │  │  - timeout → 超时降级策略                       │  │  │  │
│  │  │  │  - network → 网络重连策略                       │  │  │  │
│  │  │  │  - http → HTTP状态码策略                        │  │  │  │
│  │  │  │  - other → 通用兜底策略                         │  │  │  │
│  │  │  └────────────────────────────────────────────────┘  │  │  │
│  │  └─────────────────────────────────────────────────────┘  │  │
│  │                            ▲                               │  │
│  │  ┌─────────────────────────┼────────────────────────────┐  │  │
│  │  │  Layer 1: Tool-Level Isolation (工具级隔离)           │  │  │
│  │  │  ┌────────────────────────────────────────────────┐  │  │  │
│  │  │  │  Tool Error → tool_result (NOT exception)      │  │  │  │
│  │  │  │  - 错误作为输出返回，而非抛出异常                │  │  │  │

│  │  │  │  - 模型可自主调整策略（重试/换路径/放弃）        │  │  │  │
│  │  │  └────────────────────────────────────────────────┘  │  │  │

```

### 1.2 Tool-Level Isolation：错误即数据

这是整个错误隔离体系中最精妙的设计。传统做法中，工具执行失败会抛出异常，由调用方捕获处理。claude-code的做法完全不同：

```
┌──────────────────────────────────────────────────────────────┐
│              传统模式 vs claude-code 模式                       │
│                                                              │
│  [传统模式]                                                   │
│  Tool.execute() → throw Error → catch → 中断Agent循环         │
│  问题: 异常打断了模型的推理链，模型无法"看到"错误详情            │
│                                                              │
│  [claude-code 模式]                                           │
│  Tool.execute() → return tool_result(isError: true, ...)     │
│                   → 模型收到 tool_result                      │
│                   → 模型"看到"错误信息                         │
│                   → 模型自主决定下一步策略                      │
│                                                              │
│  关键区别: 错误是模型输入的一部分，而非控制流中断                 │
└──────────────────────────────────────────────────────────────┘
```

**核心机制**：工具执行失败时，错误被封装为`tool_result`消息返回给模型。模型在下一轮推理中看到这个错误结果，就像看到任何其他工具输出一样。模型可以根据错误内容自主决定：
- 重试相同的工具调用（可能修改参数）
- 切换到替代工具
- 向用户报告错误并请求指导
- 放弃当前路径，尝试其他方案

**关键洞察**：这种设计将错误处理从"程序逻辑"提升到"推理能力"层面。模型不是被动地被异常中断，而是主动地参与错误恢复决策。这赋予了Agent系统真正的自适应能力。

### 1.3 Query-Level Isolation：分类驱动的降级

API调用层面的错误通过`classifyAxiosError()`进行分类，不同类别触发不同的降级策略：

```
┌──────────────────────────────────────────────────────────────────┐
│                classifyAxiosError() 分类体系                      │
│                                                                  │
│  AxiosError / UnknownError                                       │
│       │                                                          │
│       ├── code === 'ECONNABORTED' ──→ timeout                    │
│       │   策略: 指数退避重试，延长超时                             │
│       │                                                          │
│       ├── !response ──→ network                                  │
│       │   (code: ECONNREFUSED/ENOTFOUND/ENETUNREACH)            │
│       │   策略: 网络重连，检查DNS/代理                            │
│       │                                                          │
│       ├── response.status === 401/403 ──→ auth                   │
│       │   策略: 刷新认证，提示用户检查凭证                         │
│       │                                                          │
│       ├── response.status === 529 ──→ overloaded                 │
│       │   策略: 前台重试 / 后台放弃（差异化处理）                  │
│       │                                                          │
│       ├── response.status >= 500 ──→ server_error                │
│       │   策略: 指数退避重试，考虑模型降级                         │
│       │                                                          │
│       └── 其他 ──→ other                                         │
│           策略: 记录日志，通用兜底                                 │
└──────────────────────────────────────────────────────────────────┘
```

**设计模式**：分类策略（Classifier Strategy）——通过错误分类将 heterogeneous 的错误空间映射到 homogeneous 的策略空间，每个策略独立演化。

### 1.4 System-Level Isolation：熔断器与因果链

当错误在查询级无法被消化，开始累积时，系统级隔离机制介入：

```
┌──────────────────────────────────────────────────────────────────┐
│                  Circuit Breaker 状态机                            │
│                                                                  │
│                    ┌──────────────┐                               │
│                    │    CLOSED    │  ← 正常状态，请求通过           │
│                    │  (正常通行)   │                               │
│                    └──────┬───────┘                               │
│                           │                                       │
│                    连续失败 >= threshold                           │
│                           │                                       │
│                           v                                       │
│                    ┌──────────────┐                               │
│                    │    OPEN      │  ← 熔断状态，请求被拒绝         │
│                    │  (熔断拒绝)   │  → 触发降级策略                 │
│                    └──────┬───────┘                               │
│                           │                                       │
│                    timeout 到期                                    │
│                           │                                       │
│                           v                                       │
│                    ┌──────────────┐                               │
│                    │  HALF_OPEN   │  ← 探测状态，放行少量请求       │
│                    │  (探测恢复)   │                               │
│                    └──────┬───────┘                               │
│                           │                                       │
│              ┌────────────┼────────────┐                          │
│              │            │            │                          │
│              v            v            v                          │
│         探测成功      探测失败      探测失败                        │
│         → CLOSED     → OPEN       → OPEN                         │
│                                                                  │
│  Cause Chain Tracking:                                            │
│  Error(A) ← Error(B) ← Error(C) ← ...                           │
│  每个错误携带前一个错误的引用，形成因果链                           │
│  用于诊断：定位根因，而非只看到表面症状                             │
└──────────────────────────────────────────────────────────────────┘
```

**关键洞察**：熔断器不仅仅是"开关"，它配合因果链追踪（cause chain）提供了完整的故障诊断能力。当系统降级时，开发者可以通过因果链快速定位根因，而不是只看到"服务不可用"的表面症状。

## 二、withRetry AsyncGenerator 重试引擎

### 2.1 核心设计

`withRetry<T>()`是一个AsyncGenerator，它将API调用包装在智能重试逻辑中。选择AsyncGenerator而非普通异步函数的原因是：重试过程是**流式的**——每次重试的中间状态（等待中、重试中、成功、失败）都可以被yield出来，供上层消费。

```
┌──────────────────────────────────────────────────────────────────┐
│                  withRetry<T>() 架构                              │
│                                                                  │
│  调用方                                                          │
│    │                                                            │
│    │  for await (const event of withRetry(fn, config)) {        │
│    │    // 处理每个事件                                           │
│    │  }                                                         │
│    │                                                            │
│    v                                                            │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │              withRetry AsyncGenerator                       │  │
│  │                                                            │  │
│  │  ┌──────────┐   ┌──────────────┐   ┌──────────────────┐  │  │
│  │  │ 初始调用  │ → │ 错误分类      │ → │ 策略决策          │  │  │
│  │  │ fn()     │   │ classifyError│   │ shouldRetry?     │  │  │
│  │  └──────────┘   └──────────────┘   └────────┬─────────┘  │  │
│  │                                              │            │  │
│  │                                    ┌─────────┴─────────┐  │  │
│  │                                    │                   │  │  │
│  │                                    v                   v  │  │
│  │                            ┌──────────────┐   ┌──────────┐ │  │
│  │                            │ yield retry  │   │ yield    │ │  │
│  │                            │ event       │   │ result   │ │  │
│  │                            │ → 等待退避   │   │ → 结束   │ │  │
│  │                            └──────┬───────┘   └──────────┘ │  │
│  │                                   │                        │  │
│  │                                   v                        │  │
│  │                            ┌──────────────┐               │  │
│  │                            │ 重新调用 fn() │               │  │
│  │                            └──────────────┘               │  │
│  └────────────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────────────┘
```

### 2.2 前台 vs 后台差异化重试

这是整个重试策略中最关键的设计决策。claude-code将重试分为两种截然不同的场景：

```
┌──────────────────────────────────────────────────────────────────┐
│              前台 vs 后台 差异化重试策略                            │
│                                                                  │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │  FOREGROUND (用户正在等待)                                  │  │
│  │                                                            │  │
│  │  来源: FOREGROUND_529_RETRY_SOURCES                        │  │
│  │  行为: 529错误 → 完整重试（指数退避 + jitter）               │  │
│  │  原因: 用户在终端前等待，延迟是可感知的                       │  │
│  │       重试成功 = 用户满意                                   │  │
│  │       重试失败 = 用户看到错误（比静默丢失好）                 │  │
│  │                                                            │  │
│  │  重试参数:                                                  │  │
│  │  - 初始延迟: 1s                                             │  │
│  │  - 最大延迟: 32s (cap)                                      │  │
│  │  - 退避因子: 2x (指数)                                      │  │
│  │  - Jitter: ±25%                                            │  │
│  │  - Retry-After: 优先遵守服务端建议                           │  │
│  └────────────────────────────────────────────────────────────┘  │
│                                                                  │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │  BACKGROUND (后台任务)                                      │  │
│  │                                                            │  │
│  │  来源: 非 FOREGROUND_529_RETRY_SOURCES                     │  │
│  │  行为: 529错误 → 直接放弃（ABANDON）                        │  │
│  │  原因: 529 = 服务过载                                       │  │
│  │       后台重试 → 3-10x 网关放大效应                          │  │
│  │       → 加剧服务过载 → 恶性循环                              │  │
│  │                                                            │  │
│  │  网关放大效应:                                               │  │
│  │  1个后台任务重试 → 经过网关 → 变成3-10个请求                  │  │
│  │  100个后台任务同时重试 → 300-1000个请求冲击服务               │  │
│  │  → 本来想帮忙，结果把服务打挂了                              │  │
│  └────────────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────────────┘
```

**关键洞察**：后台任务放弃529重试不是"偷懒"，而是**系统级自我保护**。在分布式系统中，重试风暴（retry storm）是导致级联故障的主要原因之一。claude-code通过区分前台/后台，在用户体验和系统稳定性之间找到了平衡点。

### 2.3 指数退避 + 25% Jitter

```typescript
// getRetryDelay() 核心逻辑
function getRetryDelay(
  attempt: number,
  retryAfterHeader: string | null,
): number {
  // 优先级1: 服务端 Retry-After 头
  if (retryAfterHeader) {
    const serverSuggested = parseRetryAfter(retryAfterHeader)
    if (serverSuggested > 0) {
      return Math.min(serverSuggested, MAX_RETRY_DELAY) // 32s cap
    }
  }

  // 优先级2: 指数退避 + 25% jitter
  const baseDelay = Math.min(
    INITIAL_RETRY_DELAY * Math.pow(2, attempt - 1), // 1s * 2^(n-1)
    MAX_RETRY_DELAY,                                  // 32s cap
  )

  // 25% jitter: 防止多个客户端同时重试（惊群效应）
  const jitter = baseDelay * 0.25
  const min = baseDelay - jitter
  const max = baseDelay + jitter

  return min + Math.random() * (max - min)
}
```

**退避序列示例**：

| 尝试次数 | 基础延迟 | Jitter范围 | 实际延迟范围 |
|---------|---------|-----------|------------|
| 1 | 1s | ±0.25s | 0.75s - 1.25s |
| 2 | 2s | ±0.5s | 1.5s - 2.5s |
| 3 | 4s | ±1.0s | 3.0s - 5.0s |
| 4 | 8s | ±2.0s | 6.0s - 10.0s |
| 5 | 16s | ±4.0s | 12.0s - 20.0s |
| 6+ | 32s (cap) | ±8.0s | 24.0s - 40.0s |

**设计模式**：Decorrelated Jitter——25%的随机抖动确保了在大量客户端同时遇到错误时，它们的重试时间不会完全同步，从而避免惊群效应（thundering herd）。

### 2.4 持久重试：心跳分块休眠

对于需要长时间等待恢复的场景（如服务端大规模维护），claude-code实现了"持久重试"机制：

```
┌──────────────────────────────────────────────────────────────────┐
│                  持久重试机制 (Persistent Retry)                   │
│                                                                  │
│  普通重试:                                                       │
│  attempt 1 → wait 1s → attempt 2 → wait 2s → ... → MAX → FAIL  │
│  问题: 达到最大重试次数后彻底失败，用户需要手动重试                 │
│                                                                  │
│  持久重试:                                                       │
│  attempt 1 → wait 1s → attempt 2 → wait 2s → ...                │
│       → wait 32s (cap) → attempt N → wait 32s → ...             │
│       → [心跳休眠] 30s interval → attempt N+1 → ...             │
│       → 永不终止（直到成功或用户中断）                             │
│                                                                  │
│  心跳分块休眠 (Heartbeat Chunked Sleep):                         │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  总等待时间: 32s                                          │   │
│  │  分块: [10s] → yield heartbeat → [10s] → yield → [12s]  │   │
│  │                                                          │   │
│  │  好处:                                                   │   │
│  │  1. 上层可以在每个心跳点检查是否应该取消                     │   │
│  │  2. 用户可以看到"正在重试..."的进度反馈                     │   │
│  │  3. 避免长时间阻塞导致超时检测失效                          │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  尝试计数器钳位 (Attempt Counter Clamp):                         │
│  - attempt 永远不会溢出或达到"最大值"                             │
│  - 退避延迟在32s封顶后不再增长                                    │
│  - 重试次数无限，但频率受控                                       │
└──────────────────────────────────────────────────────────────────┘
```

### 2.5 缓存感知重试

在Fast Mode下，重试策略会考虑Prompt Cache的命中率：

```
┌──────────────────────────────────────────────────────────────────┐
│                缓存感知重试 (Cache-Aware Retry)                   │
│                                                                  │
│  Fast Mode 条件:                                                 │
│  - 距离上次请求 < 20s                                            │
│  - 保持相同模型（不降级）                                         │
│                                                                  │
│  原因:                                                           │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  Prompt Cache 机制:                                       │   │
│  │  - API缓存了前缀token（system prompt + 历史消息）          │   │
│  │  - 相同模型 + 相似前缀 → Cache Hit → 更快响应 + 更低成本   │   │
│  │  - 切换模型 → Cache Miss → 重新处理全部token               │   │
│  │                                                          │   │
│  │  重试决策:                                                 │   │
│  │  如果 < 20s内重试 → 保持相同模型（即使更贵）               │   │
│  │  → Cache Hit 的加速效果可能比模型降级更快                   │   │
│  │  → 同时节省了重新处理token的成本                            │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  时间线:                                                         │
│  t=0:  Opus请求 → 529错误                                       │
│  t=5s: 重试（< 20s）→ 保持Opus → Cache Hit → 快速成功           │
│  t=25s: 如果仍失败 → 考虑降级到Sonnet（Cache已过期）             │
└──────────────────────────────────────────────────────────────────┘
```

**关键洞察**：缓存感知重试体现了"延迟-成本-速度"三维优化。在缓存有效窗口内，保持昂贵模型可能比切换到便宜模型更快更省。这是一个反直觉但正确的决策——因为Prompt Cache的加速效果可以抵消模型本身的性能差异。

## 三、FallbackTriggeredError 降级策略

### 3.1 降级优于重试

claude-code的核心降级哲学是：**降级优于重试（Degradation over Retry）**。当连续遇到529错误时，系统不是盲目重试同一个昂贵的模型，而是主动降级到更便宜、更快的模型。

```
┌──────────────────────────────────────────────────────────────────┐
│              FallbackTriggeredError 降级流程                       │
│                                                                  │
```

### 3.2 FallbackTriggeredError 结构

```typescript
// FallbackTriggeredError 携带完整的降级上下文
class FallbackTriggeredError extends Error {
  readonly originalError: Error        // 原始529错误
  readonly consecutiveFailures: number // 连续失败次数
  readonly fromModel: string           // 原始模型 (如 'opus')
  readonly toModel: string             // 降级目标模型 (如 'sonnet')
  readonly reason: string              // 降级原因 ('529_consecutive')
}
```

**设计意图**：

| 字段 | 用途 |
|------|------|
| `originalError` | 保留完整的错误链，用于诊断根因 |
| `consecutiveFailures` | 记录触发降级的阈值（通常为3），用于调优 |
| `fromModel` / `toModel` | 明确记录降级路径，用于审计和分析 |
| `reason` | 降级原因分类，支持未来扩展更多降级触发条件 |

### 3.3 降级策略的经济学分析

```
┌──────────────────────────────────────────────────────────────────┐
│                降级 vs 重试 经济学分析                             │
│                                                                  │
│  场景: 服务过载 (529)                                            │
│                                                                  │
│  [策略A: 盲目重试]                                                │
│  Opus → 529 → Opus → 529 → Opus → 529 → ... (用户等待30s+)     │
│  成本: 3次Opus调用 × 高价格 = $$$$                               │
│  用户体验: 等待30s+，可能最终仍然失败                              │
│  服务端影响: 3次重试加剧过载                                      │
│                                                                  │
│  [策略B: 降级优于重试]                                            │
│  Opus → 529 → Opus → 529 → Opus → 529 → Sonnet → 成功          │
│  成本: 3次Opus + 1次Sonnet = $$$ + $ = $$$                      │
│  用户体验: 等待12s（3次Opus退避）+ Sonnet响应时间                  │
│  服务端影响: Sonnet请求更轻量，更容易成功                          │
│                                                                  │
│  结论: 降级策略在成本、延迟、成功率三个维度上都优于盲目重试          │
└──────────────────────────────────────────────────────────────────┘
```

**关键洞察**：降级策略的本质是"用更少的资源做更多的事"。当高端模型过载时，切换到轻量模型不仅降低了单次请求的成本，还因为轻量模型消耗更少的服务端资源，反而更容易在过载期间成功响应。这是一种正反馈循环——降级减轻了服务端压力，服务端压力降低后高端模型也更容易恢复。

### 4.2 ShellError：Shell命令失败的完整上下文

```typescript
class ShellError extends Error {
  readonly stdout: string       // 标准输出（可能包含有用的错误信息）
  readonly stderr: string       // 标准错误
  readonly code: number | null  // 退出码
  readonly interrupted: boolean // 是否被信号中断

```

**设计意图**：Shell命令失败时，退出码、stdout、stderr都是诊断问题的关键信息。传统做法只传递错误消息，丢失了结构化上下文。ShellError将所有相关信息打包在一起，使上层可以做出精确的决策。

**使用场景示例**：

### 4.3 TelemetrySafeError：代码审查机制

```typescript
class TelemetrySafeError extends Error {
```

### 4.4 shortErrorStack()：上下文Token节省

```typescript
function shortErrorStack(error: Error): string {
  const stack = error.stack ?? ''
  const frames = stack.split('\n')
  // 保留前5帧，截断其余
  return frames.slice(0, 5).join('\n')
}
```

**设计意图**：错误堆栈可能非常深（50+帧），当错误信息需要流入模型上下文时，完整的堆栈会消耗大量宝贵的context token。`shortErrorStack()`将堆栈截断到5帧，保留了足够的诊断信息（函数名、文件名、行号），同时大幅节省了token。

```
┌──────────────────────────────────────────────────────────────┐
│  shortErrorStack() Token 节省分析                              │
│                                                              │
│  完整堆栈: ~2000 tokens (50帧 × 40 tokens/帧)                │
│  截断堆栈: ~200 tokens (5帧 × 40 tokens/帧)                  │
│  节省: ~90%                                                  │
│                                                              │
│  保留的信息 (前5帧):                                          │
│  1. 错误类型和消息                                            │
│  2. 直接抛出位置 (函数名 + 文件 + 行号)                       │
│  3. 调用者位置                                                │
│  4. 上层调用者                                                │
│  5. 更上层调用者                                              │
│                                                              │
│  丢失的信息 (后45帧):                                         │
│  - 框架内部调用链 (对诊断无价值)                               │
│  - Node.js 内部栈 (对诊断无价值)                              │
│  - 异步调度链 (对诊断无价值)                                  │
└──────────────────────────────────────────────────────────────┘
```

### 4.5 classifyAxiosError()：消除重复分类逻辑

在引入`classifyAxiosError()`之前，多个服务模块各自实现了~20行的错误分类链（判断ECONNABORTED、检查response.status等）。这不仅造成了代码重复，还导致分类逻辑不一致。

```typescript
// 统一分类函数
function classifyAxiosError(error: unknown): ErrorCategory {
  if (isAxiosError(error)) {
    if (error.code === 'ECONNABORTED') return 'timeout'
    if (!error.response) return 'network'
    const status = error.response.status
    if (status === 401 || status === 403) return 'auth'
    if (status === 529) return 'overloaded'
    if (status >= 500) return 'server_error'
    if (status >= 400) return 'client_error'
  }
  return 'other'
}
```

**设计模式**：分类集中化（Centralized Classification）——将分散在多处的分类逻辑收敛到单一函数，确保分类一致性，同时使分类策略的修改变更只需改一处。

### 4.6 isAbortError()：三种中止错误统一处理

```typescript
function isAbortError(error: unknown): boolean {
  // 类型1: 我们自己的 AbortError
  if (error instanceof AbortError) return true

  // 类型2: DOMException (浏览器环境)
  if (error instanceof DOMException && error.name === 'AbortError') return true

  // 类型3: SDK的 APIUserAbortError (API层封装)
  if (error instanceof APIUserAbortError) return true

  return false
}
```

**设计意图**：中止操作可能来自不同层级（应用层、浏览器层、SDK层），每种层使用不同的错误类型。`isAbortError()`统一了检测逻辑，确保无论中止信号来自哪个层级，都能被正确识别。

### 4.7 isFsInaccessible()：文件系统不可访问检测

```typescript
function isFsInaccessible(error: unknown): boolean {
  if (!(error instanceof Error)) return false
  const code = (error as NodeJS.ErrnoException).code
  return (
    code === 'ENOENT' ||    // 文件/目录不存在
    code === 'EACCES' ||    // 权限不足
    code === 'EPERM' ||     // 操作不允许
    code === 'ENOTDIR' ||   // 路径组件不是目录
    code === 'ELOOP'        // 符号链接循环
  )
}
```

**覆盖的错误场景**：

| 错误码 | 含义 | 典型场景 |
|--------|------|---------|
| `ENOENT` | 文件不存在 | 文件被删除、路径拼写错误 |
| `EACCES` | 权限不足 | 只读文件系统、权限配置错误 |
| `EPERM` | 操作不允许 | 系统级限制、沙箱策略 |
| `ENOTDIR` | 非目录 | 路径中某组件是文件而非目录 |
| `ELOOP` | 符号链接循环 | 循环符号链接导致路径解析无限递归 |

## 五、队列汇聚模式 (Queue-Sink Pattern)

### 5.1 核心概念

队列汇聚模式是后台任务处理的错误处理策略。与前台任务的"尽力重试"不同，后台任务采用"失败即放弃"的策略。

```
┌──────────────────────────────────────────────────────────────────┐
│              队列汇聚模式 (Queue-Sink Pattern)                    │
│                                                                  │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │  前台任务 (Foreground Task)                                 │  │
│  │                                                            │  │
│  │  用户请求 → API调用 → 529 → 重试 → 重试 → 降级 → 成功      │  │
│  │                                                            │  │
│  │  策略: 尽力交付 (Best Effort Delivery)                      │  │
│  │  原因: 用户在等待，延迟是可感知的                            │  │
│  │  错误处理: 完整重试链 + 模型降级                             │  │
│  └────────────────────────────────────────────────────────────┘  │
│                                                                  │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │  后台任务 (Background Task)                                 │  │
│  │                                                            │  │
│  │  触发器 → API调用 → 529 → [放弃] → 记录日志                 │  │
│  │                                                            │  │
│  │  策略: 失败即放弃 (Fail Fast, Log & Abandon)                │  │
│  │  原因: 避免重试风暴 (3-10x 网关放大)                        │  │
│  │  错误处理: 记录日志 + 任务标记为失败                         │  │
│  └────────────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────────────┘
```

### 5.2 网关放大效应分析

```
┌──────────────────────────────────────────────────────────────────┐
│                网关放大效应 (Gateway Amplification)                │
│                                                                  │
│  场景: 100个后台任务同时遇到529                                   │
│                                                                  │
│  [无队列汇聚]                                                    │
│  100个任务 × 3次重试 = 300个额外请求                              │
│  经过网关（3-10x放大）= 900 - 3000个请求                         │
│  → 服务端承受原始负载的10-30倍                                    │
│  → 恢复时间从分钟级延长到小时级                                   │
│                                                                  │
│  [有队列汇聚]                                                    │
│  100个任务 × 0次重试 = 0个额外请求                               │
│  → 服务端只承受原始负载                                          │
│  → 恢复时间保持分钟级                                            │
│                                                                  │
│  放大因子计算:                                                    │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │  原始请求: N                                             │   │
│  │  重试次数: R                                             │   │
│  │  网关放大: G (3-10x)                                     │   │
│  │  总负载: N × (1 + R) × G                                 │   │
│  │                                                          │   │
│  │  N=100, R=3, G=5:                                        │   │
│  │  无汇聚: 100 × 4 × 5 = 2000 请求                         │   │
│  │  有汇聚: 100 × 1 × 5 = 500   请求                        │   │
│  │  负载降低: 75%                                            │   │
│  └──────────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────────┘
```

**关键洞察**：队列汇聚模式的核心洞察是"后台任务的重试成本远大于前台任务"。前台任务的重试是1:1的（一个失败请求产生一个重试请求），但后台任务的重试经过网关放大后是1:3~10的。因此，后台任务的"放弃"不是损失，而是对系统整体健康度的贡献。

### 5.3 前台/后台判定机制

```typescript
// 通过来源白名单判定是否为前台任务
const FOREGROUND_529_RETRY_SOURCES: ReadonlySet<string> = new Set([
  'user-query',       // 用户直接查询
  'tool-execution',   // 工具执行（用户可见）
  'plan-generation',  // Plan生成（用户等待）
  // 注意: 不包含 'background-sync', 'telemetry', 'prefetch' 等
])

function shouldRetry529(source: string): boolean {
  return FOREGROUND_529_RETRY_SOURCES.has(source)
}
```

**设计模式**：白名单门控（Whitelist Gating）——默认不重试（安全），只有明确列入白名单的来源才允许重试。这符合Fail-Closed原则：不确定时选更安全的默认值。

```
│  │  P1 - Important (重要后台)                                │   │
│  │  影响用户体验 → 有限重试 (最多2次) + 降级                  │   │
│  │                                                          │   │
│  │  P2 - Background (普通后台)                               │   │
│  │  不影响用户体验 → 失败即放弃 + 日志记录                    │   │
│  │                                                          │   │
│  │  P3 - Telemetry (遥测)                                    │   │
│  │  统计数据 → 永不重试 + 静默丢弃                            │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  实现要点:                                                        │
│  1. 每个任务携带优先级标记                                       │
│  2. 重试引擎根据优先级选择策略                                    │
│  3. 后台任务的重试预算远小于前台任务                               │
│  4. 监控系统跟踪各优先级的失败率和重试成本                         │
└──────────────────────────────────────────────────────────────────┘
```

### 6.4 降级优于重试

**claude-code的做法**：连续3次529后触发`FallbackTriggeredError`，从Opus降级到Sonnet，而非继续重试Opus。

**对EvoAgent的启发**：

```
┌──────────────────────────────────────────────────────────────────┐
│              EvoAgent 多级降级链设计                                │
│                                                                  │
│  降级链 (Degradation Chain):                                      │
│                                                                  │
│  Level 0: 最优模型 (e.g., GPT-4o / Claude Opus)                  │
│      │  连续失败 ≥ 3                                              │
│      v                                                           │
│  Level 1: 标准模型 (e.g., GPT-4o-mini / Claude Sonnet)           │
│      │  连续失败 ≥ 3                                              │
│      v                                                           │
│  Level 2: 轻量模型 (e.g., GPT-3.5 / Claude Haiku)               │
│      │  连续失败 ≥ 3                                              │
│      v                                                           │
│  Level 3: 本地回退 (e.g., 本地小模型 / 规则引擎)                  │
│      │  连续失败 ≥ 3                                              │
│      v                                                           │
│  Level 4: 优雅降级 (返回缓存结果 / 提示用户稍后重试)               │
│                                                                  │
│  关键设计:                                                        │
│  1. 每级降级携带完整上下文 (FallbackTriggeredError模式)            │

│  3. 恢复探测: 定期尝试高级模型，成功后升级                          │
│  4. 降级事件上报: 用于分析降级频率和根因                            │
└──────────────────────────────────────────────────────────────────┘
```

### 6.5 分类驱动的错误响应

**claude-code的做法**：`classifyAxiosError()`将错误分为5类，每类对应不同的响应策略。消除了~20行的重复分类逻辑。

**对EvoAgent的启发**：

```typescript
// 推荐的EvoAgent错误分类框架
type ErrorCategory =
  | 'transient'     // 瞬态错误 → 重试
  | 'degradation'   // 降级错误 → 切换模型/服务
  | 'auth'          // 认证错误 → 刷新凭证
  | 'validation'    // 验证错误 → 修正输入
  | 'permanent'     // 永久错误 → 放弃 + 报告
  | 'unknown'       // 未知错误 → 兜底策略

// 统一分类接口
interface ErrorClassifier {
  classify(error: unknown): ErrorCategory
}

// 策略注册表
const strategies: Record<ErrorCategory, RecoveryStrategy> = {
  transient: exponentialBackoffRetry,
  degradation: modelFallback,
  auth: credentialRefresh,
  validation: inputCorrection,
  permanent: failAndReport,
  unknown: logAndRetry,  // Fail-Closed: 未知错误默认重试
}
```

### 6.6 熔断器与因果链

**claude-code的做法**：Circuit Breaker在连续失败时熔断，配合Cause Chain Tracking追踪错误传播路径。

**对EvoAgent的启发**：

```
│              EvoAgent 熔断器 + 因果链设计                           │

│  ┌──────────────────────────────────────────────────────────┐   │
│  │  dependency: 'anthropic-api'                             │   │
```
