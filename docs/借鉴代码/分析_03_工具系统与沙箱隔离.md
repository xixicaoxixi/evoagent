## 一、核心发现摘要

| 维度 | claude-code-sourcemap | openclaw |

## 二、claude-code-sourcemap 的工具系统

### 2.1 工具抽象接口 — Tool.ts

#### 2.1.1 核心接口设计

```
Tool<Input extends AnyObject, Output, P extends ToolProgressData>
├── 身份与发现
│   ├── name: string                    // 工具唯一名称
│   ├── aliases?: string[]              // 兼容性别名
│   ├── searchHint?: string             // 关键词匹配提示（3-10 词）
│   ├── shouldDefer?: boolean           // 延迟加载（需 ToolSearch）
│   ├── alwaysLoad?: boolean            // 始终加载（绕过 defer）
│   └── isMcp?: boolean                 // MCP 工具标记
│
├── Schema 定义
│   ├── inputSchema: Input              // Zod Schema（输入验证）
│   ├── inputJSONSchema?: ToolInputJSONSchema  // MCP JSON Schema
│   ├── outputSchema?: ZodType          // 输出 Schema
│   └── strict?: boolean                // 严格模式（A/B 实验控制）
│
├── 核心执行
│   ├── call(args, context, canUseTool, parentMessage, onProgress)
│   │   └── Promise<ToolResult<Output>>
│   ├── validateInput?(args, context)
│   │   └── Promise<ValidationResult>
│   └── checkPermissions(args, context)
│       └── Promise<PermissionResult>
│
├── 安全分类
│   ├── isConcurrencySafe(input): boolean   // 并发安全性
│   ├── isReadOnly(input): boolean          // 只读判定
│   ├── isDestructive?(input): boolean      // 不可逆操作
│   ├── interruptBehavior?(): 'cancel'|'block'  // 中断行为
│   ├── isOpenWorld?(input): boolean        // 开放世界（网络等）
│   └── requiresUserInteraction?(): boolean // 需要用户交互
│
├── 上下文修改
│   ├── getPath?(input): string            // 文件路径提取
│   ├── preparePermissionMatcher?(input)    // 权限规则匹配器
│   ├── inputsEquivalent?(a, b): boolean    // 输入等价判定
│   └── backfillObservableInput?(input)     // 观察者输入回填
│
├── 结果处理
│   ├── mapToolResultToToolResultBlockParam(content, toolUseID)
│   ├── maxResultSizeChars: number         // 结果大小上限
│   └── contextModifier?: (ctx) => ctx     // 上下文修改器
│
├── 渲染层（React/Ink）
│   ├── description(input, options): Promise<string>  // 动态描述
│   ├── prompt(options): Promise<string>    // 系统提示词片段
│   ├── userFacingName(input): string      // 用户可见名称
│   ├── renderToolUseMessage(input, options)  // 工具调用渲染
│   ├── renderToolResultMessage(content, progress, options)  // 结果渲染
│   ├── renderToolUseProgressMessage(progress, options)  // 进度渲染
│   ├── renderToolUseRejectedMessage(input, options)  // 拒绝渲染
│   ├── renderToolUseErrorMessage(result, options)  // 错误渲染
│   ├── renderGroupedToolUse(toolUses, options)  // 并行分组渲染
│   ├── renderToolUseTag?(input)           // 标签渲染
│   ├── getToolUseSummary?(input)          // 摘要文本
│   ├── getActivityDescription?(input)     // 活动描述
│   ├── isSearchOrReadCommand?(input)      // 搜索/读取分类
│   ├── isResultTruncated?(output)         // 结果截断判定
│   ├── isTransparentWrapper?()            // 透明包装器
│   └── extractSearchText?(out)            // 搜索文本提取
│
└── 安全分类器
    └── toAutoClassifierInput(input): unknown  // AI 分类器输入
```

**关键设计洞察**：

1. **泛型三参数设计** `<Input, Output, P>`：`Input` 绑定 Zod Schema 实现编译期类型安全；`Output` 自由类型支持任意返回结构；`P` 绑定进度数据类型，使不同工具的进度信息类型安全。

2. **buildTool() 工厂函数**：提供 7 个安全默认值，采用"**fail-closed**"原则——不确定时选择更安全的默认值：
   - `isConcurrencySafe` → `false`（假设不安全）
   - `isReadOnly` → `false`（假设会写入）
   - `isDestructive` → `false`
   - `checkPermissions` → `allow`（委托给通用权限系统）
   - `toAutoClassifierInput` → `''`（跳过分类器，安全相关工具必须覆盖）
   - `isEnabled` → `true`
   - `userFacingName` → `name`

3. **ToolResult 副作用模型**：工具结果不仅可以携带数据（`data`），还可以携带新消息（`newMessages`）和上下文修改器（`contextModifier`）。这使工具调用可以产生副作用——例如修改全局状态、注入系统消息等。

4. **ToolUseContext 上下文对象**：包含 30+ 字段，是"**上下文对象模式**"的教科书实现——将所有环境依赖打包传递，避免全局状态耦合。关键子对象包括：
   - `options`：命令列表、调试标志、模型配置、MCP 客户端等
   - `abortController`：取消信号
   - `readFileState`：文件状态缓存（Read-before-Write 的基础）
   - `contentReplacementState`：工具结果预算（跨线程共享）
   - `localDenialTracking`：异步代理的本地拒绝追踪

#### 2.1.2 工具结果大小控制

```
maxResultSizeChars 策略：
├── Read: Infinity（永不持久化，避免 Read→file→Read 循环）
├── BashTool: 30,000（大输出写入磁盘，返回预览）
├── GrepTool: 20,000（搜索结果截断）
├── FileWrite/FileEdit/NotebookEdit: 100,000
├── WebFetch: 100,000
└── 其他工具: 默认值
```

超过 `maxResultSizeChars` 的结果会被持久化到磁盘，Claude 收到的是包含文件路径的预览而非完整内容。

### 2.4 文件工具统一安全模式

#### 2.4.1 Read-before-Write 机制

所有文件写入工具（FileWrite、FileEdit、NotebookEdit）都实施 **Read-before-Write** 策略：

```
写入流程：
1. validateInput() → 检查 readFileState.get(filePath) 是否存在
2. 如果文件未被读取过 → 拒绝（提示先 Read）
3. 如果文件已被读取 → 比较 mtime（Staleness 检查）
4. mtime 晚于读取时间 → 拒绝（文件被外部修改）
5. 通过检查 → 执行写入
```

**设计意图**：确保 Agent 在编辑文件前"看到"了文件的当前状态，防止盲目覆盖用户或其他工具的修改。

#### 2.4.2 Staleness 检查

```
原子性检查流程：
1. readFileState.get(filePath) → 获取上次读取的 mtime
2. statSync(filePath).mtime → 获取当前 mtime
3. 比较：
   ├── mtime 未变 → 允许写入
   ├── mtime 已变 → FILE_UNEXPECTEDLY_MODIFIED_ERROR
   └── Windows 回退 → 内容比较（mtime 精度不足）
```

#### 2.4.3 UNC 路径保护

所有涉及文件路径的工具统一跳过 UNC 路径（`\\` 或 `//` 开头），防止 NTLM 凭证泄露攻击。

## 四、memdir — 基于文件系统的记忆系统

### 4.1 设计哲学

memdir **不是**虚拟文件系统，而是基于真实文件系统的持久化存储方案。记忆以 Markdown 文件形式存储在磁盘上：

```
~/.claude/projects/<sanitized-project-root>/memory/
  ├── MEMORY.md          # 入口索引（自动加载到上下文）
  ├── user_role.md       # 用户角色记忆
  ├── feedback_testing.md # 反馈记忆
  ├── team/              # 团队共享记忆
  └── logs/              # KAIROS 模式日志
      └── YYYY/MM/YYYY-MM-DD.md
```

### 4.2 四类型分类法

| 类型 | 可见性 | 范围 | 说明 |
|------|--------|------|------|
| `user` | 始终 private | 用户角色、目标、偏好 | 不可从项目推导的信息 |
| `feedback` | 默认 private | 行为指导 | 避免重复的行为纠正 |
| `project` | 偏 team | 项目工作、事件 | 不可从代码/git 推导 |
| `reference` | 通常 team | 外部系统指针 | Linear、Grafana、Slack 等 |

### 4.3 记忆老化与漂移防护

```
老化机制：
├── 基于 mtime 的年龄计算（0 天=today, 1 天=yesterday, 2+ 天=N days ago）
├── 超过 1 天 → 自动附加过期警告
└── 警告内容："Memories are point-in-time observations, not live state"

漂移防护（提示词层面）：
├── MEMORY_DRIFT_CAVEAT：提醒记忆可能过时
├── TRUSTING_RECALL_SECTION："Before recommending from memory" — 要求验证
└── 核心原则："The memory says X exists" ≠ "X exists now"
```

### 4.4 语义检索

```
findRelevantMemories(query, alreadySurfaced)
    │
    ├── scanMemoryFiles() — 扫描目录中所有 .md（排除 MEMORY.md）
    ├── 读取每个文件前 30 行 frontmatter
    ├── formatMemoryManifest() — 格式化为文本清单
    ├── 使用 Sonnet 模型（sideQuery）选择最多 5 条相关记忆
    └── 过滤已展示记忆（alreadySurfaced 集合）
```

## 五、openclaw 的安全与沙箱系统

### 5.1 审计系统总体架构

```
runSecurityAudit(options)
    │
    ├── 1. collectAttackSurfaceSummaryFindings()    — 攻击面摘要
    ├── 2. collectSyncedFolderFindings()            — 同步文件夹
    ├── 3. collectGatewayConfigFindings()           — 网关配置（最核心）
    ├── 4. collectPluginSecurityAuditFindings()     — 插件安全
    ├── 5. collectLoggingFindings()                 — 日志脱敏
    ├── 6. collectElevatedFindings()                — 提权执行
    ├── 7. collectExecRuntimeFindings()             — 执行运行时（最核心）
    ├── 8. Hooks 加固 + 沙箱空操作 + 危险配置 + 节点命令
    ├── 9. collectFilesystemFindings()（可选）      — 文件系统权限
    ├── 10. collectChannelSecurityFindings()（可选） — 频道安全
    └── 11. deep: 探测网关连通性（可选）
```

**设计模式**：**收集器编排模式**。每个收集器是独立函数，主函数按序调用并汇总为 `SecurityAuditReport`。

### 5.2 三级严重度分类

```typescript
type SecurityAuditSeverity = "info" | "warn" | "critical";

type SecurityAuditFinding = {
  checkId: string;      // 点分命名空间，如 "gateway.bind_no_auth"
  severity: SecurityAuditSeverity;
  title: string;
  detail: string;
  remediation?: string; // 修复建议
};
```

**环境感知严重度升级**：同一检查项根据部署环境动态调整严重度。例如 `gateway.control_ui.allowed_origins_wildcard` 在 `bind !== "loopback"` 时从 `warn` 升级为 `critical`。

### 5.3 Docker 沙箱隔离实现

#### 5.3.1 多维度纵深防御

| 层次 | 实现方式 | 说明 |
|------|---------|------|
| **文件系统** | `--read-only` 根 + tmpfs `/tmp` `/var/tmp` `/run` | 防止任意文件写入 |
| **网络** | 默认 `network: "none"`，阻止 `host` 和 `container:*` | 完全网络隔离 |
| **能力** | `--cap-drop ALL` | 移除所有 Linux capabilities |
| **特权提升** | `--security-opt no-new-privileges` | 阻止 uid 提权 |
| **系统调用** | seccomp profile（阻止 `unconfined`） | 限制危险系统调用 |
| **MAC** | AppArmor profile（阻止 `unconfined`） | 强制访问控制 |
| **资源限制** | `--pids-limit` / `--memory` / `--cpus` / `--ulimit` | 防止资源耗尽 |
| **环境变量** | 敏感变量过滤（API keys, tokens） | 防止凭证泄露 |
| **挂载** | 阻止系统目录挂载 + 符号链接逃逸防护 | 防止文件系统逃逸 |

#### 5.3.2 挂载安全验证

**阻止的宿主路径**：
```
/etc, /proc, /sys, /dev, /root, /boot, /run, /var/run
/var/run/docker.sock, /run/docker.sock
```

**阻止的 Home 子路径**：
```
.aws, .cargo, .config, .docker, .gnupg, .netrc, .npm, .ssh
```

**符号链接逃逸防护**：通过 `resolveSandboxHostPathViaExistingAncestor()` 解析现有祖先目录的真实路径，防止不存在的叶路径绕过检查。

#### 5.3.3 环境变量过滤

**阻止模式**（正则匹配）：
- API Keys: `ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, `GEMINI_API_KEY` 等
- Bot Tokens: `TELEGRAM_BOT_TOKEN`, `DISCORD_BOT_TOKEN`, `SLACK_BOT_TOKEN`
- 网关凭证: `OPENCLAW_GATEWAY_TOKEN`, `OPENCLAW_GATEWAY_PASSWORD`
- 云凭证: `AWS_SECRET_ACCESS_KEY`, `AWS_SESSION_TOKEN`, `GH_TOKEN`
- 通用模式: `/_?(API_KEY|TOKEN|PASSWORD|PRIVATE_KEY|SECRET)$/i`

#### 5.3.4 沙箱后端注册机制

```typescript
registerSandboxBackend("docker", { factory, manager });
registerSandboxBackend("ssh", { factory, manager });
```

**设计模式**：**插件式后端注册**。每个后端实现 `SandboxBackendFactory` 和 `SandboxBackendManager` 接口，通过 `SandboxBackendHandle` 暴露统一的命令执行和文件桥接能力。

#### 5.3.5 沙箱工具策略

```
默认允许：exec, process, read, write, edit, apply_patch, image,
          sessions_*, subagents, session_status
默认拒绝：browser, canvas, nodes, cron, gateway, 以及所有频道 ID

策略解析链：Agent 级 → 全局级 → 默认值
合并规则：deny 优先于 allow
```

### 5.4 外部内容安全

**设计模式**：**边界标记 + 安全警告注入 + 防伪造**

1. **唯一随机边界标记**：每次包装生成 16 字节随机 hex ID
2. **Unicode 同形字折叠**：全角 ASCII、各种 Unicode 尖括号同形字折叠为标准 ASCII
3. **零宽字符剥离**：忽略零宽空格、零宽连接符等不可见字符
4. **标记清洗**：检测并替换伪造的边界标记为 `[[MARKER_SANITIZED]]`
5. **注入检测**：正则匹配 6+ 种提示注入模式（ignore previous instructions、system override 等）

### 5.5 技能安全扫描

**两级规则引擎**：

| 级别 | 规则 | 严重度 |
|------|------|--------|
| **行级** | `child_process` exec/spawn | critical |
| **行级** | `eval()` / `new Function()` | critical |
| **行级** | 加密货币挖矿引用 | critical |
| **行级** | 非标准端口 WebSocket | warn |
| **源码级** | 文件读取 + 网络发送（数据渗出） | warn |
| **源码级** | Hex 编码序列 / 大 base64 解码（混淆代码） | warn |
| **源码级** | 环境变量访问 + 网络发送（凭证收割） | critical |

**性能优化**：文件级缓存（size + mtime）、目录条目缓存、最大 500 文件、最大 1MB。

### 5.6 安全正则表达式（ReDoS 防护）

**自研正则安全分析器**（不依赖外部库）：

```
tokenizePattern(regex) → Token 流
    │
    ▼
analyzeTokensForNestedRepetition(tokens) → 栈式帧分析
    │
    ├── 检测嵌套重复（如 (a+)+）
    └── 拒绝编译 → reason: "unsafe-nested-repetition"
```

同时提供 `testRegexWithBoundedInput()` 限制输入窗口大小（默认 2048 字符），从输入侧限制回溯爆炸。

### 5.7 密钥安全比较

```typescript
function safeEqualSecret(provided, expected): boolean {
  const hash = (s) => createHash("sha256").update(s).digest();
  return timingSafeEqual(hash(provided), hash(expected));
}
```

**恒定时间比较 + 哈希归一化**：先 SHA-256 哈希消除长度差异，再 `timingSafeEqual` 防止时序侧信道攻击。

### 5.8 声明式安全修复

```typescript
type SecurityPermissionTarget = {
  path: string;
  mode: number;       // 0o700 (目录) 或 0o600 (文件)
  require: "dir" | "file";
};
```

修复目标包括：state 目录 → 0o700、配置文件 → 0o600、OAuth 凭证目录 → 0o700、会话记录 → 0o600。

**配置修复**：自动将 `groupPolicy: "open"` 改为 `"allowlist"`，将 `logging.redactSensitive: "off"` 改为 `"tools"`。

### 5.9 HTTP 工具拒绝列表

```typescript
DEFAULT_GATEWAY_HTTP_TOOL_DENY = [
  "exec", "spawn", "shell",           // RCE 面
  "fs_write", "fs_delete", "fs_move",  // 文件变异
  "apply_patch",                       // 文件重写
  "sessions_spawn", "sessions_send",   // 会话编排/注入
  "cron", "gateway", "nodes",          // 控制面
  "whatsapp_login",                    // 交互式流程
];
```

**设计模式**：**默认拒绝 + 显式覆盖**。通过 HTTP `/tools/invoke` 端点默认拦截高风险工具。

## 七、通用设计模式提炼

### 7.1 可直接复用的模式

#### 模式 1：buildTool() 工厂函数 + 安全默认值

```
适用场景：需要定义大量工具接口，且默认行为应偏向安全
实现方式：工厂函数填充 7 个默认值，fail-closed 原则
优势：所有工具共享统一基线，新增工具只需定义差异部分
```

#### 模式 2：Read-before-Write + Staleness 检查

```
适用场景：Agent 编辑文件时需要确保不盲目覆盖
实现方式：readFileState 缓存 + mtime 比较 + Windows 内容回退
优势：防止并发修改冲突，无需锁机制
```

#### 模式 3：AST 级命令安全分析

```
适用场景：需要精确控制 shell 命令的执行权限
实现方式：tree-sitter 解析 → 语义检查 → 多层规则匹配
优势：比正则匹配更准确，能理解命令结构而非纯文本
```

#### 模式 4：Prompt Cache 优化的工具排序

```
适用场景：工具列表频繁变化但需要最大化缓存命中率
实现方式：内置工具按名称排序作为连续前缀 + MCP 工具追加
优势：新增 MCP 工具不影响内置工具的 cache key
```

#### 模式 5：Docker 多维度纵深防御

```
适用场景：需要隔离不可信代码执行
实现方式：read-only + cap-drop + seccomp + AppArmor + network:none + 资源限制
优势：任一维度被突破仍有其他维度防护
```

#### 模式 6：收集器编排审计

```
适用场景：需要全面扫描系统安全状态
实现方式：独立收集器函数 + 主函数按序编排 + 汇总报告
优势：新增检查只需添加收集器，不影响现有逻辑
```

#### 模式 7：边界标记 + 防伪造（外部内容）

```
适用场景：需要安全地包装外部内容注入 LLM 上下文
实现方式：随机 ID 边界 + Unicode 同形字折叠 + 零宽字符剥离 + 注入检测
优势：多层防护防止提示注入攻击
```

#### 模式 8：插件式沙箱后端

```
适用场景：需要支持多种沙箱隔离技术
实现方式：registerSandboxBackend() 注册表 + 统一 Handle 接口
优势：运行时可切换后端，新增后端只需实现接口
```

### 7.2 需要适配的设计

| 设计 | 来源 | 适配建议 |
|------|------|---------|
| macOS Seatbelt 沙箱 | claude-code | 仅限 macOS，Linux 需替换为 Docker/seccomp |
| React/Ink 渲染集成 | claude-code | 与 UI 框架耦合，通用场景需抽象渲染层 |
| tree-sitter AST 解析 | claude-code | 依赖 tree-sitter 二进制，需预编译对应语言 |
| MCP 工具前缀匹配 | claude-code | 与 MCP 协议耦合 |
| 插件配置契约 | openclaw | 需要建立契约注册机制 |
