### 2.2 核心抽象体系

#### 2.2.1 QueryEngine — 会话级查询引擎

**设计模式**：依赖注入 + 异步生成器 + 会话状态封装

```
QueryEngineConfig (依赖注入)
├── cwd: string                    // 工作目录
├── tools: Tools                   // 工具集
├── commands: Command[]            // 命令集
├── mcpClients: MCPServerConnection[] // MCP 客户端
├── agents: AgentDefinition[]      // 子 Agent 定义
├── canUseTool: CanUseToolFn       // 权限检查函数
├── model: string                  // 模型配置
├── budget: TokenBudget            // Token 预算
└── ... (20+ 配置项)

QueryEngine (核心类)
├── submitMessage(prompt) → AsyncGenerator<SDKMessage>  // 主入口
├── 内部状态：消息历史、文件缓存、用量统计、权限拒绝记录
└── 调用 query.ts 的 query() 执行实际循环
```

**关键洞察**：
- 每个对话实例化一个 `QueryEngine`，会话状态完全封装在类内部
- 通过 `AsyncGenerator` 实现流式输出，调用方逐步消费模型响应
- 所有外部依赖通过 `QueryEngineConfig` 注入，引擎与 UI 完全解耦
- SDK 模式（`agentSdkTypes.ts`）的 `query()` 函数和 V2 API 的 `createSession()` 都基于此引擎

#### 2.2.2 query.ts — Agentic Loop（核心查询循环）

**设计模式**：异步生成器协程 + 隐式状态机

```
query(params) → AsyncGenerator<StreamEvent | Message>
  └── queryLoop(params) — while(true) 无限循环
        ├── 1. 消息准备（系统提示 + 用户上下文 + 历史消息）
        ├── 2. API 调用（Claude Messages API，流式响应）
        ├── 3. 工具执行（runTools → StreamingToolExecutor）
        ├── 4. 压缩检查（autoCompact / reactiveCompact）
        ├── 5. 停止判断（stopHooks: maxTurns / taskBudget / endTurn）
        └── 6. yield 事件 → 继续下一轮迭代
```

**关键洞察**：
- 这是整个 Claude Code 的"心脏"——驱动"模型推理 → 工具执行 → 结果反馈 → 继续推理"的迭代
- 使用 `AsyncGenerator` + `yield*` 组合嵌套生成器，实现复杂控制流的线性表达
- `State` 对象在迭代间传递，通过解构/重建实现不可变状态转换
- 查询子模块分离：`config.ts`（配置构建）、`deps.ts`（依赖注入）、`stopHooks.ts`（停止判断）、`tokenBudget.ts`（Token 预算）

#### 2.2.3 Tool.ts — 工具抽象接口

**设计模式**：策略模式 + 上下文对象模式

```
Tool<Input, Output, P> (核心接口，约 30 个属性/方法)
├── name: string                    // 工具名称
├── description(): string           // 动态描述生成
├── inputSchema: JSONSchema         // 输入 Schema
├── call(input, context): Promise<ToolResult>  // 核心执行
├── checkPermissions(input, ctx): PermissionResult  // 权限检查
├── validateInput(input): ValidationResult  // 输入验证
├── renderToolResultMessage(): ReactNode  // 结果渲染
├── isConcurrencySafe: boolean      // 并发安全性
└── isEnabled(): boolean            // 是否启用

ToolUseContext (执行上下文，30+ 字段)
├── options, abortController        // 基础控制
├── commands, mcpClients, agents    // 能力注册表
├── getAppState(), updateAppState() // 状态访问
├── permissionContext               // 权限上下文
├── queryTracking                   // 查询链追踪
└── ...
```

**关键洞察**：
- `Tool` 接口极为丰富（~30 个属性/方法），覆盖工具完整生命周期
- `ToolUseContext` 是"上下文对象模式"的典型实现，将 30+ 环境依赖打包传递
- 每个工具实现自己的权限检查策略（`checkPermissions`），支持分级错误处理
- `ToolResult` 支持携带新消息、上下文修改器、MCP 元数据等副作用

#### 2.2.4 tools.ts — 工具注册中心

**设计模式**：注册表模式 + 管道过滤 + 特性门控

```
getAllBaseTools() → Tools (30+ 工具)
  │
  ├── 条件编译：feature() + 环境变量控制工具可用性
  │
  ▼
getTools(permissionContext) → Tools
  │
  ├── 1. simple 模式排除
  ├── 2. REPL 模式过滤
  ├── 3. deny 规则过滤（支持 MCP 服务器前缀匹配）
  ├── 4. isEnabled() 检查
  │
  ▼
assembleToolPool(permissionContext, mcpTools) → Tools
  │
  ├── 内置工具 + MCP 工具合并
  ├── 按名称排序（保证 prompt cache 稳定性）
  └── 内置工具优先（同名覆盖）
```

**关键洞察**：
- 工具获取经过多层过滤管道，每层逐步精简
- 按名称排序是关键设计——保证工具列表在 prompt 中的位置稳定，最大化 prompt cache 命中率
- 大量使用 `feature()` 函数实现条件编译，不同构建产物包含不同工具集

### 2.3 类型体系核心概念

```typescript

type SessionId = string & { readonly __brand: 'SessionId' }

// AgentId 格式：a + 可选标签 + 16位十六进制 (如 a-coder-3f2a1b0c4d5e6f7a)
```

```typescript
type Command = CommandBase & (
  | PromptCommand      // 注入 LLM 提示词（inline/fork 上下文）

  | LocalJSXCommand    // 本地 JSX 命令（返回 React 组件）
)
```

### 4.1 架构风格对比

| 维度 | claude-code-sourcemap | openclaw |

| **上下文管理** | `context.ts` + `services/compact/`（LLM 驱动压缩） | `context-engine/`（可插拔引擎框架）+ `agents/compaction.ts` |

| 设计模式 | claude-code-sourcemap | openclaw |

### 4.3 各自的独特优势

#### claude-code-sourcemap 的独特优势

1. **极致的 Agentic Loop 设计**：`query.ts` 的异步生成器 + 隐式状态机是教科书级的 Agent 循环实现
2. **精细的权限系统**：7 种模式 + 11 种决策原因 + AI 分类器自动决策，覆盖所有安全场景
3. **丰富的钩子系统**：4 种钩子类型 × 16+ 种事件，提供极强的可扩展性
4. **Prompt Cache 优化**：工具按名称排序保证 prompt cache 稳定性，是生产级性能优化
5. **品牌类型安全**：`SessionId` / `AgentId` 在编译期防止 ID 混淆
6. **事件溯源日志**：`Entry` 联合类型记录所有系统事件，支持完整的会话回放

#### openclaw 的独特优势

1. **插件化架构**：90+ 扩展 + 严格边界控制（AGENTS.md），真正的平台化设计
2. **多提供商支持**：40+ AI 提供商 + 故障转移 + 运行时切换，不锁定单一厂商
3. **记忆系统**：四引擎架构（Foundation + Storage + Embeddings + QMD）+ "梦境"整理
4. **可插拔上下文引擎**：`context-engine/` 注册表模式，支持运行时切换引擎实现
5. **安全审计系统**：70+ 文件覆盖全维度安全扫描，含自动修复能力
6. **Gateway 架构**：WebSocket + HTTP 双协议 + 50+ RPC 方法 + 配置热重载
7. **Cron 定时任务引擎**：864KB 的完整调度系统，支持主动巡检

## 七、为后续轮次准备的关键发现

### 7.1 必须在第 2 轮深入的问题

1. **claude-code 的 coordinator/ 如何实现多 Agent 协调？** 文件极少（19KB）但被标记为 ⭐⭐⭐⭐⭐，可能包含高密度的编排逻辑
2. **openclaw 的 harness/ AgentHarness 如何选择和优先级排序？** 这是 Agent 编排的核心抽象
3. **Git Worktree 如何实现安全并行？** 这是 A/B 测试和并行执行的基础设施

### 7.2 必须在第 3 轮深入的问题

1. **Tool.ts 的 30+ 属性/方法中，哪些是核心生命周期方法？** 需要区分"必须实现"和"可选覆盖"
2. **权限系统的 AI 分类器（YOLO Classifier）如何工作？** 两阶段推理（fast + thinking）的具体实现
3. **LSP 集成如何降低编码 Agent 的错误率？** 查询类型映射和结果解析

### 7.3 架构差异的关键启示

1. **单体 vs 微内核**：claude-code 的单体架构使其内部模块耦合更深，但协作更紧密；openclaw 的微内核架构使其更灵活，但模块间通信更复杂
2. **单模型 vs 多模型**：claude-code 深度集成 Claude，可以针对模型特性做极致优化；openclaw 需要抽象出模型无关的接口，灵活性更高
3. **编码 Agent vs 通用助手**：claude-code 专注于编码场景，工具集（Bash/FileEdit/Glob/Grep/LSP）都是编码工具；openclaw 覆盖全场景，工具集更广泛
4. **权限深度差异**：claude-code 的权限系统远比 openclaw 精细（7 模式 vs 审计扫描），这是因为编码 Agent 的操作风险更高
