## 一、核心发现摘要

本轮深入分析了两个项目的插件/技能/扩展系统，发现了**两种截然不同的"可扩展性"哲学**：

| 维度 | claude-code-sourcemap | openclaw |
|------|----------------------|----------|
| **扩展模型** | 技能即提示词（Markdown + Frontmatter） | 插件即模块（TypeScript + definePluginEntry） |
| **技能定义** | SKILL.md 文档（声明式） | SKILL.md 文档（声明式）+ TypeScript 插件（编程式） |
| **插件架构** | 内置插件 + Bundled 技能（编译时固定） | 外部插件 SDK + 90+ 独立扩展包（运行时动态） |
| **加载机制** | 五源目录扫描 + 动态发现 + 条件激活 | 插件清单注册 + definePluginEntry 回调 |
| **模型接入** | 无（单一 Anthropic Claude） | Provider 插件（Anthropic/OpenAI/Copilot Proxy 等 40+） |
| **钩子系统** | HooksSettings（配置驱动，16+ 事件） | 内部钩子引擎（代码驱动，5 类事件） |
| **安全模型** | MCP 技能不执行 shell；安全文件写入 | 五级来源优先级 + 路径边界检查 + 完整性验证 |
| **热重载** | 无（重启生效） | 插件级 restartPrefixes + 钩子级缓存破坏 |
| **能力注册点** | 1 个（Command 注册） | 14 个（Provider/Tool/CLI/Gateway/Service/Command 等） |

### 2.3 技能加载引擎 — loadSkillsDir.ts

#### 2.3.1 五源扫描策略

```
getSkillDirCommands() [memoized]
    │
    ├── 1. managed — 策略级技能（~/.claude/commands/）
    │      优先级最高，组织统一管控
    │
    ├── 2. user — 用户级技能（~/.claude/skills/）

    ├── 3. project — 项目级技能（<project>/.claude/skills/）

    │
```

#### 2.3.3 条件技能激活

```
文件操作发生（Read/Write/Edit/Grep）
    │
    ▼
discoverSkillDirsForPaths(filePaths, cwd)
    │
    ├── 从每个文件路径向上遍历查找 .claude/skills/ 目录
    ├── 排除 cwd 级别（已在启动时加载）
    ├── 检查 .gitignore
    │
    ▼
addSkillDirectories(dirs)
    │
    ├── 加载新发现的技能目录
    ├── 深层路径覆盖浅层路径
    │
    ▼
activateConditionalSkillsForPaths(filePaths, cwd)
    │
    ├── 遍历 conditionalSkills Map
    ├── 使用 ignore 库做 gitignore 风格匹配
    ├── 匹配成功 → 移入 dynamicSkills
    │
    ▼
onDynamicSkillsLoaded() → 通知观察者清除缓存
```

**关键设计洞察**：
- 条件技能不在启动时全部加载，而是等到匹配文件被操作时才激活——这是**延迟激活模式**的典型实现
- 使用 `ignore` 库（gitignore 风格匹配）而非简单的前缀/后缀匹配，支持复杂的路径模式
- 动态发现只向上遍历一级（不递归），避免性能问题

### 2.4 文件提取安全机制

Bundled 技能的 `files` 字段支持将参考文件惰性提取到磁盘：

```
registerBundledSkill({ files: SKILL_FILES, ... })
    │
    ▼
首次调用 getPromptForCommand()
    │
    ├── extractionPromise ??= extractBundledSkillFiles()
    │   ├── 创建 nonce 目录（per-process 唯一）
    │   ├── O_NOFOLLOW | O_EXCL 防止符号链接攻击
    │   ├── 0o700/0o600 权限限制
    │   └── 路径规范化防止目录遍历
    │
    ▼
prependBaseDir(blocks, extractedDir)
    └── 在提示前缀加入 "Base directory for this skill: <dir>"
```

**安全防护层次**：

| 层次 | 机制 | 防护目标 |
|------|------|---------|
| 1 | `O_NOFOLLOW` | 防止符号链接跟随 |
| 2 | `O_EXCL` | 防止预创建攻击（TOCTOU） |
| 3 | `0o700/0o600` | 限制文件权限 |
| 4 | nonce 目录 | 防止路径预测 |
| 5 | 路径规范化 | 防止目录遍历 |
| 6 | 跨平台标志 | Windows 使用 `wx` 字符串标志 |

### 2.5 MCP 技能桥接 — mcpSkillBuilders.ts

这是一个解决循环依赖的精妙设计：

```
问题：mcpSkills.ts 需要 loadSkillsDir.ts 的函数
      但直接导入会形成循环依赖
      变量动态 import() 在 Bun bunfs 中无法解析
      字面量动态 import() 被 dep-cruiser 检测为循环

解决方案：引入叶子模块（Leaf Module）
      mcpSkillBuilders.ts 不导入任何运行时模块（仅类型）
      loadSkillsDir.ts 在模块初始化时注册函数引用
      mcpSkills.ts 通过 getMCPSkillBuilders() 获取
```

```typescript
// 一次写入注册表
let builders: MCPSkillBuilders | undefined

export function registerMCPSkillBuilders(b: MCPSkillBuilders): void {
  builders = b  // 不检查重复，依赖调用顺序保证
}

export function getMCPSkillBuilders(): MCPSkillBuilders {
  if (!builders) throw new Error('MCPSkillBuilders not registered')
  return builders
}
```

**关键洞察**：
- 这是**服务定位器模式**的极简实现——全局注册表解耦模块间的直接依赖
- 叶子模块不导入任何运行时依赖，确保自身处于依赖图的最末端
- 注册保证在 MCP 服务器连接之前完成（因为 loadSkillsDir.ts 通过 commands.ts 的静态导入在启动时被立即求值）

### 2.6 典型 Bundled 技能分析

#### 2.6.1 /batch — 并行工作编排

```
Phase 1: Plan Mode
  └── 模型研究代码库 → 分解为 5-30 个独立单元
      每个单元必须在隔离的 git worktree 中可独立实现

Phase 2: Parallel Execution
  └── AgentTool × N（isolation: "worktree", run_in_background: true）
      每个 worker 收到统一指令模板：
      1. Simplify → 2. Unit Test → 3. E2E Test → 4. Commit/PR → 5. Report

Phase 3: Progress Tracking
  └── 渲染状态表格 → 收集 PR 链接
```

**设计模式**：Map-Reduce——Coordinator 分解任务，Worker 并行执行，Coordinator 汇总结果。

#### 2.6.2 /simplify — 并行代码审查

```
Agent 1: 代码复用（重复功能、可复用工具）
Agent 2: 代码质量（冗余状态、参数膨胀、复制粘贴）
Agent 3: 效率（不必要工作、错过的并发、内存泄漏）
    │
    ▼
汇总发现 → 直接修复
```

**设计模式**：并行审查——三个独立维度同时进行，关注点分离。

#### 2.6.3 /skillify — 知识捕获

```
提取会话记忆 + 用户消息
    │
    ▼
四步流程：
  1. 分析会话（识别可重复过程）
  2. 采访用户（多轮 AskUserQuestion）
  3. 编写 SKILL.md（frontmatter + 结构化步骤）
  4. 确认并保存
```

**设计模式**：交互式知识捕获——通过多轮结构化对话将隐含知识显式化。

#### 2.6.4 /update-config — Schema 驱动文档

```typescript
// 从 Zod schema 动态生成 JSON Schema — 保持文档与类型同步
function generateSettingsSchema(): string {
  const jsonSchema = toJSONSchema(SettingsSchema(), { io: 'input' })
  return jsonStringify(jsonSchema, null, 2)
}
```

**设计模式**：Schema-driven Documentation——从类型定义自动生成参考文档，消除文档与代码的不一致。

### 2.7 内置插件系统 — builtinPlugins.ts

```
注册表：Map<string, BuiltinPluginDefinition>
    │
    ├── registerBuiltinPlugin(definition)
    │   └── ID 格式：{name}@builtin
    │
    ├── getBuiltinPlugins()
    │   ├── 检查 isAvailable()（平台兼容性）
    │   ├── 用户设置 > 插件默认值 > true
    │   └── 返回 { enabled: LoadedPlugin[], disabled: LoadedPlugin[] }
    │

```

**与 Bundled 技能的边界**：

| 维度 | 内置插件 | Bundled 技能 |
|------|---------|-------------|
| **用户控制** | 通过 /plugin UI 显式切换 | 自动启用或通过 isEnabled() 控制 |
| **适用场景** | 功能开关类（用户决定是否启用） | 复杂自动启用逻辑（如 claude-in-chrome） |
| **ID 格式** | `{name}@builtin` | 无特殊 ID |
| **注册方式** | `registerBuiltinPlugin()` | `registerBundledSkill()` |

### 3.3 插件注册 API — 14 个能力注册点

```typescript
type OpenClawPluginApi = {
  // 核心能力
  registerProvider(provider): void;           // 模型 Provider
  registerTool(factory): void;                // Agent 工具
  registerCli(registerFn, opts?): void;       // CLI 命令
  registerCommand(cmd): void;                 // 斜杠命令
  registerService(service): void;             // 后台服务

  // Provider 扩展
  registerCliBackend(backend): void;          // CLI 后端
  registerMediaUnderstandingProvider(p): void;// 媒体理解
  registerImageGenerationProvider(p): void;   // 图像生成
  registerSpeechProvider(p): void;            // 语音合成
  registerRealtimeVoiceProvider(p): void;     // 实时语音
  registerRealtimeTranscriptionProvider(p): void; // 实时转录
  registerVideoGenerationProvider(p): void;   // 视频生成

  // 网关
  registerGatewayMethod(name, handler, opts?): void; // 网关方法

  // 生命周期
  on(event: string, handler): void;          // 事件钩子

  // 运行时
  config: OpenClawConfig;
  pluginConfig: unknown;
  runtime: {
    state: { resolveStateDir(): string };
    config: { loadConfig(); writeConfigFile(cfg) };
    agent: {
      session: { resolveStorePath(); loadSessionStore() };
      runEmbeddedPiAgent(opts): Promise<{ payloads }>;
    };
  };
}
```

**关键设计洞察**：
- 14 个注册点覆盖了 AI Agent 平台的几乎所有扩展维度——从模型接入到工具注册，从 CLI 命令到后台服务
- `on(event, handler)` 提供了生命周期钩子能力，插件可以在 prompt 构建前注入上下文（如 active-memory 的 `before_prompt_build`）
- `runtime.agent.runEmbeddedPiAgent` 允许插件运行嵌入子代理——这是 active-memory 扩展的核心能力

### 3.4 插件 SDK 设计原则 — AGENTS.md

插件 SDK 的设计遵循以下原则（来自 `src/plugin-sdk/AGENTS.md`）：

| 原则 | 说明 | 体现 |
|------|------|------|
| **窄入口优先** | 优先使用窄的、专用的子路径导出 | 30+ 独立子路径，而非一个宽泛的 `import *` |
| **延迟加载友好** | 热通道入口廉价；异步路径放 `*.runtime` 子路径 | `provider-auth` vs `provider-auth-runtime` |
| **无循环依赖** | SDK 外观必须保持无环 | 叶子模块模式 + 服务定位器 |
| **静态/动态分离** | 不混合使用静态和动态导入 | 所有入口要么静态要么动态，不混用 |
| **通用优于专用** | 先确认是否真正通用 | Provider 级接口优于 Provider 特定接口 |
| **命名辅助函数** | 导出有意义的辅助函数 | `buildProviderToolCompatFamilyHooks` |
| **破坏性变更需大版本** | 移除/重命名是主版本工作 | 严格的版本管理 |

### 5.2 五级来源优先级策略

```
┌─────────────────────────────────────────────────────────────────┐
│  优先级 10: bundled（内置钩子）                                  │
│  ├── 默认启用                                                   │

├─────────────────────────────────────────────────────────────────┤
│  优先级 20: plugin（插件钩子）                                   │
│  ├── 默认启用                                                   │
│  ├── 可被 managed 覆盖，可覆盖 bundled                          │
│  └── 需通过插件激活检查 + 槽位决策                               │
├─────────────────────────────────────────────────────────────────┤
│  优先级 30: managed（管理钩子）                                  │
│  ├── 默认启用                                                   │

├─────────────────────────────────────────────────────────────────┤

├─────────────────────────────────────────────────────────────────┤

│  ├── 向后兼容                                                   │
```

**冲突解决规则**：

```typescript
// 双向声明实现精确覆盖控制
type HookSourcePolicy = {
  precedence: number,           // 优先级数值
  trustedLocalCode: boolean,    // 是否信任本地代码
  defaultEnableMode: 'enabled' | 'explicit-opt-in',
  canOverride: HookSource[],    // 可覆盖哪些来源
  canBeOverriddenBy: HookSource[], // 可被哪些来源覆盖
}
```

**关键设计洞察**：
- `canOverride` 和 `canBeOverriddenBy` 是**双向声明**——不是简单的优先级数值比较，而是显式声明"我可以覆盖谁"和"谁可以覆盖我"
- workspace 钩子默认禁用（`explicit-opt-in`）——这是**默认安全模式**，防止不受信任的工作区代码自动执行
- 插件钩子始终启用，不受配置禁用影响——插件已经过安装审核，信任级别高于工作区

### 5.3 事件引擎 — 两级分发

```typescript
// 全局单例（通过 Symbol.for + globalThis 跨 chunk 共享）
const handlers = new Map<string, InternalHookHandler[]>()

triggerInternalHook(event: InternalHookEvent): Promise<void>
    │
    ├── 1. 匹配通用事件类型（如 "command"）
    │   handlers.get("command")
    │
    ├── 2. 匹配具体事件+动作（如 "command:new"）
    │   handlers.get("command:new")
    │
    ├── 3. 合并处理器列表（去重）
    │
    └── 4. 按注册顺序依次调用
        ├── 每个处理器的错误被独立捕获
        ├── 错误不阻断其他处理器
        └── 处理器可向 event.messages 推送消息
```

**五种事件类型**：

| 类型 | Actions | 典型用途 |
|------|---------|---------|
| `command` | `new` | 命令处理拦截 |
| `session` | `create`, `patch` | 会话生命周期 |
| `agent` | `bootstrap` | Agent 引导 |
| `gateway` | `startup` | 网关启动 |
| `message` | `received`, `sent`, `transcribed`, `preprocessed` | 消息收发 |

**关键设计洞察**：
- `Symbol.for` + `globalThis` 实现跨 bundle chunk 共享——解决 Vite/Rollup bundle splitting 导致的多副本问题
- 两级分发（type + type:action）允许处理器同时监听一类事件的所有动作和特定动作
- 错误隔离确保单个处理器的异常不会影响其他处理器——这是**断路器模式**的轻量实现

### 5.4 差异化缓存策略

```typescript
buildImportUrl(handlerPath: string, source: HookSource): string
    │
    ├── openclaw-bundled（不可变源）
    │   └── file:// URL，无缓存破坏
    │       让 V8 复用模块缓存，最大化启动性能
    │
    └── 其他来源（可变源）
        └── file:// URL + ?t=<mtimeMs>&s=<size>
            基于文件元数据的缓存破坏：
            - 文件未变 → 缓存键稳定 → V8 复用缓存
            - 文件变化 → 缓存键失效 → 重新加载
```

**关键设计洞察**：
- 基于文件元数据（mtime + size）比纯时间戳更优——文件未变时缓存键保持稳定
- 不可变源（bundled）完全不破坏缓存——因为 `npm install` 之间不会变化
- `statSync` 失败时回退到 `Date.now()`——保证至少刷新一次

```
// claude-code: 条件技能
conditionalSkills.set(name, skill)  // 注册时不激活
  → 文件操作时 activateConditionalSkillsForPaths()
  → 匹配成功 → dynamicSkills.set(name, skill)

// openclaw: 插件服务
startLazyPluginServiceModule()  // 首次使用时才加载
```

**适用场景**：扩展点众多但大部分时间不活跃，需要最小化启动开销。

#### 模式 5：安全写入模式（Secure Write Pattern）

```
// claude-code: 多层防护文件写入
O_NOFOLLOW | O_EXCL  // 防止符号链接 + 防止预创建
0o700/0o600          // 限制权限
nonce 目录            // 防止路径预测
路径规范化            // 防止目录遍历

// openclaw: 双重路径验证
isPathInside(logicalPath, root)           // 逻辑路径检查
isPathInsideWithRealpath(realPath, root)  // 真实路径检查（解析符号链接）
```

**适用场景**：处理不受信任的文件输入，需要防止路径遍历和符号链接攻击。

#### 模式 6：差异化缓存策略（Differentiated Cache Strategy）

```
// openclaw: 按来源区分缓存策略
bundled → 无缓存破坏（不可变源）
managed → mtime+size 缓存破坏（可变源）
```

**适用场景**：混合了可变和不可变扩展源的系统。

### 7.2 可直接复用的设计建议
