## 一、核心发现摘要

本轮深入分析了两个项目的进化/自优化系统，发现了**两种截然不同但同样精妙的"自进化"哲学**：

| 维度 | claude-code-sourcemap | openclaw |
|------|----------------------|----------|
| **压缩策略** | 三级递进（Microcompact → AutoCompact → ReactiveCompact）+ API 原生 Context Management | 两级修剪（Context Pruning）+ 多阶段摘要（Compaction Safeguard）+ 质量守卫 |
| **记忆系统** | 三层记忆（SessionMemory 会话笔记 + memdir 文件记忆 + AutoDream 整合） | 四引擎向量记忆（Foundation + Storage + Embeddings + QMD）+ 三阶段梦境（Light/Deep/REM） |
| **记忆提取** | Forked Agent 后台提取（四类型分类 + 游标增量） | Active Memory 插件（before_prompt_build 拦截 + 子代理检索 + 缓存） |
| **质量保证** | 无显式质量检查 | 质量守卫（5 段必需 + 标识符保留 + 用户问题覆盖 + 可重试） |
| **成本追踪** | 精细 Token 计费（按模型聚合 + 会话持久化 + advisor 递归） | 上下文窗口守卫 + 使用量技能 |
| **推测执行** | Speculation（Copy-on-Write 文件隔离 + 流水线推测链） | 未发现 |
| **提示建议** | PromptSuggestion（Forked Agent + 12+ 过滤规则 + 缓存感知） | 未发现独立模块 |
| **分析遥测** | Datadog + 1P 日志（采样率 + PII 脱敏 + Proto 类型安全） | 会话事件日志 |

### 2.2 Microcompact — 轻量级压缩

#### 2.2.1 双路径策略

```
microcompactMessages(messages, context)
    │
    ├── 路径 A：时间触发（缓存冷启动场景）
    │   ├── 条件：距上次 assistant 消息 > 60 分钟（对齐服务端缓存 TTL）
    │   ├── 操作：保留最近 5 个工具结果，清除其余为占位文本
    │   └── 特点：直接修改消息内容，不依赖 API 特性
    │
    └── 路径 B：缓存编辑（缓存热场景）
        ├── 条件：工具结果注册数达到触发阈值
        ├── 操作：生成 cache_edits 块，由 API 层在请求时插入
        └── 特点：不修改本地消息，不破坏 prompt cache 前缀
```

**关键设计洞察**：
- **TTL 对齐**：默认 60 分钟阈值与服务端 1 小时缓存 TTL 对齐，确保只在缓存已过期时才触发清除
- **可压缩工具白名单**：仅对 FileRead、Shell、Grep、Glob、WebSearch、WebFetch、FileEdit、FileWrite 的结果进行压缩
- **Token 估算**：遍历所有内容块类型（text、tool_result、image、thinking、tool_use 等）并乘以 4/3 保守系数

#### 2.2.2 API 原生 Context Management

```
getAPIContextManagement()
    │
    ├── Thinking 块策略
    │   ├── 正常：keep: 'all'（保留所有 thinking）
    │   ├── 长时间空闲：keep: { type: 'thinking_turns', value: 1 }（仅保留最后一轮）
    │   └── redact-thinking 激活时：添加 clear_thinking 策略
    │
    ├── 工具结果清除（仅 ant 用户）
    │   ├── 触发阈值：默认 180K tokens
    │   ├── 目标：默认 40K tokens
    │   └── 范围：Read、Shell、Grep 等可清除工具
    │
    └── 工具调用清除（仅 ant 用户）
        ├── 范围：Edit、Write、NotebookEdit
        └── 通过 exclude_tools 指定
```

**关键设计洞察**：
- **API 原生能力利用**：直接使用 Anthropic API 的 context management 参数，而非客户端修改消息内容
- **角色分级**：工具清除策略仅对 ant 内部用户开放，外部用户仅获得 thinking 保留策略
- **不破坏缓存**：通过 API 参数而非消息修改实现，保证 prompt cache 前缀不受影响

### 2.3 AutoCompact — 自动压缩决策

#### 2.3.1 多层守卫体系

```
shouldAutoCompact(messages, querySource)
    │
    ├── 守卫 1：递归防护
    │   └── 查询源为 session_memory/compact/marble_origami → 跳过
    │
    ├── 守卫 2：Reactive Compact 抑制
    │   └── REACTIVE_COMPACT feature flag 激活 → 跳过
    │
    ├── 守卫 3：Context Collapse 抑制
    │   └── CONTEXT_COLLAPSE feature flag 激活 → 跳过
    │
    ├── 守卫 4：Circuit Breaker
    │   └── 连续失败 ≥ 3 次 → 跳过（防止无限重试）
    │
    └── 通过所有守卫 → 比较 token 数与阈值
        ├── 阈值 = 有效上下文窗口 - 13K buffer
        └── 有效窗口 = 模型上下文窗口 - summary 输出预留
```

#### 2.3.2 策略优先级链

```
autoCompactIfNeeded()
    │
    ├── 策略 1：SessionMemoryCompaction（优先）
    │   ├── 条件：feature flag + session memory 内容存在
    │   ├── 优势：无 API 调用开销（使用已提取的记忆替代摘要）
    │   ├── 滑动窗口：从 lastSummarizedMessageId 向前扩展
    │   ├── API 不变量保护：确保 tool_use/tool_result 配对
    │   └── 效果不足时回退到策略 2
    │
    └── 策略 2：compactConversation（回退）
        ├── Forked Agent 路径（复用主线程 prompt cache）
        ├── 直接流式调用路径（Fork 失败时）
        ├── PTL 重试循环（最多 3 次，逐轮截断头部）
        └── 附件生成（文件/Agent/Plan/Skill 恢复）
```

### 2.4 compact.ts — 核心压缩执行

#### 2.4.1 全量压缩流程

```
compactConversation(params)
    │
    ├── 1. Pre-compact Hooks（用户自定义前置处理）
    │
    ├── 2. streamCompactSummary（摘要生成）
    │   ├── 优先：runForkedAgent（复用 prompt cache）
    │   └── 回退：直接流式调用（可配置重试）
    │
    ├── 3. PTL 重试循环
    │   ├── compact 请求自身可能触发 prompt-too-long
    │   └── truncateHeadForPTLRetry：按 API 轮次分组，从最旧组开始丢弃
    │
    ├── 4. 附件生成（并行）
    │   ├── createPostCompactFileAttachments（最近读取文件恢复）
    │   ├── createPlanAttachmentIfNeeded（计划文件）
    │   ├── createSkillAttachmentIfNeeded（已调用技能，按 token 预算截断）
    │   ├── createPlanModeAttachmentIfNeeded（Plan 模式）
    │   └── createAsyncAgentAttachmentsIfNeeded（异步 Agent 状态）
    │
    ├── 5. Post-compact Hooks
    │
    └── 6. 返回 CompactionResult
```

#### 2.4.2 Prompt-too-long 递减截断

```
truncateHeadForPTLRetry(messages, tokenGap)
    │
    ├── groupMessagesByApiRound(messages) → 二维数组
    │   └── 基于 message.id 变化检测 API 轮次边界
    │
    ├── 从最旧组开始逐组丢弃
    │   ├── 计算每组的 token 节省
    │   └── 累计节省 ≥ tokenGap → 停止
    │
    └── 保护：至少保留一组（防止完全清空）
```

#### 2.4.3 摘要提示词设计

```
NO_TOOLS_PREAMBLE（强烈禁止工具调用）
    +
BASE_COMPACT_PROMPT（9 段式结构化摘要要求）
    ├─ ## Primary Request（主要请求）
    ├─ ## Key Concepts（关键概念）
    ├─ ## Files/Code（文件/代码）
    ├─ ## Errors/Fixes（错误/修复）
    ├─ ## Problem Solving（问题解决）
    ├─ ## All User Messages（所有用户消息）
    ├─ ## Pending Tasks（待办任务）
    ├─ ## Current Work（当前工作）
    └─ ## Optional Next Step（可选后续步骤）
    +
自定义指令（用户 Hook 注入）
    +
NO_TOOLS_TRAILER（再次提醒不使用工具）
```

**两阶段输出**：`<analysis>` 作为草稿思考区（提升质量），`<summary>` 作为最终输出（进入上下文）。

### 2.5 SessionMemoryCompaction — 实验性零 API 压缩

```
trySessionMemoryCompaction(messages, config)
    │
    ├── 检查 feature flag + session memory 内容
    │
    ├── 计算 lastSummarizedIndex
    │
    ├── calculateMessagesToKeepIndex（滑动窗口扩展）
    │   ├── 从 lastSummarizedIndex 向前扩展
    │   ├── 约束：minTokens（10K）+ minTextBlockMessages（5）
    │   ├── 上限：maxTokens（40K）
    │   └── 不跨越最后一个 compact boundary
    │
    ├── adjustIndexToPreserveAPIInvariants（向后扫描修复）
    │   ├── 确保 tool_result 都有对应 tool_use
    │   └── 确保相同 message.id 的 assistant 消息不被拆分
    │
    └── 验证：post-compact token 数 < autoCompactThreshold
        └── 超过 → 放弃，回退传统压缩
```

**关键设计洞察**：
- **零 API 成本**：利用持续提取的 session memory 替代 LLM 摘要，完全无 API 调用开销
- **API 不变量保护**：`adjustIndexToPreserveAPIInvariants` 通过向后扫描确保 tool_use/tool_result 配对和 streaming 消息合并的正确性
- **配置远程化**：通过 GrowthBook 动态配置实现运行时阈值调整

### 3.2 SessionMemory — 会话笔记

#### 3.2.1 触发机制

```
extractSessionMemory(messages, toolUseContext)
    │
    ├── 检查 feature flag（tengu_session_memory）
    ├── 检查自动压缩是否启用（互斥）
    │
    ├── shouldExtractMemory(messages)
    │   ├── 初始化阈值：token 增长 ≥ 10,000
    │   └── 更新阈值：token 增长 ≥ 5,000 且工具调用 ≥ 3 次
    │
    ├── 设置记忆文件（首次创建时加载模板）
    │
    ├── runForkedAgent（隔离子代理）
    │   ├── 共享 prompt cache
    │   ├── 仅允许 FileEdit 工具（指定记忆文件路径）
    │   └── 其他所有工具均被拒绝
    │
    └── 记录 token 用量和上下文大小
```

#### 3.2.2 Markdown 模板结构

```
# Session Title
## Current State
## Task specification
## Approach / Strategy
## Progress
## Errors & Corrections
## Key Insights / Learnings
## Files & Code
## Worklog
## Next Steps
```

**段落级 Token 预算管理**：每段落 2000 token，总计 12000 token。超出时在提示词末尾附加压缩提醒。

#### 3.2.3 关键设计模式

- **Forked Agent + Cache Sharing**：子代理复用父代理的 prompt cache，但通过 `canUseTool` 回调禁止除 FileEdit 外的所有工具
- **双重阈值触发**：同时要求 token 增长和工具调用次数，防止过度提取
- **顺序执行**：`sequential()` 包装确保同一时间只有一个提取在运行
- **互斥机制**：与自动压缩互斥，避免压缩和提取同时操作消息数组

### 3.3 extractMemories — 持久化记忆提取

#### 3.3.1 四类型分类体系

| 类型 | 可见性 | 范围 | 示例 |
|------|--------|------|------|
| `user` | 始终 private | 用户角色、目标、偏好 | "用户偏好 TypeScript 严格模式" |
| `feedback` | 默认 private | 行为指导 | "用户不喜欢过度解释" |
| `project` | 偏 team | 项目工作、事件 | "项目使用 pnpm monorepo" |
| `reference` | 通常 team | 外部系统指针 | "Linear 项目 ID: ENG-123" |

#### 3.3.2 游标式增量处理

```
initExtractMemories() → 闭包作用域状态
    │
    ├── lastMemoryMessageUuid（游标位置）
    ├── inFlightExtractions（进行中的提取 Set）
    ├── pendingContext（待处理的上下文队列）
    │
    └── executeExtractMemories(context)
        ├── 计算新增消息数（当前末尾 - 游标位置）
        ├── 互斥检查：主代理已直接写记忆？→ 跳过
        ├── 扫描现有记忆文件 → 生成清单
        ├── 构建提示词（含现有记忆清单，避免重复）
        ├── runForkedAgent（限制最多 5 轮）
        │   ├── 可用工具：Read/Grep/Glob/只读 Bash/记忆目录内 Edit+Write
        │   └── 并行读取 → 并行写入策略
        └── 推进游标
```

**尾部提取（Trailing Run）**：提取进行中到达的新请求被暂存，当前提取完成后自动执行一次追加提取。

### 3.4 AutoDream — 记忆整合

#### 3.4.1 三级门控机制

```
executeAutoDream(context)
    │
    ├── 门控 1：时间门（成本最低）
    │   └── 距上次整合 ≥ minHours（默认 24 小时）
    │
    ├── 门控 2：会话门
    │   └── 累积会话 ≥ minSessions（默认 5 个）
    │   └── 扫描节流：10 分钟间隔避免频繁扫描
    │
    ├── 门控 3：锁门（成本最高）
    │   └── tryAcquireConsolidationLock()
    │       ├── 检查现有 PID 是否存活
    │       ├── 超过 1 小时视为过期（防 PID 复用）
    │       └── 写入 PID 后重读验证竞争
    │
    └── 通过所有门控 → runForkedAgent 执行整合
        └── 失败时回滚锁文件 mtime
```

#### 3.4.2 四阶段整合流程

```

Phase 2: 收集信号 — 从日志和会话中提取新信息
Phase 3: 整合 — 合并/更新记忆文件
Phase 4: 修剪和索引 — 更新入口文件（控制在 25KB 以内）
```

#### 3.4.3 文件锁双用途设计

```
.consolidate-lock 文件
├── mtime → lastConsolidatedAt 时间戳
├── 文件体 → 持有者 PID
├── 获取锁：检查 PID 存活 → 写入 PID → 重读验证
└── 回滚：恢复 mtime 到获取前的值
```

### 6.2 三阶段梦境模型

**设计哲学**：模拟人类睡眠的 Light/Deep/REM 阶段，每阶段有不同的频率、成本和思考深度。

| 阶段 | 频率 | 成本 | 思考深度 | 核心任务 |
|------|------|------|---------|---------|
| **Light** | 每 6 小时 | 低 | 快速 | 快速整合新记忆，清理过期条目 |
| **Deep** | 每天凌晨 3 点 | 中 | 高 | 深度重组记忆结构，半衰期衰减（14 天） |
| **REM** | 每周日 | 高 | 最高 | 全面反思和模式发现 |

**半衰期衰减算法**：
```
recencyScore = 2^(-ageDays / recencyHalfLifeDays)
```
记忆的新鲜度随时间指数衰减，14 天后半衰期使旧记忆权重降至 50%。

#### 6.5.2 质量守卫详解

**5 段必需结构**：
```
## Decisions         — 已做出的决策及理由
## Open TODOs        — 待办事项
## Constraints/Rules — 约束和规则
## Pending user asks — 用户待处理请求
## Exact identifiers  — 精确标识符（hex ID、URL、路径等）
```

**标识符提取**：正则匹配 hex 字符串（8+ 位）、URL、文件路径、端口号等，最多 12 个。

**用户问题覆盖检测**：用停用词过滤后的 token 集合判断摘要是否覆盖了用户最新问题。

#### 6.5.3 Context Pruning — 两级修剪

```
pruneContextMessages(messages, settings)
    │
    ├── 字符估算：contextWindow * CHARS_PER_TOKEN_ESTIMATE
    │   └── 支持 CJK 字符权重估算
    │
    ├── Soft Trim（上下文 > 30%）
    │   ├── 保护：第一条用户消息之前的内容不修剪
    │   ├── 保护：保留最后 3 条 assistant 消息
    │   └── 可修剪工具结果：保留 headChars + tailChars
    │
    └── Hard Clear（上下文 > 50%）
        └── 工具结果内容 → [Old tool result content cleared]
```

**工具可剪裁性谓词**：通过 `makeToolPrunablePredicate(match)` 构建，支持 deny（黑名单）+ allow（白名单）glob 模式。

## 八、通用设计模式提炼

### 8.1 可直接复用的模式

#### 模式 1：三级递进压缩（claude-code）

```
适用场景：需要在性能和信息保留之间取得平衡
实现方式：API 原生 → 轻量级（无 API 调用）→ 重量级（LLM 摘要）→ 紧急响应
优势：每层独立运行，轻量层处理大部分场景，重量级仅在必要时触发
```

#### 模式 2：质量守卫循环（openclaw）

```
适用场景：需要确保 LLM 生成的摘要质量
实现方式：5 段必需检查 + 标识符保留 + 用户问题覆盖 + 带反馈重试（最多 3 次）
优势：自动检测和修复低质量摘要，无需人工干预
```

#### 模式 3：三阶段梦境模型（openclaw）

```
适用场景：需要定期整理和优化知识库
实现方式：Light（高频低成本）→ Deep（低频中成本）→ REM（极低频高成本）
优势：模拟人类睡眠周期，平衡整理频率和资源消耗
```

#### 模式 4：半衰期衰减算法（openclaw）

```
适用场景：需要评估记忆/知识的新鲜度
实现方式：recencyScore = 2^(-ageDays / halfLifeDays)
优势：平滑衰减，可配置半衰期，比硬截止更自然
```

#### 模式 5：Copy-on-Write 推测执行（claude-code）

```
适用场景：需要预测用户下一步操作以减少等待时间
实现方式：overlay 目录隔离写入 + 边界检测 + 流水线推测链
优势：零风险预执行，失败时完全回退
```

#### 模式 6：Forked Agent + Cache Sharing（claude-code）

```
适用场景：需要在后台执行 LLM 任务但不增加 API 成本
实现方式：runForkedAgent 共享父代理的 prompt cache 前缀
优势：子代理复用已缓存的系统提示和工具定义，仅支付增量 token 成本
关键约束：不修改 cache-key 参数（如 effort/model），否则缓存失效
```

#### 模式 7：游标式增量处理（claude-code）

```
适用场景：需要从持续增长的数据流中增量提取信息
实现方式：lastMessageUuid 游标 + 互斥检测 + 尾部提取
优势：避免重复处理，支持并发安全
```

#### 模式 8：文件锁双用途（claude-code）

```
适用场景：需要互斥访问同时记录时间戳
实现方式：锁文件的 mtime 即为最后操作时间，文件体为持有者 PID
优势：单一文件承担两个职责，减少文件系统操作
```

#### 模式 9：主动记忆召回（openclaw）

```
适用场景：需要在用户提问前主动注入相关记忆
实现方式：before_prompt_build 事件拦截 + 受限子代理 + 噪声过滤
优势：用户无需显式请求，系统自动判断是否需要注入记忆
```

#### 模式 10：嵌入引擎自动发现与降级（openclaw）

```
适用场景：需要支持多种嵌入后端并保证可用性
实现方式：auto 模式逐个尝试 → 全部失败 → FTS-only 优雅降级
优势：零配置可用，API Key 缺失不报错
```

|------|------|---------|

| JSONL Session 文件 | openclaw | 适用于单机场景，分布式需替换为数据库 |

## 九、为后续轮次准备的关键发现

### 9.1 第 6 轮（通信协议）应关注的问题

1. **claude-code 的 Forked Agent 模式是否可以视为一种轻量级"进程内通信"？** 共享 prompt cache + 隔离执行
2. **openclaw 的 Active Memory 插件的 `before_prompt_build` 事件拦截是否可以推广为通用的"上下文注入"协议？**
3. **claude-code 的文件锁双用途设计是否适用于分布式场景？** PID 检测在分布式环境中无效

### 9.2 第 7 轮（服务层与状态管理）应关注的问题

1. **claude-code 的成本追踪（按模型聚合 + 会话持久化）是否可以应用于 openclaw 的多提供商场景？**
2. **openclaw 的三阶段梦境模型是否可以作为通用的"知识库维护"框架？**
3. **claude-code 的分析遥测 PII 脱敏策略是否足够？** MCP 工具名默认替换为 `mcp_tool` 可能丢失有价值的调试信息

### 9.3 第 8 轮（插件与扩展系统）应关注的问题

1. **openclaw 的 Active Memory 插件架构（事件拦截 + 受限子代理 + 缓存）是否可以作为通用插件开发范本？**
2. **claude-code 的质量守卫缺失是否是一个需要弥补的短板？** openclaw 的 5 段必需 + 标识符保留检查显著提升了摘要质量
