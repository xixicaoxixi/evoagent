### 1.2 核心设计特点

1. **AST 扫描自注册**：工具文件在模块级别调用 `registry.register()`，通过 AST 扫描发现而非显式列表
2. **三层导入架构**：registry.py（无依赖）→ tools/*.py（导入 registry）→ model_tools.py（导入全部），彻底避免循环导入
3. **三层审批防线**：Hardline（无条件阻止）→ Dangerous（模式匹配）→ Smart Approval（LLM 风险评估）
4. **ContextVars 会话隔离**：网关并发场景下，每个线程有独立的审批会话上下文
5. **工具集组合系统**：`includes` 支持递归嵌套，自动检测循环依赖
6. **动态 Schema 重建**：`execute_code` 和 `discord` 工具根据可用工具动态重建 schema

## 三、关键设计模式

### 3.1 AST 扫描自注册

**模式名称**：声明式自发现

**应用位置**：`discover_builtin_tools()` (registry.py 行 57-74)

**代码示例**：
```python
def _is_registry_register_call(node: ast.AST) -> bool:
    """Return True when *node* is a ``registry.register(...)`` call expression."""
    if not isinstance(node, ast.Expr) or not isinstance(node.value, ast.Call):
        return False
    func = node.value.func
    return (
        isinstance(func, ast.Attribute)
        and func.attr == "register"
        and isinstance(func.value, ast.Name)
        and func.value.id == "registry"
    )

def _module_registers_tools(module_path: Path) -> bool:
    """Only inspects module-body statements so that helper modules
    which happen to call ``registry.register()`` inside a function
    are not picked up."""
    source = module_path.read_text(encoding="utf-8")
    tree = ast.parse(source, filename=str(module_path))
    return any(_is_registry_register_call(stmt) for stmt in tree.body)
```

**设计亮点**：
- 无需维护显式工具列表，新增工具文件自动发现
- AST 过滤避免导入纯 helper 模块（如 `utils.py`）
- 仅在模块顶层调用 `register()` 才被视为工具模块

### 3.2 代际计数器缓存失效

**模式名称**：版本向量

**应用位置**：`ToolRegistry._generation` (registry.py 行 159)

**代码示例**：
```python
class ToolRegistry:
    def __init__(self):
        self._generation: int = 0

    def register(self, ...):
        with self._lock:
            ...
            self._generation += 1

# 在 model_tools.py 中缓存工具定义
def get_tool_definitions(enabled_toolsets, disabled_toolsets, quiet_mode):
    cache_key = (
        frozenset(enabled_toolsets),
        frozenset(disabled_toolsets),
        registry._generation,  # 自动失效
```

**设计亮点**：
- 无需显式缓存失效 API，任何注册/注销操作自动递增 generation
- 缓存键包含 generation，旧缓存自动失效
- 线程安全：generation 更新在锁内完成

### 3.3 ContextVars 会话隔离

**模式名称**：上下文本地存储

**应用位置**：`_approval_session_key` (approval.py 行 30-33)

**代码示例**：
```python
_approval_session_key: contextvars.ContextVar[str] = contextvars.ContextVar(
    "approval_session_key",
    default="",

def set_current_session_key(session_key: str) -> contextvars.Token[str]:
    """Bind the active approval session key to the current context."""

    1. approval-specific contextvars
    2. session_context contextvars
    3. os.environ fallback"""
```

**设计亮点**：
- 网关并发场景下，每个 executor 线程有独立的审批状态
- 无需传递 session_key 参数，自动从上下文解析
- 三层回退：ContextVars → session_context → os.environ

### 3.4 网关阻塞审批队列

**模式名称**：生产者-消费者 + 事件通知

**应用位置**：`_gateway_queues` (approval.py 行 390-445)

**代码示例**：
```python
class _ApprovalEntry:
    __slots__ = ("event", "data", "result")
    def __init__(self, data: dict):
        self.event = threading.Event()
        self.data = data
        self.result: Optional[str] = None

_gateway_queues: dict[str, list] = {}  # session_key → [_ApprovalEntry, ...]

def check_all_command_guards(command, env_type):
    # ... 检测到需要审批 ...
    entry = _ApprovalEntry(approval_data)
    with _lock:
        _gateway_queues.setdefault(session_key, []).append(entry)

    # 通知网关发送审批请求给用户
    notify_cb(approval_data)

    # 阻塞等待用户响应（带心跳）
    while True:
        _remaining = _deadline - time.monotonic()
        if entry.event.wait(timeout=min(1.0, _remaining)):
            break
        touch_activity_if_due(...)  # 防止网关不活跃超时

    choice = entry.result  # "once" | "session" | "always" | "deny"

def resolve_gateway_approval(session_key, choice, resolve_all=False):
    """由网关的 /approve 或 /deny handler 调用"""
    with _lock:
        queue = _gateway_queues.get(session_key)
        targets = list(queue) if resolve_all else [queue.pop(0)]
    for entry in targets:
        entry.result = choice
        entry.event.set()
```

**设计亮点**：
- 支持并发审批：多个子 Agent 同时等待不同审批
- `/approve all` 批量处理队列中的所有审批
- 心跳机制防止网关不活跃超时（默认 1800s）

### 3.5 智能审批（Smart Approval）

**模式名称**：辅助 LLM 风险评估

**应用位置**：`_smart_approve()` (approval.py 行 743-787)

**代码示例**：
```python
def _smart_approve(command: str, description: str) -> str:
    """Use the auxiliary LLM to assess risk and decide approval.
    Inspired by OpenAI Codex's Smart Approvals guardian subagent."""
    prompt = f"""You are a security reviewer for an AI coding agent.
Command: {command}
Flagged reason: {description}

Assess the ACTUAL risk of this command. Many flagged commands are false positives.

Rules:
- APPROVE if clearly safe (benign script execution, safe file operations, ...)
- DENY if genuinely dangerous (recursive delete, overwriting system files, ...)
- ESCALATE if uncertain

Respond with exactly one word: APPROVE, DENY, or ESCALATE"""

    response = call_llm(
        task="approval",
        messages=[{"role": "user", "content": prompt}],
        temperature=0,
        max_tokens=16,
    )
    answer = response.choices[0].message.content.strip().upper()
    return "approve" if "APPROVE" in answer else "deny" if "DENY" in answer else "escalate"
```

**设计亮点**：
- 减少用户干扰：明显安全的命令自动通过
- 明确危险的命令自动阻止
- 不确定时升级到人工审批
- 使用专用 "approval" 任务配置，可独立配置模型

### 3.6 参数类型强制

**模式名称**：Schema 驱动的类型转换

**应用位置**：`coerce_tool_args()` (model_tools.py 行 503-574)

**代码示例**：
```python
def coerce_tool_args(tool_name: str, args: Dict[str, Any]) -> Dict[str, Any]:
    """Coerce tool call arguments to match their JSON Schema types.

    LLMs frequently return numbers as strings ("42" instead of 42)
    and booleans as strings ("true" instead of true)."""
    schema = registry.get_schema(tool_name)
    properties = (schema.get("parameters") or {}).get("properties")

    for key, value in list(args.items()):
        prop_schema = properties.get(key)
        expected = prop_schema.get("type")

        # Wrap bare non-list values when schema declares array
        if expected == "array" and value is not None and not isinstance(value, (list, tuple)):
            if isinstance(value, str):
                coerced = _coerce_value(value, expected, schema=prop_schema)
                if coerced is not value:
                    args[key] = coerced
                    continue
            args[key] = [value]  # 包裹为单元素列表
            continue

        if not isinstance(value, str):
            continue
        coerced = _coerce_value(value, expected, schema=prop_schema)
        if coerced is not value:
            args[key] = coerced

    return args

def _coerce_value(value: str, expected_type, schema):
    if isinstance(expected_type, list):  # Union type
        for t in expected_type:
            result = _coerce_value(value, t, schema=schema)
            if result is not value:
                return result
        return value
    if expected_type in ("integer", "number"):
        return _coerce_number(value, integer_only=(expected_type == "integer"))
    if expected_type == "boolean":
        return _coerce_boolean(value)
    if expected_type == "array":
        return _coerce_json(value, list)
    if expected_type == "object":
        return _coerce_json(value, dict)
```

**设计亮点**：
- 解决 DeepSeek/Qwen/GLM 等开源模型的类型漂移问题
- 支持 Union 类型（`["integer", "string"]`）
- 自动包裹裸值为列表（`{"urls": "https://a.com"}` → `{"urls": ["https://a.com"]}`）
- 处理 nullable 类型（`"null"` → `None`）

### 3.7 动态 Schema 重建

**模式名称**：运行时 Schema 生成

**应用位置**：`_compute_tool_definitions()` (model_tools.py 行 400-460)

**代码示例**：
```python
def _compute_tool_definitions(...):
    # Rebuild execute_code schema to only list sandbox tools that are
    # actually available. Without this, the model sees "web_search is
    # available in execute_code" even when the API key isn't configured.
    if "execute_code" in available_tool_names:
        sandbox_enabled = SANDBOX_ALLOWED_TOOLS & available_tool_names
        dynamic_schema = build_execute_code_schema(sandbox_enabled, mode=_get_execution_mode())
        for i, td in enumerate(filtered_tools):
            if td.get("function", {}).get("name") == "execute_code":
                filtered_tools[i] = {"type": "function", "function": dynamic_schema}
                break

    # Rebuild discord schemas based on the bot's privileged intents
    _discord_schema_fns = {
        "discord": "get_dynamic_schema_core",
        "discord_admin": "get_dynamic_schema_admin",
    }
    for discord_tool_name in _discord_schema_fns:
        if discord_tool_name in available_tool_names:
            schema_fn = getattr(_dt, _discord_schema_fns[discord_tool_name])
            dynamic = schema_fn()  # 根据 intents 动态生成
            if dynamic is None:
                # Bot 没有所需 intents，从列表中移除
                filtered_tools = [t for t in filtered_tools if ...]
```

**设计亮点**：
- `execute_code` 的 sandbox 可用工具列表实时更新
- Discord 工具根据 bot 的 privileged intents 动态调整
- 避免模型看到不可用工具的幻觉调用

**移植建议**：使用 TypeScript 的 `fs` + `esbuild`/`swc` 解析 AST

```typescript
// TypeScript 移植示例
import { parse } from '@swc/core';
import * as fs from 'fs';
import * as path from 'path';

interface ToolModule {
  register: (registry: ToolRegistry) => void;
}

async function discoverBuiltinTools(toolsDir: string): Promise<string[]> {
  const files = fs.readdirSync(toolsDir)
    .filter(f => f.endsWith('.ts') && !['index.ts', 'registry.ts'].includes(f));
  
  const modules: string[] = [];
  for (const file of files) {
    const source = fs.readFileSync(path.join(toolsDir, file), 'utf-8');
    const ast = await parse(source, { syntax: 'typescript' });
    
    // 检查顶层是否有 registry.register 调用
    if (hasRegisterCall(ast)) {
      const moduleName = `./tools/${file.replace('.ts', '')}`;
      await import(moduleName);  // 触发注册
      modules.push(moduleName);
    }
  }
  return modules;
}
```

```typescript
function coerceToolArgs(toolName: string, args: Record<string, unknown>): Record<string, unknown> {
  const schema = registry.getSchema(toolName);
  if (!schema?.parameters?.properties) return args;
  
  const result = { ...args };
  for (const [key, propSchema] of Object.entries(schema.parameters.properties)) {
    const value = result[key];
    if (value === undefined) continue;
    
    // 数组包裹
    if (propSchema.type === 'array' && !Array.isArray(value) && value !== null) {
      result[key] = [value];
      continue;
    }
    
    // 字符串类型强制
    if (typeof value === 'string') {
      if (propSchema.type === 'integer') {
        const parsed = parseInt(value, 10);
        if (!isNaN(parsed)) result[key] = parsed;
      } else if (propSchema.type === 'boolean') {
        if (value.toLowerCase() === 'true') result[key] = true;
        else if (value.toLowerCase() === 'false') result[key] = false;
      }
    }
  }
  return result;
}
```

### 7.1 异步桥接（_run_async）

**位置**：model_tools.py 行 82-173

```python
def _run_async(coro):
    """Run an async coroutine from a sync context.

    If the current thread already has a running event loop (e.g., inside
    the gateway's async stack), we spin up a disposable thread so
    asyncio.run() can create its own loop without conflicting.

    For the common CLI path (no running loop), we use a persistent event
    loop so that cached async clients (httpx / AsyncOpenAI) remain bound
    to a live loop and don't trigger "Event loop is closed" on GC.
    """
    try:
        loop = asyncio.get_running_loop()
    except RuntimeError:
        loop = None

    if loop and loop.is_running():
        # Inside an async context — run in a fresh thread
        import concurrent.futures

        worker_loop: Optional[asyncio.AbstractEventLoop] = None
        loop_ready = threading.Event()

        def _run_in_worker():
            nonlocal worker_loop
            worker_loop = asyncio.new_event_loop()
            loop_ready.set()
            try:
                asyncio.set_event_loop(worker_loop)
                return worker_loop.run_until_complete(coro)
            finally:
                # Cancel anything still pending
                pending = asyncio.all_tasks(worker_loop)
                for t in pending:
                    t.cancel()
                if pending:
                    worker_loop.run_until_complete(
                        asyncio.gather(*pending, return_exceptions=True)
                    )
                worker_loop.close()

        pool = concurrent.futures.ThreadPoolExecutor(max_workers=1)
        future = pool.submit(_run_in_worker)
        try:
            return future.result(timeout=300)
        except concurrent.futures.TimeoutError:
            # Cancel the coroutine inside its own loop
            if loop_ready.wait(timeout=1.0) and worker_loop is not None:
                for t in asyncio.all_tasks(worker_loop):
                    worker_loop.call_soon_threadsafe(t.cancel)
            raise
        finally:
            pool.shutdown(wait=False)

    # Worker thread path
    if threading.current_thread() is not threading.main_thread():
        worker_loop = _get_worker_loop()
        return worker_loop.run_until_complete(coro)

    # Main thread path
    tool_loop = _get_tool_loop()
    return tool_loop.run_until_complete(coro)
```

## 八、总结

### 8.1 核心收获

1. **AST 扫描自注册是最优雅的发现机制**：无需维护显式列表，新增工具文件自动发现
2. **三层导入架构彻底解决了循环导入问题**：registry → tools → model_tools 的层级设计
3. **三层审批防线是生产级安全设计**：Hardline（无条件）→ Dangerous（模式）→ Smart（LLM）
4. **ContextVars 是会话隔离的正确方案**：网关并发场景下每个线程有独立的审批状态
5. **代际计数器是缓存失效的优雅模式**：无需显式 API，自动失效
6. **参数类型强制解决了开源模型的类型漂移**：DeepSeek/Qwen/GLM 的 `"42"` → `42` 转换

### 8.2 对 EvoAgent 的建议

1. **采纳 AST 扫描自注册**：使用 swc/esbuild 解析 TypeScript AST
2. **采纳三层审批防线**：直接移植 Hardline 和 Dangerous 模式
3. **采纳 ContextVars 会话隔离**：使用 Node.js AsyncLocalStorage
4. **采纳参数类型强制**：使用 Zod 进行 Schema 验证和转换
5. **采纳代际计数器缓存失效**：简洁高效的缓存失效模式
6. **改进全局状态**：使用依赖注入替代模块级全局变量
7. **改进同步阻塞**：网关场景使用 async/await 替代 threading.Event
