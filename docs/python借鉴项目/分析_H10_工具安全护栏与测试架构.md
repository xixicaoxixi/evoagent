## 一、核心发现摘要

### 1.1 ToolGuardrails：幂等/变异分类 + per-turn 状态机

`agent/tool_guardrails.py`（455 行）实现了一个**纯函数决策引擎**，对工具调用进行安全分类和重复调用检测：

**三类工具分类**：
- **幂等（Idempotent）**：`read_file`、`search_files`、`web_search`、`list_directory` 等——重复调用安全，无副作用
- **变异（Mutating）**：`write_file`、`patch`、`terminal`、`execute_code` 等——有副作用，重复调用可能导致数据损坏
- **有状态（Stateful）**：需要 per-turn 观察跟踪的工具

**per-turn 状态机**：
- `before_call(tool, args)` → 幂等工具始终放行，变异工具检查重复
- `after_call(result, failed)` → 记录失败签名或清除，返回 Decision

**设计意图**：LLM 在某些场景下会反复执行相同的失败操作（如写入只读文件）。护栏系统检测这种行为模式，通过 warn/block 机制打断循环，避免浪费 token 和 API 调用。

### 1.2 CredentialFiles：凭证文件挂载与路径安全

`tools/credential_files.py`（~300 行）管理凭证文件（如 `token.json`、`credentials.json`）的安全挂载：

**核心机制**：
- **注册**：`register_credential_files(files)` 注册凭证文件路径
- **验证**：所有路径必须通过 `validate_within_dir()` 检查（防止路径遍历）
- **挂载**：`get_credential_file_mounts()` 返回容器内挂载点映射
- **ContextVar 隔离**：每会话独立的凭证文件集合

### 1.3 EnvPassthrough：环境变量白名单传递

`tools/env_passthrough.py`（~200 行）实现环境变量的安全传递机制：

**三层来源合并**：
1. **配置层**：`config.yaml` 中的 `terminal.env_passthrough` 列表
2. **运行时层**：`register_env_passthrough()` 动态注册
3. **内置白名单**：`_BUILTIN_PASSTHROUGH`（`PATH`、`HOME`、`TERM`、`LANG` 等基础变量）

**安全设计**：
- **默认拒绝**：不在白名单中的环境变量不会传递给子进程
- **Provider 变量阻断**：`_HERMES_PROVIDER_ENV_BLOCKLIST` 动态构建，包含所有 provider 的 API key 和 base URL 环境变量
- **ContextVar 隔离**：每会话独立的 passthrough 集合

### 1.4 ProcessRegistry：进程生命周期管理 + 双层 Watch 熔断

`tools/process_registry.py`（~400 行）管理后台进程的完整生命周期：

**核心功能**：
- **进程注册**：`register()` 创建 `ProcessSession`，分配唯一 ID
- **输出监控**：`watch_patterns` 实时匹配进程输出中的关键词
- **完成通知**：`notify_on_complete` 进程退出时发送通知
- **进程清理**：`cleanup()` 终止孤儿进程

**双层 Watch 熔断器**：
- 第一层：per-session 熔断（cooldown + strike 机制）
- 第二层：全局熔断（跨所有 session 的滑动窗口）

**设计意图**：当进程输出大量匹配行时（如编译日志中的重复警告），无限制的通知会淹没 agent 上下文。双层熔断确保通知频率可控，且在极端情况下优雅降级。

## 三、关键设计模式

### 3.1 ToolCallSignature：敏感值剥离哈希

**设计意图**：
1. **敏感值不参与哈希**：API key 的具体值不影响"是否重复调用"的判断
2. **长值截断**：避免大文件内容导致哈希无意义（每次内容不同但语义相同）
3. **可审计**：`metadata` 保留规范化后的参数（脱敏），用于日志和调试

### 3.2 幂等分类的声明式注册

**设计意图**：`frozenset` 不可变且 O(1) 查找。分类逻辑是纯函数——给定工具名，返回分类，无副作用，可独立测试。

### 3.3 ToolCallGuardrailConfig：软启动策略

**软启动设计**：默认 `hard_stop_enabled=False`，只发出警告不阻止。用户可以通过配置启用硬阻止。这避免了误报导致合法操作被中断，同时仍然提供可观测性。

### 3.4 EnvPassthrough 的三层合并

**Provider 变量阻断优先级**：即使变量在白名单中，如果它在 `_HERMES_PROVIDER_ENV_BLOCKLIST` 中，仍然会被阻断。这防止了通过 `env_passthrough` 配置意外泄露 API key。

### 3.5 ProcessRegistry 的 Suppress-After-Exit 机制

**设计意图**：
1. **Suppress-after-exit**：进程退出后，缓冲区中可能还有未读数据。这些数据是"噪声"，不应触发通知。
2. **Strike 机制**：连续超速（每次 cooldown 窗口内都有事件）表明进程输出持续高频匹配。达到 limit 后禁用 watch，降级为 `notify_on_complete`（进程结束时发送一次汇总）。
3. **全局熔断**：跨所有 session 的二级保护，防止多个进程同时高频输出导致通知风暴。

### 3.6 AnsiStrip：ECMA-48 全覆盖

**快速路径优化**：`_HAS_ESCAPE` 预检查文本中是否包含 `\x1b` 或 `\x9b`。大多数输出不包含 ANSI 序列，快速路径避免不必要的正则匹配。

## 五、对 EvoAgent 的参考价值

### 5.1 应采纳的模式

1. **幂等/变异工具分类**：声明式 `frozenset` 分类 + 纯函数决策。分类逻辑无副作用，可独立测试，可扩展。

2. **敏感值剥离哈希**：计算工具调用签名时剥离 API key 等敏感值。避免密钥出现在日志和哈希中。

3. **三级决策（allow/warn/block）+ 软启动**：默认只警告不阻止，用户可配置升级为硬阻止。平衡安全性和可用性。

4. **环境变量默认拒绝 + Provider 自动阻断**：不在白名单中的变量不传递，Provider API key 自动阻断。动态构建阻断列表，新增 provider 时自动生效。

5. **双层 Watch 熔断**：per-session strike 机制 + 全局滑动窗口熔断。防止通知风暴，优雅降级。

6. **全局 conftest 的四大 hermetic 不变量**：环境隔离 + 模块状态重置 + 事件循环 + 超时保护。确保测试之间完全独立。

7. **Suppress-after-Exit**：进程退出后的延迟输出不触发通知。消除"进程已结束但通知仍在到达"的噪声。

### 5.2 应改进的模式

1. **幂等分类的硬编码列表**：`_IDEMPOTENT_TOOLS` 和 `_MUTATING_TOOLS` 是静态列表，新增工具需要手动添加。建议改为工具注册时的声明式属性（如 `@tool(idempotent=True)`）。

2. **缺失的安全模块**：`path_security.py`、`redact.py`、`tirith_security.py` 三个关键安全模块不在当前代码库中。这表明安全层可能存在未完成的实现。

3. **测试超时仅 Unix**：`_enforce_test_timeout` 使用 `SIGALRM`，Windows 不支持。建议使用 `threading.Timer` 作为跨平台替代。

### 5.3 创新点

1. **Cooldown + Strike 降级**：不是简单的速率限制，而是连续超速后逐步降级（suppress → strike → disable watch → notify_on_complete）。每一步都有明确的状态转换和用户可观测的事件。

2. **动态 Provider 阻断**：从 provider 注册表自动提取 API key 环境变量名，无需维护硬编码列表。新增 provider 时零配置安全。

3. **SHA-256 短哈希签名**：取 SHA-256 的前 16 位十六进制字符（64 bit），在碰撞概率极低（~2^64 种可能）的同时保持签名的简洁性。

4. **ANSI 全覆盖剥离**：覆盖 ECMA-48 完整规范，包括罕见的 DCS/SOS/PM/APC 序列。快速路径优化使无 ANSI 的文本零开销。

## 六、动态发现

### 6.1 缺失的安全模块

| 模块 | 被引用位置 | 推断功能 |
|------|-----------|---------|
| `tools/path_security.py` | credential_files, skill_manager_tool, cronjob_tools, skill_tools | `validate_within_dir()` 路径遍历防护 |
| `agent/redact.py` | 10+ 个生产文件 | `redact_sensitive_text()`, `RedactingFormatter` 敏感信息脱敏 |
| `tools/tirith_security.py` | gateway/run, approval, 3 个测试文件 | 安全策略引擎（名称暗示为安全框架） |

这三个模块被广泛引用但不在当前提取的代码库中，属于**待创建的 P0 安全文件**。

### 6.2 ANSI 转义剥离的完整性

`tools/ansi_strip.py` 覆盖了完整的 ECMA-48 标准：
- **CSI**：`ESC [ ... final_byte`（SGR 颜色、光标移动等）
- **OSC**：`ESC ] ... BEL/ST`（窗口标题、超链接等）
- **DCS/SOS/PM/APC**：`ESC { \| } ~ ... BEL/ST`（设备控制字符串等）
- **C1 8-bit**：`0x80-0x9F` 范围（8 位控制字符）
- **nF**：`ESC` + `0x40-0x5F`
- **Fp/Fe/Fs**：`ESC` + `0x60-0x7E`

这比大多数 ANSI 剥离实现更完整，防止了通过罕见转义序列进行的终端注入攻击。

### 6.3 测试中的安全回归测试

`test_cron_prompt_injection_skill.py` 是 #3968 漏洞的回归测试，验证技能内容中的注入载荷在运行时被检测到。这表明安全测试是**事件驱动**的——每次发现漏洞后添加回归测试。

### 6.4 Provider 环境变量阻断的动态构建

`_HERMES_PROVIDER_ENV_BLOCKLIST` 不是硬编码列表，而是从 `PROVIDER_REGISTRY` 和 `OPTIONAL_ENV_VARS` 动态构建。这意味着添加新 provider 时，其 API key 环境变量会自动被阻断，无需手动更新阻断列表。

### 7.2 工具调用签名（tool_guardrails.py 行 126-141）

```python
@dataclass(frozen=True)
class ToolCallSignature:
    """Stable, non-reversible identity for a tool name plus canonical args."""

    tool_name: str
    args_hash: str

    @classmethod
    def from_call(cls, tool_name: str, args: Mapping[str, Any] | None) -> "ToolCallSignature":
        canonical = canonical_tool_args(args or {})
        return cls(tool_name=tool_name, args_hash=_sha256(canonical))

    def to_metadata(self) -> dict[str, str]:
        """Return public metadata without raw argument values."""
        return {"tool_name": self.tool_name, "args_hash": self.args_hash}
```

**设计亮点**：`frozen=True` 确保签名不可变，`to_metadata()` 方法提供脱敏的公开元数据。

### 7.3 护栏决策（tool_guardrails.py 行 143-173）

### 7.4 双层 Watch 熔断（process_registry.py 行 61-74）

```python
# Watch pattern rate limiting — PER SESSION.
# Hard rule: at most ONE watch-match notification every WATCH_MIN_INTERVAL_SECONDS.
# Any match arriving inside that cooldown window is dropped and counted as a strike.
# After WATCH_STRIKE_LIMIT consecutive strike windows, watch_patterns for that
# session is permanently disabled and the session falls back to notify_on_complete
# semantics (one notification when the process actually exits).
WATCH_MIN_INTERVAL_SECONDS = 15   # Minimum spacing between consecutive watch matches
WATCH_STRIKE_LIMIT = 3            # Strikes in a row → disable watch + promote to notify_on_complete

# Global circuit breaker — across all sessions. Secondary safety net so concurrent
# siblings can't collectively flood the user even when each is under its own cap.
WATCH_GLOBAL_MAX_PER_WINDOW = 15
WATCH_GLOBAL_WINDOW_SECONDS = 10
WATCH_GLOBAL_COOLDOWN_SECONDS = 30
```

**设计亮点**：双层熔断设计——per-session 的 strike 机制 + 全局滑动窗口熔断，确保通知频率可控。

### 7.5 Suppress-After-Exit 机制（process_registry.py 行 189-209）

```python
def _check_watch_patterns(self, session: ProcessSession, new_text: str) -> None:
    """Scan new output for watch patterns and queue notifications."""
    if not session.watch_patterns or session._watch_disabled:
        return
    # Suppress-after-exit: once the reader loop has declared the process
    # exited, any late chunk we still see is post-exit noise. Dropping these
    # prevents the "stale notifications delivered minutes after the process
    # ended" spam when completion_queue consumers run async.
    if session.exited:
        return

    # Scan new text line-by-line for pattern matches
    matched_lines = []
    for line in new_text.splitlines():
        for pat in session.watch_patterns:
            if pat in line:
                matched_lines.append(line.rstrip())
                break
```

**设计亮点**：进程退出后的延迟输出不触发通知，消除"进程已结束但通知仍在到达"的噪声。

### 7.6 Strike 降级机制（process_registry.py 行 227-243）

```python
# Case 1: still inside the cooldown from the last emission.
# Count this as a strike for the current window (only once per window)
# and drop the event. If we've hit the strike limit, disable watch
# and promote to notify_on_complete.
if session._watch_cooldown_until and now < session._watch_cooldown_until:
    session._watch_suppressed += len(matched_lines)
    if not session._watch_strike_candidate:
        # First drop in this window — count one strike.
        session._watch_strike_candidate = True
        session._watch_consecutive_strikes += 1
        if session._watch_consecutive_strikes >= WATCH_STRIKE_LIMIT:
            session._watch_disabled = True
            # Promote to notify_on_complete so the agent still gets
            # exactly one notification when the process actually ends.
            session.notify_on_complete = True
            should_disable = True
```

**设计亮点**：连续超速后逐步降级，达到 strike limit 后禁用 watch 并升级为 `notify_on_complete`，确保用户仍能收到最终通知。
